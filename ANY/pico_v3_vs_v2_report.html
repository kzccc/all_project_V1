总体判断： 如果你只关心 agent 本身是不是更会做事，那答案是： 没有看到特别硬的能力跳升。 V3 主要是把 V2 已有方向往工程化和可复盘上继续推，而不是在工具能力、推理流程或任务完成范式上出现明显新台阶。

V3 真正算得上改进的地方

只保留对核心使用体验有感知的部分，不放大边角项。
配置体系

从全局隐式配置，改成项目内显式配置

  • V2 会读取 ~/.pico/config.json，首次还会尝试从 ~/.codex/config.toml、~/.codex/auth.json 引导默认配置。
  • V3 改成新增 pico/config.py，正式支持项目根目录向上查找 .env，并提供 PICO_* 命名空间。
  • 这更像使用方式优化，不是 agent 智能本身增强。
恢复机制

resume 从“能接着聊”升级到“能判断恢复是否可信”

  • V3 对 checkpoint 新增了 schema_version、runtime_identity、freshness、key_files 等结构。
  • 恢复状态被细分为 full-valid、partial-stale、workspace-mismatch、schema-mismatch、no-checkpoint。
  • 这是 V3 最接近核心能力提升的一项，因为它直接影响“恢复后还能不能继续正确做事”。
提示词预算

上下文压缩逻辑更明确，也更可解释

  • V3 的 ContextManager 把 prompt 拆成 prefix / memory / relevant_memory / history / current_request 五段。
  • 超预算时的压缩顺序明确为先砍 relevant_memory，再砍 history，最后才动更基础的 memory 和 prefix。
  • 但这部分在 V2 里已经有框架，V3 更像细化和补强，不是新范式。
安全与脱敏

敏感信息识别覆盖更完整

  • V3 默认敏感变量名单纳入 PICO_OPENAI_API_KEY、PICO_ANTHROPIC_API_KEY、PICO_DEEPSEEK_API_KEY、PICO_RIGHT_CODES_API_KEY 等项目级变量。
  • 项目 .env 会在 agent 装配前加载，这样 trace/report 的脱敏规则能及时生效。
  • 这属于工程卫生改进，不算核心 agent 能力提升。
运行可观测性

trace/report 不只是记录结果，也能解释运行过程

  • V3 会记录 prompt cache 相关字段、resume 状态、stale 路径、runtime identity mismatch 字段。
  • context reduction、workspace mismatch、freshness mismatch、tool executed、run finished 等场景都能自动创建 checkpoint 并打 trace。
  • 这提升了可调试性，但不直接代表 agent 更聪明或更强。
工程收敛

实现方式更聚焦主产品

  • 从 diff 结果看，V3 在核心包里是“边增强边收敛”的方向，不是无节制扩写。
  • 本质上是收敛和整理，不是能力跃迁。
  • README、.gitignore、环境样例文件都更贴近真实使用场景。

V3 明确新增的内容

这些是“确实新增了”，但不等于都属于核心能力升级。
新增项 V3 具体内容 价值 代表位置
DeepSeek Provider 新增 deepseek provider、默认模型 deepseek-v4-pro、默认 base URL https://api.deepseek.com/anthropic。 这是接入面扩展，不是 agent 内核升级。 pico/cli.py、README.md、tests/test_pico.py
项目级 .env 新增 .env.example，支持 PICO_OPENAI_*、PICO_ANTHROPIC_*、PICO_DEEPSEEK_*。 属于使用成本优化。 .env.example、pico/config.py、.gitignore
Durable Memory 新增长期记忆存储：.pico/memory/MEMORY.md + .pico/memory/topics/*.md。 这是新增能力，但从代码看更像把 V2 的 memory 路线继续做完整。 pico/memory.py、pico/runtime.py
Checkpoint Taxonomy 新增恢复状态分类、runtime identity 比对、schema 版本检查、stale path 检查。 这是最值得保留的新增点，确实更影响实际连续工作。 pico/runtime.py、tests/test_pico.py
Prompt Cache Telemetry V3 在 OpenAI-compatible 链路里记录 cached_tokens、cache_hit、prompt_cache_key。 偏观测和评测，不直接提升任务解决能力。 pico/models.py、pico/runtime.py、pico/metrics.py
Recovery / Context / Memory Ablation 补强 V3 的脚本和指标支持 real provider 里加入 DeepSeek，并对 recovery、prompt compression、memory hit rate 做更完整归档。 偏实验体系，不是主执行链路增强。 scripts/*.py、pico/metrics.py

核心判断

按“它到底是不是更会做任务”来收缩结论。

V2 已经具备的核心能力

  • 能跑本地 coding agent。
  • 有 OpenAI-compatible / Anthropic-compatible / Ollama 基础能力。
  • 有 memory、context、trace、report、benchmark 的雏形。
  • resume、memory、context reduction、trace/report、benchmark 主骨架都已经在。
  • 也就是说，V2 并不是“只有壳子”，核心思路其实已经写出来了。

V3 相比 V2 的真实增量

  • 最实在的增量是 checkpoint/resume 判定更细，恢复更稳。
  • 其次是 durable memory 落盘做得更完整，但仍属于 memory 路线延长，不是全新能力层。
  • DeepSeek、PICO_*、cache telemetry、实验脚本都更像外围增强。
  • 所以如果你的问题是“V3 有没有明显更强”，我的判断是没有特别明显。

给你省时间的结论

如果你现在只想抓重点，可以只看这 4 条。

1. 真正偏核心的提升：恢复可靠性

如果你经常中断后继续任务，V3 的 checkpoint/resume 判定比 V2 更严格，这部分是最有实际意义的增强。

2. 次级增强：durable memory 更完整

它让 memory 从工作记忆延伸到跨 session 落盘，但从能力性质上看仍是已有 memory 方向的延长。

3. 大部分其余变化不该高估

DeepSeek、.env、cache 指标、脚本和报告，对工程使用有帮助，但不应该被解读成 agent 任务能力大幅变强。

4. 对你当前判断最重要的一句

V2 的核心设计并不弱，V3 更多是把部分链路做得更稳、更细，而不是把核心范式重写了一遍。

代表性源码锚点

这些文件最能体现 V3 相比 V2 的升级方向。
pico/config.py：项目级 .env 发现与加载
pico/cli.py：DeepSeek、PICO_*、agent 装配入口
pico/runtime.py：checkpoint、resume、durable memory、prompt metadata
pico/memory.py：durable topic store、freshness、召回融合
pico/context_manager.py：relevant memory 与预算压缩策略
pico/models.py：prompt cache telemetry、provider 适配
pico/metrics.py：provider / recovery / memory / context 指标扩展
tests/test_pico.py：DeepSeek、checkpoint、durable memory、cache hit 测试
tests/test_metrics.py：DeepSeek 配置与 benchmark 指标回归测试
.env.example：项目内配置模版

收缩后的结论是：V3 有增量，但核心能力增量没有大到值得你转移很多注意力。 如果你现在只想抓最值钱的东西，优先看 resume/checkpoint，其次看 durable memory；其余大多属于外围工程整理。
