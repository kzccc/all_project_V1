# EchoChat 熔断器、限流、降级策略方案

## 1. 这份文档解决什么问题

这份文档不是泛讲“什么是熔断、限流、降级”，而是专门结合你当前 `echochat` 的架构，给出一版**最适合现在这套系统的保护策略**。

当前你这套链路的核心特点是：

- WebSocket 作为实时入口
- Kafka 作为主干缓冲层
- Consumer 做顺序、persist、分发、缓存、状态更新
- 目前主要瓶颈不在单次 MySQL 写慢，而在：
  - `ready wait`
  - 热路径负担
  - 部分队列堆积

所以这份文档目标是：

1. 防止系统在高峰时“慢慢拖死”
2. 在过载时优先保核心功能
3. 让保护动作可观测、可恢复、可逐步启用

---

## 2. 先说结论

对于你当前 `echochat`，我认为最好的保护体系不是先上很复杂的统一熔断框架，而是：

**“入口限流 + 队列背压 + 非关键路径降级 + 依赖级熔断 + 明确优先级”**

优先级顺序应该是：

1. **先做降级和背压**
2. **再做入口限流**
3. **最后补依赖熔断**

原因很简单：

- 你现在主问题更像系统过载下链路堆积
- 不是典型“某个下游 RPC 持续报错”

所以最值的不是先引入一个 Hystrix 风格的统一熔断器，而是：

- 先避免消息热路径继续变重
- 再避免慢请求无限堆积
- 最后再处理依赖级失败放大

---

## 3. 先定义保护目标

你这套 IM 系统里，不同能力的重要性不一样。

必须先分层，不然过载时不知道该保谁。

### 3.1 核心必须保

这些是系统过载时最应该保住的：

1. 消息进入 Kafka
2. consumer 核心消费
3. `session_seq` 正确生成
4. 核心消息持久化
5. 在线消息实时投递

### 3.2 可以延迟但尽量保

这些可以慢一点，但最好别直接丢：

1. `status=sent` 更新
2. 会话摘要更新
3. 未读数更新
4. 活跃会话更新

### 3.3 可以优先降级

这些是过载时最应该先砍掉或简化的：

1. 完整消息列表缓存同步重写
2. 非关键 benchmark / hot-path 日志
3. 大量 debug 观测
4. 非关键 webhook
5. 辅助统计类写入

这三层分清楚以后，熔断、限流、降级才有方向。

---

## 4. 你当前系统最适合的保护点在哪里

我建议按链路看。

## 4.1 WebSocket 入口层

这里的目标不是“拦掉所有请求”，而是防止：

- 单连接刷爆
- 单用户恶意刷消息
- 瞬时峰值把后面所有队列都顶满

结合你现在的代码：

- `Client.Read()` 是直接 `Conn.ReadMessage()` 后继续下游处理  
  [client.go](/workspace/czk/Personal/KKK/internal/service/chat/client.go:240)
- `enqueueLocal` / `enqueueBack` 已经是非阻塞思路  
  [client.go](/workspace/czk/Personal/KKK/internal/service/chat/client.go:123)

所以入口层最适合补的是：

### 方案 A：单连接令牌桶限流

建议限制：

- 每连接每秒最大消息数
- 每连接短时 burst

目的：

- 防止单连接异常刷流量

推荐第一版：

- 普通连接：`50~100 msg/s`
- benchmark 路由：可放开或单独配置

### 方案 B：单用户并发写保护

对同一用户的消息发送，可以增加轻量“在途请求数”限制。

目的：

- 避免单用户把自己的 websocket 输入通道占满

### 方案 C：消息体大小限制

这不算典型限流，但很值。

限制：

- 单条文本长度
- 附带 JSON 大小

目的：

- 防止大包拖垮 decode / Redis / Kafka / DB

---

## 4.2 WebSocket 回写层

你现在的 `SendBack` / `CriticalBack` 已经是个很好的保护基础。

当前特点：

- 普通消息回写走 `SendBack`
- 关键失败提示走 `CriticalBack`
- 写协程已经做了隔离

见：

- [client.go](/workspace/czk/Personal/KKK/internal/service/chat/client.go:45)

这里最适合的策略不是熔断，而是：

### 方案 A：慢连接剔除

如果：

- `SendBack` 长期高水位
- websocket write 持续超时
- 连接多次触发 back queue wait 过高

则主动断开该连接。

这本质上是：

- **慢消费者淘汰**

IM 系统里非常常见，而且很有必要。

### 方案 B：普通消息丢弃策略，关键消息保底

在极端过载时：

- 普通通知/回执消息可以丢弃或合并
- 关键控制消息保留

你现在已经有 `CriticalBack`，这是非常好的基础。

---

## 4.3 Kafka 生产层

这层是你入口后第一道真正的缓冲层。

因为你是 Kafka 主干，所以这里不建议做重熔断，而是做：

### 方案 A：Kafka 生产失败快速降级

如果 Kafka producer 出现：

- 连续超时
- 连续 send failure
- broker 不可用

不要让 websocket 请求无限等待。

应该：

1. 快速返回“系统繁忙，请稍后重试”
2. 打开短期熔断窗口
3. 在窗口内拒绝新消息进入

这就是一个典型的：

- **依赖级熔断**

### 方案 B：按错误类型区分

要区分：

- 短暂 broker 抖动
- topic 不存在
- metadata 问题
- timeout

不能所有错误都统一当成永久故障。

---

## 4.4 Consumer 队列与 ready queue 层

这块是你当前最值得保护的地方。

因为你现在主要问题已经是：

- ready queue 等待高
- worker 调度压力高

所以这里最好的不是“请求入口再多挡一点”，而是：

- **局部背压**
- **热点会话限流**
- **过载时拒绝继续扩散副作用**

### 方案 A：基于 ready queue 深度的背压

如果某个 partition / bucket 的：

- ready queue 使用率超过阈值

可以触发：

1. 新消息只保核心路径
2. 关闭重缓存更新
3. 暂时提高 worker 连续处理配额
4. 必要时对热点会话做限流

这个是你最该做的一层。

### 方案 B：热点会话局部限流

不是全局限流，而是对单个会话做：

- 每秒最大消息数
- 每秒最大积压数

如果某个群/单聊会话远超平均值：

- 限它
- 不要让它拖死整个 partition

这点尤其适合群聊热点治理。

### 方案 C：bucket 级别 shed load

当某个 bucket 内积压超过阈值时，优先：

- 停止该 bucket 的非关键 side-effect
- 而不是让整分区一起变慢

---

## 4.5 mysql_persist 层

你现在单次持久化不是主瓶颈，但 persist 仍然是主链路骨架，必须保护。

这里不建议轻易熔断“核心落库”，而应该做：

### 方案 A：persist queue 高水位保护

如果某个 persist worker 的 queue 深度持续超过阈值：

触发策略：

1. 关闭完整消息列表缓存同步更新
2. 关闭非关键日志
3. 暂停 summary/unread 的同步更新
4. 必要时提高 flush 侵略性

本质是：

- **保 persist，砍副作用**

### 方案 B：数据库异常时的短熔断

如果 MySQL 出现：

- 连接拿不到
- lock wait timeout 暴增
- 批量 insert 连续失败

需要有一个短熔断器：

- 暂停新副作用写入
- 对入口快速报繁忙
- 保留 Kafka backlog

这里一定要注意：

**不要用熔断去“丢消息”，而是用熔断去“阻止继续放大损害”。**

---

## 4.6 Redis 层

Redis 在你系统里承担了很多辅助职责：

- `session_seq`
- 路由
- 缓存

所以 Redis 的降级要分层。

### 方案 A：缓存层 Redis 故障时优先降级缓存功能

如果 Redis 抖动：

- 先关掉 message list 缓存更新
- 再关掉 summary/unread 缓存更新
- 核心 seq 和路由尽量保

### 方案 B：在线路由 Redis 故障时退化为本机能力

如果跨实例路由 Redis 有问题：

- 本机在线用户照常投递
- 跨实例远投快速失败并记录
- 不要因为 Redis 路由异常把整个 websocket 服务卡死

### 方案 C：session_seq Redis 故障是高危依赖

这层不能简单降级。

对 `session_seq` 来说，更适合的是：

- 快速探活
- 短时间错误重试
- 熔断后直接拒绝新消息

而不是“跳过 seq 继续发消息”。

因为跳过 seq 会伤到系统一致性。

---

## 5. 最适合你当前项目的降级策略

这里我给一个明确顺序。

## 第一级降级：先砍重缓存

一旦检测到：

- ready queue 高水位
- persist queue 高水位
- consumer lag 明显增长

第一时间做：

1. 关闭单聊 `message_list_xxx` 同步重写
2. 关闭群聊 `group_messagelist_xxx` 同步重写
3. 改成“只失效，不重写”

这是当前最值的一刀。

## 第二级降级：再砍非关键状态写

包括：

1. `status=sent` 改成更大批次、更长延迟
2. summary/unread 延迟更新
3. 非关键 webhook 延迟或停发

## 第三级降级：最后才动入口接收

只有在下面这些都没用了，才做：

- 单连接限流收紧
- 单用户限流收紧
- 热点会话限流
- Kafka 入口短熔断

这顺序很重要。

因为你当前最该先保的是：

- 核心消息链路

而不是一上来就把用户请求挡掉。

## 6. 熔断器怎么设计最适合你

我不建议你一开始引入特别重的统一熔断平台。

对你最适合的是：

## 6.1 依赖级短熔断器

优先放在：

1. Kafka producer
2. Redis `session_seq`
3. MySQL persist

触发条件建议：

- 连续 N 次失败
- 或 10s 窗口失败率 > 阈值

恢复条件建议：

- 半开探测成功 M 次再恢复

动作建议：

- Kafka producer 熔断：拒绝新消息进入
- Redis cache 熔断：关闭缓存副作用
- MySQL persist 熔断：停止副作用，保 backlog，不直接丢消息

## 6.2 功能级熔断器

对这些功能单独做：

1. 消息列表缓存更新
2. summary/unread 更新
3. webhook

特点：

- 它们熔断后核心聊天功能仍能活

这才是最适合你项目的“功能降级熔断”。

---

## 7. 限流怎么设计最适合你

## 7.1 三层限流最合适

建议只做三层，不要太散。

### 第一层：连接级限流

目的：

- 防止单连接刷爆

### 第二层：用户级限流

目的：

- 防止单用户恶意或异常流量

### 第三层：会话级限流

目的：

- 防止热点群/热点单聊拖垮整个 partition

这三层比“全局一个 QPS 限流器”更适合 IM。

因为 IM 的问题更常见的是：

- 局部热点

而不是所有用户一起平均超额。

---

## 8. 我建议你优先观测哪些指标来驱动这些策略

没有观测，保护策略很容易误伤。

建议最少看这几类：

1. Kafka producer error rate
2. Kafka consumer lag
3. `conversation bucket ready queue` 深度
4. `mysql_persist queue` 深度
5. Redis 超时率
6. WebSocket sendback queue 深度
7. MySQL batch insert error / timeout rate

保护动作建议全部由这些指标驱动，而不是拍脑袋。

---

## 9. 一版最适合 `echochat` 的落地优先级

如果只讲现实可落地顺序，我建议这样做。

### P0

1. 完整消息列表缓存更新改成“失效优先”
2. 增加慢连接剔除
3. 增加单连接限流

### P1

1. 增加 ready queue / persist queue 高水位降级
2. 增加热点会话限流
3. 增加 Kafka producer 短熔断

### P2

1. 增加 MySQL persist 依赖熔断
2. 增加 Redis cache 功能熔断
3. 增加 summary/unread 功能级降级

### P3

1. 再考虑统一熔断框架或更精细的策略中心

## 10. 一句话总结

对当前 `echochat` 最好的熔断、限流、降级策略，不是先做一个很重的统一熔断器，而是**优先做“重缓存降级 + 队列背压 + 慢连接剔除 + 会话级限流 + 依赖级短熔断”**；你的系统当前最该保护的是 Kafka 主干、`session_seq`、核心持久化和在线投递，而不是为了“保护”把入口一开始就挡死。

---

## 11. 压缩版关键点

1. `echochat` 现在最该优先保护的是 Kafka 主干、`session_seq`、核心持久化和在线投递。
2. 最先降级的应该是完整消息列表缓存同步重写，而不是入口直接拒绝消息。
3. 最适合你的限流是连接级、用户级、会话级三层，而不是单一全局 QPS 限流。
4. 最适合你的熔断是 Kafka producer、Redis `session_seq`、MySQL persist 这类依赖级短熔断。
5. 最适合你的过载保护核心是 ready queue / persist queue 高水位触发的背压和功能降级。
6. WebSocket 层应尽快补慢连接剔除和回写队列保护，避免少数慢客户端拖垮系统。
7. Redis 缓存层故障时应优先降级缓存功能，`session_seq` 则不应轻易跳过。
8. 真正的保护顺序应是先砍副作用、再控热点、最后才收紧入口。
