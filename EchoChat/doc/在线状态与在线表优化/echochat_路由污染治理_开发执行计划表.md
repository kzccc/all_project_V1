## 1. 计划表目标

本文档用于承接路由污染治理方案的具体开发执行计划，并承担以下职责：

1. 把设计方案拆成可执行、可验收、可跟踪的三级任务。
2. 为每个任务明确目标、交付物、依赖关系和完成标准。
3. 为后续开发过程中的中断恢复提供进度锚点，确保任何阶段都能快速判断“做到哪了、还差什么”。

## 2. 进度跟踪规则

### 2.1 状态定义

- `未开始`：尚未进入开发。
- `进行中`：已经开始，但还未满足验收条件。
- `已完成`：代码、配置、测试和文档均满足当前条目验收条件。
- `阻塞`：因外部依赖、设计决策或联调条件不足而无法继续推进。

### 2.2 进度字段说明

- `总体进度`：一级任务完成度，用于观察整个阶段推进情况。
- `模块进度`：二级任务完成度，用于观察模块内部推进情况。
- `条目进度`：三级任务完成度，用于观察当前最小执行单元。

### 2.3 更新约定

每次开发推进时，至少要更新以下内容：

1. 当前进行中的三级条目状态。
2. 当前进行中的二级模块进度。
3. 一级任务总体进度。
4. 当前阻塞点或已完成说明。

## 3. 当前实现前提

在进入本轮开发前，需要明确当前系统前提：

1. 当前本地在线表是 `user_id -> *Client` 的单连接模型。
2. 当前 Redis 路由表是 `user_id -> instance_id` 的永久 key 模型。
3. 当前项目尚未真正落实“用户多端多设备并存”的连接集合模型。
4. 本轮开发默认维持现有单用户单活跃连接语义，不在本轮内扩展为多连接集合。

## 4. 总体阶段拆分

本轮开发拆为五个一级阶段：

1. 服务端连接层心跳续约机制落地。
2. Redis 路由 TTL 与结构化路由落地。
3. 客户端自动重连机制落地。
4. 观测、测试与故障场景验证。
5. 收尾、灰度与上线准备。
