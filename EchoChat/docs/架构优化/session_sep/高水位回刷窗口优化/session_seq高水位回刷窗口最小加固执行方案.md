# session_seq 高水位回刷窗口最小加固执行方案

## 1. 这份方案要解决什么问题

这份方案只解决一个非常具体的问题：

**在 `session_seq` 已经改成 Redis 主发号后，如果高水位还没 flush 到 MySQL 就发生崩溃，同时 Redis 状态又丢失，恢复时可能出现 `session_seq` 回退或重复。**

当前最危险的场景是：

1. Redis 已经把某个会话的序号发到了更大的值
2. 部分消息也已经成功落库到 `message`
3. 但 `conversation_sequence.last_seq` 还没来得及 flush 到最新
4. 进程崩溃
5. Redis key 丢失
6. 恢复时系统只看到了一个滞后的 `conversation_sequence.last_seq`

如果恢复逻辑只信这条滞后的高水位，就会出现：

1. 已经用过的 `session_seq` 被再次发出
2. 同一会话内 `session_seq` 回退或重复
3. 历史消息顺序污染

这不是吞吐问题，而是：

**顺序语义的恢复窗口问题。**

---

## 2. 这次为什么只先做两步

当前可选的加固手段很多，但这次我只建议先落最小闭环：

1. 恢复时取 `max(高水位表, message 表最大 seq)`
2. 给 `(conversation_key, session_seq)` 加唯一约束

原因很明确：

1. 这两步就能挡住最危险的“回退 + 静默污染”
2. 对现有主链路吞吐影响很小
3. 不需要立刻把高水位 flush 改成强一致事务模式
4. 改动范围可控，验证口径也比较清楚

也就是说，这不是最终最强方案，而是：

**当前最值得先补的最小加固方案。**

---

## 3. 方案一：恢复时取 `max(高水位表, message表最大seq)`

### 3.1 现在的问题

当前恢复语义里，如果 Redis 没有这个会话的计数器，系统会去拿恢复起点。

但如果恢复起点只看：

1. `conversation_sequence.last_seq`

那一旦这条记录滞后于 `message` 表真实已落库的最大 `session_seq`，恢复后的发号就可能回退。

### 3.2 这一步改成什么

恢复起点不再只取高水位表，而是改成：

**恢复起点 = `max(conversation_sequence.last_seq, message表当前最大session_seq)`**

语义上就是：

1. Redis 丢了，需要为这个会话重新建发号器
2. 先看高水位表记到了哪里
3. 再看消息表真实已经写到了哪里
4. 两边取更大值
5. 再从这个更大值往后发新号

### 3.3 为什么这一步有效

因为真正危险的不是“高水位 flush 慢”，而是：

**恢复时把一个旧高水位误当成了最新真相。**

只要恢复时多看一眼 `message` 表真实最大值，这个问题就能被兜住。

也就是说：

1. 高水位表允许稍微滞后
2. 但恢复起点不能比真实已落库消息更小

这样即使崩溃发生在 flush 之前，也最多出现：

1. 跳号

而不应该出现：

1. 回退
2. 重复发号

### 3.4 这一步的代价

这一步对正常热路径吞吐影响很小。

因为：

1. 它只发生在 Redis miss / 会话恢复时
2. 不是每条消息都多查一次 `message` 表
3. 它属于低频恢复路径，而不是高频实时路径

所以这一步本质上是在：

**用低频恢复开销，换高风险窗口的收敛。**

---

## 4. 方案二：给 `(conversation_key, session_seq)` 加唯一约束

### 4.1 现在的问题

即使恢复逻辑写得更稳，也不能假设它绝对不会出错。

如果未来某次：

1. 恢复逻辑遗漏
2. 边界情况没兜住
3. 或者别的路径直接写入了重复 `session_seq`

当前数据库层如果没有唯一约束，就会出现最危险的一种情况：

**错误序号被静默写入成功。**

这意味着：

1. 系统不会立刻报错
2. 数据已经坏了
3. 后面历史查询或前端排序时才暴露问题

### 4.2 这一步改成什么

给 `message` 表增加唯一约束：

**`UNIQUE(conversation_key, session_seq)`**

语义上就是：

1. 同一个会话内
2. 同一个 `session_seq`
3. 只能存在一条消息

### 4.3 为什么这一步有效

这一步不是解决根因，而是解决“静默污染”。

它能把原来这种情况：

1. 回退了
2. 重号了
3. 数据悄悄写进去了

变成：

1. 数据库直接拒绝写入
2. 系统明确报错
3. 后续可以走重试、补偿或者人工定位

它的价值非常大，因为对顺序系统来说，最可怕的不是“报错”，而是：

**错了但没人知道。**

### 4.4 这一步的代价

这一步对写入会有一点点成本，但通常远小于重新设计整条链路。

真正要注意的不是吞吐，而是上线前必须先确认：

1. 历史数据里有没有已经存在的 `(conversation_key, session_seq)` 冲突
2. 如果有，要先清理或修复

否则加唯一约束时会直接失败。

---

## 5. 这两步结合起来后，系统语义会变成什么

做完这两步以后，这条线的恢复语义会明显更稳：

### 正常运行时

1. 仍然是 Redis 主发号
2. 高水位仍然可以异步 flush
3. 吞吐模型基本不变

### Redis 丢失、需要恢复时

1. 恢复起点不再只信高水位表
2. 会和消息表真实最大 seq 取更大值
3. 避免恢复起点回退

### 即使恢复逻辑还有遗漏

1. 数据库唯一约束会阻止同一会话重复序号被静默写入

所以这两步组合起来，防的是两类问题：

1. 恢复时回退
2. 写入时静默污染

---

## 6. 为什么说这两步对吞吐影响很小

这是这份方案最关键的一点。

### 6.1 恢复取 max 的成本

它发生在：

1. Redis miss
2. 会话冷启动
3. 崩溃恢复

不是每条消息都发生。

所以它增加的是：

**恢复路径成本**

不是：

**热路径成本**

### 6.2 唯一约束的成本

这会让 `message` 表多一层唯一性检查，但这个代价通常是可控的。

相比你当前系统里已经存在的：

1. `uuid` 唯一约束
2. `conversation_key + session_seq` 普通索引

这一步更像是：

1. 把一个原本已经在索引层维护的维度
2. 提升成硬约束

它不是零成本，但也不是那种会把吞吐砍掉一大截的改动。

所以整体上，这两步属于：

**一致性收益很大，吞吐代价较小。**

---

## 7. 我的执行顺序

### 第一步：先检查历史数据

先确认历史 `message` 表里是否已经存在：

1. 同一个 `conversation_key`
2. 同一个 `session_seq`
3. 对应多条消息

如果有，必须先清理。

这一步不是优化，而是给后面加唯一约束扫雷。

### 第二步：修改恢复逻辑

把当前 Redis miss / 会话恢复时的起点逻辑改成：

1. 读 `conversation_sequence.last_seq`
2. 读 `message` 表当前最大 `session_seq`
3. 取两边最大值作为恢复 floor

这一步先落，因为它直接解决“回退”问题。

### 第三步：补唯一约束

在确认历史数据干净后，给 `message` 表增加：

1. `UNIQUE(conversation_key, session_seq)`

这一步落完以后，就算未来还有边界遗漏，也不会再静默写脏数据。

### 第四步：补最小观测

建议顺手补 3 类观测：

1. 恢复时命中了哪种 floor 来源
   - 只来自高水位表
   - 只来自 message 表
   - 两边取 max 后来自 message 表
2. `conversation_key + session_seq` 唯一约束冲突次数
3. 恢复路径触发次数

这样后面你才能知道：

1. 这个窗口问题到底是否真实发生过
2. 新方案上线后有没有把它压住

## 8. 验收标准

这次不是吞吐专项，验收要分成“一致性验收”和“性能验收”两部分。

### 8.1 一致性验收

重点验证下面这个故障场景：

1. Redis 已经发号
2. 消息已部分落库
3. 高水位未 flush 到最新
4. 进程崩溃
5. Redis 状态丢失
6. 服务重启恢复

验收目标：

1. 恢复后的起点不小于 `message` 表真实最大值
2. 不出现 `session_seq` 回退
3. 不出现同一会话内重复 `session_seq`

### 8.2 性能验收

关注下面两点就够了：

1. 常规热路径吞吐不应出现明显回退
2. `session_seq` 热路径 stage 不应重新变重

也就是说，这次性能验收只要求：

**不要把恢复逻辑重新拖回热路径。**

## 9. 我给自己的执行清单

### P0

1. 先查历史 `message` 表是否已有 `(conversation_key, session_seq)` 冲突
2. 明确如果有冲突，怎么修数据

### P1

1. 改恢复逻辑：恢复 floor 取 `max(高水位表, message表最大seq)`
2. 补恢复路径日志或指标

### P2

1. 给 `message` 表补唯一约束 `UNIQUE(conversation_key, session_seq)`
2. 处理上线迁移和失败回滚方案

### P3

1. 做一次故障注入验证
2. 确认 Redis 丢失 + flush 未完成时，不会发生回退和静默污染

---

## 10. 一句话执行口令

**先把恢复逻辑改成“高水位表和 message 表最大 seq 取最大值”，再给 `(conversation_key, session_seq)` 加唯一约束，用最小改动先堵住 `session_seq` 的回退和静默污染窗口，同时尽量不伤热路径吞吐。**
