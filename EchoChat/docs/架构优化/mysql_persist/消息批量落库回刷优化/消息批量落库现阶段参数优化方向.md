# EchoChat 消息批量落库现阶段参数优化方向

## 1. 这份文档讨论什么

这份文档只讨论一件事：

**Kafka 单机链路里，`mysql_persist` 这一段现阶段应该怎么继续做参数和指标优化，目标是尽量榨干单机性能。**

这里的“榨干机器性能”不是单纯把参数往大了堆，而是要同时满足：

1. 单聊 / 群聊最大稳定吞吐尽量高
2. `p95` 不要被明显拉坏
3. MySQL 不要因为过度并发进入明显抖动区
4. 结论能够稳定复现，而不是某一次偶然跑高

这份文档口径限定为：

- 单机
- Kafka 模式
- 重点是消息落库 `mysql_persist`
- 不展开 `status=sent` 异步回刷
- 不展开 commit 前链路轻量化

---

## 2. 当前实现现状

现阶段 `mysql_persist` 的关键特征是：

1. consumer 主协程不会直接执行 SQL
2. 消息会先按会话 scope 哈希到某个 persist worker
3. worker 会做批量聚合后统一 `INSERT INTO message`
4. 但 consumer 仍然会等待这次落库结果返回，再继续后续流程

所以它不是彻底 fire-and-forget，而是：

**异步执行 + 同步等待结果。**

这条语义很重要，因为它决定了：

1. `mysql_persist` 仍然在 commit 前主链路里
2. worker 虽然把 SQL 执行线程和 consumer 主协程拆开了
3. 但只要 worker 出口过窄、batch 不合适、连接池不匹配，吞吐仍然会先被这一段卡住

---

## 3. 当前默认值和现阶段推荐停点

### 3.1 当前代码默认值

如果不显式配置，当前默认值是：

1. `mysqlPersistWorkerCount = 8`
2. `mysqlPersistBatchSize = 64`
3. `mysqlPersistFlushIntervalMs = 5`
4. `maxOpenConns = 200`
5. `maxIdleConns = 50`

但你现在实际单机压测常用配置并不是这组默认值，而是更激进的压测基线：

1. `maxOpenConns = 500`
2. `maxIdleConns = 100`

### 3.2 当前已验证过的 `mysql_persist` 三档

第三次优化第一次执行里，`mysql_persist` 已经做过三档扫描：

第一档：

1. `worker: 8 -> 24`
2. `batch: 64 -> 128`
3. `flush: 1 -> 2`

结果：

1. 单聊：`1684.839 msg/s`，`p95 = 776.05 ms`
2. 群聊：`4434.853 delivery/s`，`p95 = 269 ms`

第二档：

1. `worker: 24 -> 32`
2. `batch: 128 -> 256`
3. `flush: 2 -> 5`

结果：

1. 单聊：`1681.304 msg/s`，`p95 = 676.2 ms`
2. 群聊：`4458.665 delivery/s`，`p95 = 121 ms`

第三档：

1. `worker: 32 -> 40`
2. `batch: 256 -> 384`
3. `flush: 5 -> 8`

结果：

1. 单聊：`1530.825 msg/s`，`p95 = 398.05 ms`
2. 群聊：`3595.012 delivery/s`，`p95 = 469 ms`

### 3.3 当前最重要的已知结论

现阶段已经有 3 个结论可以直接当成前提：

1. `mysql_persist` 确实是主瓶颈之一，这条线调宽是有收益的。
2. 第一档吃到了最大吞吐收益，第二档更像“稳住尾延迟”的最优折中点。
3. 第三档已经开始回退，说明这条线不能再靠无脑继续加 `worker / batch / flush` 往上推。

所以如果只看当前已有证据，现阶段推荐停点仍然是：

1. `mysqlPersistWorkerCount = 32`
2. `mysqlPersistBatchSize = 256`
3. `mysqlPersistFlushIntervalMs = 5`
4. `maxOpenConns = 500`
5. `maxIdleConns = 100`

---

## 4. 现阶段真正该优化的，不只是 3 个参数

如果目标只是“把 `mysql_persist` 继续调一调”，那只看：

1. `workerCount`
2. `batchSize`
3. `flushInterval`

还勉强够用。

但如果目标已经升级成：

**尽量榨干单机性能。**

那现阶段至少应该把 5 个参数放在一起看：

1. `mysqlPersistWorkerCount`
2. `mysqlPersistBatchSize`
3. `mysqlPersistFlushIntervalMs`
4. `maxOpenConns`
5. `maxIdleConns`

辅助参数再看：

1. `connMaxLifetimeMinutes`
2. `connMaxIdleTimeMinutes`
3. persist worker channel 容量（当前还没配置化）

原因很简单：

1. 应用 worker 再多，DB 入口太窄，最终还是卡连接池
2. DB 入口放很宽，worker / batch / flush 不匹配，最终还是会打碎批次
3. 连接池和批量窗口一旦都开太大，单机 MySQL 又可能进入明显抖动区

所以单机榨干性能的核心不是“某一个参数尽量大”，而是：

**worker 并发、批量聚合、等待窗口、连接池宽度，要一起落在同一个甜点区。**

---

## 5. 现阶段最值得盯的 5 个主参数

### 5.1 `mysqlPersistWorkerCount`

它决定：

1. 批量落库的并发出口宽度
2. 前面 claim 拉进来的消息，最后会不会在少量 persist worker 重新排队

太小的信号：

1. `mysql_persist` stage 长时间偏高
2. 同时 backlog 持续增长
3. batch 经常打满仍消化不完

太大的信号：

1. 单次 batch 变碎
2. MySQL 并发竞争抬头
3. 单聊或群聊稳定吞吐不升反降

### 5.2 `mysqlPersistBatchSize`

它决定：

1. 单次 `INSERT` 最多合并多少条消息
2. 固定 SQL 成本能被摊薄到什么程度

太小的风险：

1. SQL 次数太多
2. 固定 insert 成本摊不薄

太大的风险：

1. 单次等待时间变长
2. 热点会话更容易在 worker 内部堆波峰
3. 到某个点以后吞吐不再涨，只剩尾延迟波动

### 5.3 `mysqlPersistFlushIntervalMs`

它决定：

1. worker 最长愿意等多久再刷出当前 batch
2. batch 是更偏“及时”，还是更偏“聚合”

太小的风险：

1. timer flush 过多
2. batch 太碎
3. SQL 频次变高

太大的风险：

1. 单条消息等待时间被拉长
2. 热点 worker 更容易积波峰
3. 单聊高压下 `p95` 可能恶化

### 5.4 `maxOpenConns`

它决定：

1. 单机 MySQL 同时允许多少打开连接
2. `mysql_persist`、历史消息查询、登录鉴权等路径总共能争到多宽的 DB 入口

太小的信号：

1. `in_use` 经常贴着上限
2. `wait_count / wait_duration` 抬升
3. worker 看起来不够快，但本质是等连接

太大的信号：

1. 连接池等待下降了
2. 但 DB CPU、锁等待、单次 flush SQL 耗时反而上升

### 5.5 `maxIdleConns`

它不像 `maxOpenConns` 那样决定理论最大并发，但它决定高压时连接复用是否稳定。

太小的风险：

1. 连接频繁创建回收
2. 高压下更容易出现抖动

太大的风险：

1. 不是吞吐主风险
2. 但会带来更多长期占用连接

所以它更像“把机器榨稳”的参数，而不是“把峰值再拔高”的第一主旋钮。

## 6. 现阶段最关键的问题，不是参数不够多，而是指标还不够细

现在已经有的观测里，最有价值的是：

1. `mysql_persist` stage duration
2. `consumer total`
3. 吞吐
4. `p95`
5. MySQL 连接池基础指标

但如果想继续把单机榨到更高，现有指标还不够。

至少还需要补下面这些：

### 6.1 persist worker 队列指标

建议新增：

1. `echochat_mysql_persist_queue_depth{worker}`
2. `echochat_mysql_persist_queue_capacity{worker}`

作用：

1. 看 persist worker 是不是在持续积压
2. 判断当前瓶颈是在“出口过窄”，还是只是临时波峰

### 6.2 persist flush 粒度指标

建议新增：

1. `echochat_mysql_persist_flush_batch_size`
2. `echochat_mysql_persist_flush_duration_seconds`
3. `echochat_mysql_persist_flush_total{reason=batch_full|timer,result=success|failure}`

作用：

1. 看真实 batch 有没有吃满
2. 看 flush 主要靠“攒满”还是“定时器”
3. 判断 batch / flush 是否已经进入收益拐点

### 6.3 enqueue 反压指标

建议新增：

1. `echochat_mysql_persist_enqueue_block_duration_seconds`
2. `echochat_mysql_persist_enqueue_total{result=success|blocked}`

作用：

1. 判断 consumer 往 persist worker 投任务时有没有被反压
2. 如果 enqueue 已经开始阻塞，说明 `mysql_persist` 已经反向拖主链路

### 6.4 连接池利用率聚合指标

当前已有：

1. `open`
2. `in_use`
3. `idle`
4. `wait_count`
5. `wait_duration`

建议再补一个更直接的聚合视角：

1. `mysql_in_use_ratio = in_use / maxOpenConns`

作用：

1. 一眼看连接池是不是长期贴顶
2. 更方便和 `workerCount` 一起联动判断

## 7. 现阶段最合理的执行方向

### 7.1 先承认一个事实

当前 `mysql_persist` 这条线不是“完全没调开”，而是已经调到过一轮有效停点了。

所以后面正确方向不是：

**从 32/256/5 再继续盲目往上堆。**

而应该是：

**在当前推荐停点附近，用更细指标判断机器真正还剩多少余量。**

### 7.2 第一方向：先把可观测性补齐

如果没有 persist worker 自己的：

1. queue depth
2. flush batch size
3. flush reason
4. enqueue block

那你很难分清：

1. 是 worker 太少
2. 是 batch 太碎
3. 是 flush 过急
4. 还是连接池已经在卡你

所以从工程优先级看，先补观测，比直接再调一轮参数更重要。

### 7.3 第二方向：围绕当前推荐停点做“小范围榨干扫描”

现阶段最值得扫描的不是“大跨度跳参数”，而是围绕当前停点小范围试探。

推荐首轮扫描范围：

1. `mysqlPersistWorkerCount = 24 / 32 / 40`
2. `mysqlPersistBatchSize = 128 / 256 / 320`
3. `mysqlPersistFlushIntervalMs = 3 / 5 / 7`
4. `maxOpenConns = 400 / 500 / 600`
5. `maxIdleConns = 100 / 150 / 200`

这里的逻辑是：

1. 第一档已经证明 24 起步就有明显收益
2. 第二档已经证明 32/256/5 是已知最好停点
3. 第三档已经证明直接冲到 40/384/8 已经过头
4. 所以后续应该做的是“围绕甜点区微调”，不是“继续放大一整档”

### 7.4 第三方向：先把 `maxOpenConns` 扫干净

如果目标是榨单机，当前最值得优先再确认的，其实不是 `batch`，而是 `maxOpenConns`。

因为：

1. 你现在 `mysql_persist` 已经放大到 32 worker
2. 但连接池仍固定在 500
3. 这个 500 是工程上比较激进的值，但不等于它已经是你这台机器的最佳值

后面真正可能还有收益的地方，很可能在：

1. `500` 仍偏小，worker 在隐性等连接
2. 或者 `500` 已经过大，MySQL 内部竞争只是暂时没被细指标量出来

所以 `maxOpenConns` 应该成为下一轮最优先确认项之一。

### 7.5 第四方向：不要急着再碰更大的 `batch + flush`

现有结果已经说明：

1. `256 / 5` 还处在甜点区
2. `384 / 8` 已经出现收益回退

所以后面如果要继续扫：

1. `batch` 不建议再直接冲 `384+`
2. `flush` 不建议再直接冲 `8ms+`

更合理的是：

1. 先围绕 `256` 左右做细扫
2. 先围绕 `5ms` 左右做细扫

---

## 8. 推荐的现阶段参数扫描顺序

为了保证结论干净，建议按下面顺序来：

### 第 1 步：补指标

先补：

1. persist queue depth
2. flush batch size
3. flush duration
4. flush reason
5. enqueue block

### 第 2 步：固定 `32 / 256 / 5`，先扫 `maxOpenConns`

推荐：

1. `400`
2. `500`
3. `600`

目标：

1. 看连接池是不是当前隐性瓶颈
2. 找 DB 入口宽度甜点区

### 第 3 步：固定较优连接池，再扫 `workerCount`

推荐：

1. `24`
2. `32`
3. `40`

目标：

1. 确认 `32` 是不是仍然最佳
2. 判断 `40` 回退到底是因为 worker 过多，还是因为当时连 `batch / flush / pool` 一起推大造成的

### 第 4 步：再扫 `batchSize`

推荐：

1. `192`
2. `256`
3. `320`

目标：

1. 找单次 SQL 聚合度的真实甜点

### 第 5 步：最后扫 `flushInterval`

推荐：

1. `3ms`
2. `5ms`
3. `7ms`

目标：

1. 找 timer flush 和等待时间的平衡点

## 9. 什么现象说明机器优势还没吃满

如果后续扫出来下面这些现象，说明你的机器优势可能还没完全发挥：

1. `mysql_persist` queue backlog 明显增长，但 MySQL CPU 和锁等待还很低
2. `wait_count / wait_duration` 明显存在，说明连接池还在卡入口
3. batch 经常打满，但单次 flush SQL 耗时没有明显恶化
4. 提高 `maxOpenConns` 后吞吐继续上涨，而 `p95` 仍能稳住
5. 提高 `workerCount` 后，queue 明显回落，但 DB 内部竞争还没有明显上升

这种情况说明：

**不是机器没能力，而是当前应用出口和 DB 入口还没有对齐。**

---

## 10. 什么现象说明已经接近当前单机甜点

如果你扫到下面这些现象，就说明当前机器已经比较接近甜点区：

1. 再加 worker，吞吐不涨，反而 batch 变碎
2. 再加 `maxOpenConns`，wait 降了，但 DB CPU / row lock wait 上升
3. 再加 batch 或 flush，吞吐不涨，反而群聊波动更明显
4. 单聊和群聊都开始出现“throughput 不再升，但 `p95` 波动加大”

这时就不应该再继续堆参数，而应该转去做结构优化，例如：

1. commit 前链路继续轻量化
2. websocket / Redis 进一步后移
3. 更彻底的持久化链路重构

---

## 11. 现阶段一句话结论

如果只用一句话总结当前消息批量落库的执行方向，我会写成：

**现阶段 `mysql_persist` 已经从默认值调到有效甜点区，下一步不是继续粗暴放大 `worker / batch / flush`，而是先补齐 persist 内部指标，再围绕 `32 / 256 / 5 + 500/100` 这组停点做小范围精扫，把应用侧并发和 MySQL 入口宽度对齐，尽量把单机余量吃干净。**
