# EchoChat 消息批量落库回刷优化执行计划

## 1. 目标

这次执行计划只做一件事：

**把 `mysql_persist` 这条线从“已有推荐停点”推进到“更接近单机上限的可验证停点”。**

最终要回答的问题是：

1. 当前 `32 / 256 / 5 + 500/100` 是不是你这台机器上的真实甜点
2. 机器余量到底还剩在应用侧，还是 DB 入口侧
3. 下一步应该继续调 `mysql_persist`，还是该转去做结构优化

---

## 2. 这次执行不做什么

为了保证结论干净，这次先不碰：

1. `session_seq`
2. commit 参数
3. websocket / Redis 后移
4. `status=sent` 异步回刷
5. consumer member 数

也就是说，这次只盯：

**消息批量落库 `mysql_persist` 自己。**

---

## 3. 执行阶段

### 阶段 A：补 `mysql_persist` 内部指标

先补下面 5 类指标：

1. `mysql_persist_queue_depth`
2. `mysql_persist_flush_batch_size`
3. `mysql_persist_flush_duration_seconds`
4. `mysql_persist_flush_total{reason=...}`
5. `mysql_persist_enqueue_block_duration_seconds`

同时保留现有：

1. `mysql_persist` stage duration
2. `consumer total`
3. MySQL 连接池指标
4. 吞吐与 `p95`

阶段 A 的目标不是提吞吐，而是把后面参数扫描的判断依据补齐。

### 阶段 B：基线复跑

用当前推荐停点先跑一轮基线：

1. `mysqlPersistWorkerCount = 32`
2. `mysqlPersistBatchSize = 256`
3. `mysqlPersistFlushIntervalMs = 5`
4. `maxOpenConns = 500`
5. `maxIdleConns = 100`

场景：

1. 单聊 3 档 target
2. 群聊 3 档 target

目标：

1. 确认补完指标后，基线现象和历史结论一致
2. 避免后面误把“版本波动”当成“参数收益”

### 阶段 C：先扫连接池，再扫应用侧

按下面顺序执行：

1. 固定 `32 / 256 / 5`，扫描 `maxOpenConns = 400 / 500 / 600`
2. 固定较优连接池，再扫 `workerCount = 24 / 32 / 40`
3. 固定前两者较优值，再扫 `batchSize = 192 / 256 / 320`
4. 最后扫 `flushInterval = 3 / 5 / 7`

原因：

1. 你现在的目标是单机榨干性能
2. 当前最可能还没被量干净的，是 DB 入口宽度
3. 如果连接池没先扫干净，后面 `worker / batch / flush` 的结论会掺杂

### 阶段 D：产出最终停点结论

最后要给出：

1. 推荐停点参数组
2. 每一步为什么停在这里
3. 哪些现象说明已经接近单机甜点
4. 下一步应不应该继续调 `mysql_persist`

---

## 4. 每轮实验固定模板

为了保证不同参数组可比，每组 run 固定：

1. 预热 `30s`
2. 正式压测 `180s`
3. drain `30s`

每轮采集：

1. `/metrics`
2. `pidstat`
3. `iostat`
4. `mpstat`
5. MySQL 连接池指标
6. InnoDB 锁等待相关指标

每轮输出最少字段：

1. `observed_throughput`
2. `p95_ms`
3. `mysql_persist_stage_avg/p95`
4. `persist_queue_depth_p95/end`
5. `persist_flush_avg_batch_size`
6. `persist_flush_p95_duration_ms`
7. `persist_timer_flush_ratio`
8. `mysql_in_use_conn_peak`
9. `mysql_wait_duration_delta`
10. `innodb_row_lock_waits_delta`

---

## 5. 判定规则

### 5.1 什么叫变好

满足下面条件，才算参数组真的更优：

1. 最大稳定吞吐更高，或同吞吐下 `p95` 更稳
2. `mysql_persist` queue 不持续增长
3. batch 不明显变碎
4. MySQL 连接池等待没有明显恶化
5. DB CPU / row lock wait 没有明显放大

### 5.2 什么叫只是瓶颈转移

出现下面任一情况，都不能算成功：

1. 吞吐略涨，但 `wait_count / wait_duration` 明显恶化
2. queue 回落了，但 DB 锁等待明显上升
3. batch 更大了，但 `p95` 波动明显放大
4. 单聊变好，群聊明显回退

### 5.3 什么叫可以停

满足下面现象，说明已经接近当前单机甜点：

1. 再加 worker，吞吐不再涨
2. 再加连接池，DB 内部竞争上升
3. 再加 batch / flush，throughput 不升反降
4. 单聊和群聊都开始出现波动放大

## 6. 我给自己的执行清单

### P0

1. 给 `mysql_persist` worker 增加 queue / flush / enqueue 相关指标
2. 确认这些指标能被现有 metrics 采集脚本拿到
3. 用当前推荐停点跑一轮新基线

### P1

1. 先完成 `maxOpenConns` 三档扫描
2. 再完成 `workerCount` 三档扫描
3. 输出一版“小范围甜点区判断”

### P2

1. 再完成 `batchSize` 和 `flushInterval` 微调扫描
2. 对比单聊 / 群聊两条线是否一致
3. 写出最终推荐停点

### P3

1. 如果 `mysql_persist` 已经明显逼近甜点区，停止继续堆参数
2. 把优化主线切换到结构层

---

## 7. 最终交付物

这轮执行结束后，应该产出 3 类东西：

1. `mysql_persist` 参数扫描实验报告
2. 最终推荐配置文件
3. 下一阶段方向判断

下一阶段方向只能是二选一：

1. `mysql_persist` 还有余量，继续细扫
2. `mysql_persist` 已经接近甜点，转去做链路轻量化和结构优化
