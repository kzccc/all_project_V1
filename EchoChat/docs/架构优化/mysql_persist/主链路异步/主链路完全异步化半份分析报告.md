# EchoChat `mysql_persist` 主链路完全异步化半份分析报告

## 1. 这半份报告先回答什么

这半份先回答 4 个问题：

1. 当前版本的主链路架构到底是什么样。
2. 当前 `mysql_persist` 真实做法是什么，不是名字上的“异步”，而是代码语义上的真实行为。
3. 如果要把主链路的落库彻底异步化，现有批量落库 worker 是冲突、还是可兼容。
4. 如果要做，当前版本最适合怎么做，以及先能预见到的缺点是什么。

这半份**先不展开**：

1. 详细压测矩阵
2. 监控项完整清单
3. 回滚预案
4. 灰度切流步骤

这些更适合放在下半份执行计划里。

---

## 2. 当前版本主链路的真实架构

先把当前消息链路按代码真实行为收一下。

### 2.1 入口链路

当前 websocket 客户端发消息后，并不是直接写 MySQL，而是：

1. 前端 websocket 把消息送到后端
2. 后端把消息写入 Kafka chat topic
3. Kafka consumer 再统一消费并执行业务处理

对应代码：

1. `Client.Read()` 在 Kafka 模式下调用 `PublishChatMessage(...)`
2. `KafkaServer.ConsumeClaim()` 消费 chat topic
3. `handleConsumedMessage()` 做主业务处理

这意味着当前系统已经是：

**`WS -> Kafka -> consumer 主链路 -> MySQL/Redis/WebSocket`**

而不是传统的：

**`WS -> 业务线程 -> MySQL`**

所以你现在说“主链路完全异步化”，本质上不是把 websocket 改异步，而是要把：

**consumer 在 commit 前必须同步等待的 MySQL 落库动作挪掉。**

---

## 3. 当前 `mysql_persist` 的真实做法

## 3.1 现在不是“consumer 直接执行 SQL”

当前 consumer 在拿到消息后，会先做：

1. 反序列化
2. `message_id` 归一化
3. `session_seq` 分配
4. 构造 `model.Message`
5. 进入 `saveKafkaMessage(...)`

这里最重要的点是：

**consumer 主协程不会直接执行 insert SQL。**

它会把消息投给一个进程内的批量落库 worker 池。

---

## 3.2 现有批量落库 worker 的结构

当前 `kafkaMessagePersister` 的结构是：

1. 启动多个 worker channel
2. 每条消息按 `conversation scope` 哈希到固定 worker
3. worker 在内存里攒 batch
4. 满批次或超时后执行一次批量 `INSERT INTO message`

这个设计有两个明确目的：

1. 让同一会话尽量落到同一 worker，减少乱序和竞争
2. 用批量 insert 摊薄单条写库成本

这一层从“执行线程模型”看，确实已经是异步 worker。

但这还不是主链路异步化。

---

## 3.3 当前 `mysql_persist` 的真实语义是：异步执行 + 同步等待结果

这件事是当前判断里最关键的点。

`saveKafkaMessage(...)` 虽然会先把 job 投到 worker channel，但随后立刻：

1. `result := <-job.result`
2. 等待 worker 返回
3. 拿到 `alreadyProcessed / persisted / err`
4. 主链路再决定后续是否继续

所以当前 `mysql_persist` 的真实语义不是 fire-and-forget，而是：

**worker 异步执行 SQL，但 consumer 主协程同步等待这次落库结果。**

这意味着：

1. `mysql_persist` 仍然在 commit 前主链路里
2. 现在的 worker 只是把 SQL 执行线程和 consumer 线程拆开了
3. 但没有把“消费成功语义”从“真实写库成功”上摘下来

换句话说，当前版本只是：

**把 MySQL 落库做成了批量化的同步 RPC。**

还不是主链路真正异步。

---

## 3.4 当前 duplicate 判定强依赖落库结果

当前 worker 在批量 insert 时，如果打到唯一键冲突，会走 fallback：

1. 单条重试 insert
2. 命中重复键后查现有 message
3. 如果现有记录 `status = sent`，直接判定 `alreadyProcessed = true`
4. 如果现有记录还没 `sent`，判定 `alreadyProcessed = false`，允许后续补分发

这个语义非常重要。

当前 consumer 不是简单地“写库成功就继续”，而是要先拿到：

1. 这是新消息
2. 还是重复消息
3. 重复消息到底该跳过还是重放后续分发

所以当前 `mysql_persist` 实际上承担了两层职责：

1. 消息持久化
2. duplicate decision 的判定入口

这也是为什么你现在不能简单说“把 saveKafkaMessage 改成扔队列就行”。

---

## 3.5 当前 websocket `sent` 回刷虽然是异步的，但它依赖消息已经存在

当前 websocket 写成功以后，会异步把 `status=sent` 扔给另一个批量更新 worker。

这条线已经是：

**后置异步回刷**

但它成立的前提是：

**message 表中那条记录已经存在。**

因为当前 `status_update` 只是：

1. 按 uuid 批量 `UPDATE message SET status = sent`
2. 没有检查 “本次是否实际更新到行”
3. 也没有 “消息还没插入，稍后再补更新” 这一层语义

所以如果你让 websocket 分发先发生，而真实 insert 还没发生，就会出现一个很现实的问题：

1. 客户端已经收到了消息
2. `status_update` 已经执行
3. 但因为消息行还没插进去，update 可能更新 0 行
4. 后面也没有补偿逻辑把这条消息再打成 `sent`

这说明：

**当前版本里，`mysql_persist`、websocket 分发、`status=sent` 回刷三者是有先后依赖的。**

只把 `mysql_persist` 抽异步，但不改后两段，语义会断。

---

## 4. 当前如果想把主链路落库“完全异步化”，到底意味着什么

这里要先把词说准。

如果你说的“完全异步化”是指：

**consumer 不再等待真实 MySQL insert 成功，只要把持久化任务安全交出去，就允许 `MarkMessage / Commit`。**

那它带来的变化不是参数调优，而是：

**commit 成功语义改变。**

从现在的：

**“消息已经真实落到 MySQL message 表”**

变成：

**“消息已经被可靠转交给后续持久化系统，哪怕 MySQL 还没真正写入。”**

这个差异是架构级别的，不是实现细节。

---

## 5. 现有批量落库 worker 和“主链路完全异步化”是冲突还是兼容

先给结论：

**不是完全冲突，但也绝对不是直接兼容。**

更准确地说：

1. **按职责看，可兼容**
2. **按当前调用方式看，直接冲突**

---

## 5.1 为什么说“按职责看，可兼容”

现有 worker 有三件事其实很有价值，完全可以复用：

1. 按会话 scope 路由，保证同一聊天流尽量进入同一持久化执行通道
2. 批量聚合 insert，降低 MySQL 单条写入成本
3. duplicate fallback：`sent` 跳过、`unsent` 重放

这三件事放到“异步持久化消费者”里仍然成立。

也就是说，未来就算主链路不再等待 MySQL，**这套 worker 仍然适合做真正写库的下游执行器。**

---

## 5.2 为什么说“按当前调用方式看，直接冲突”

因为当前 worker 的接口本质上是：

**内存队列 + 结果回传 channel + 主链路阻塞等待**

而主链路完全异步化要求的是：

**持久化任务交接后，主链路就能结束，不再等这次 SQL 的返回。**

这两者在“确认点”上是冲突的。

另外，当前 worker 还有两个更硬的问题，使它不能直接承担 commit 前异步确认：

### 5.2.1 它是内存队列，不是持久化队列

现在 worker channel 是进程内存里的 `chan *kafkaPersistJob`。

如果你把主链路改成：

1. 只要成功写进这个 channel 就 commit
2. 然后等 worker 慢慢写 MySQL

那一旦进程在真正 flush 到 MySQL 前崩掉，就会出现：

1. Kafka offset 已经 commit
2. 消息不再重放
3. 但内存里的持久化任务已经丢了
4. 这条消息永久丢失

这在当前语义下是不可接受的。

### 5.2.2 它目前没有独立的“异步确认/补偿模型”

当前 worker 只知道把结果回给调用方。

但真正异步后，worker 需要自己处理：

1. 批量写库成功后的确认
2. 写库失败后的重试
3. duplicate 时是否补分发
4. shutdown 时 drain
5. offset 何时 commit

这些逻辑当前都还在主链路调用方那里。

所以结论是：

**现有 worker 可以保留成下游 SQL 执行器，但不能原封不动地拿来做“commit 前异步确认点”。**

---

## 6. 当前版本最不适合做的方案

最不适合的是下面这个“看起来最省事”的版本：

1. 保留当前 worker
2. `saveKafkaMessage(...)` 只 enqueue，不等 `result`
3. consumer 直接 `MarkMessage / Commit`

这个方案的问题不是“可能有点风险”，而是**语义直接断裂**。

### 6.1 会直接丢可靠性

原因上面已经说了：

1. 现在 worker 队列不持久
2. commit 在前，真实落库在后
3. 进程一挂，中间任务丢失

### 6.2 duplicate decision 没有落点

当前 duplicate 判定发生在真实写库阶段。

如果主链路不等这个结果：

1. consumer 当前轮根本不知道这是新消息还是重复消息
2. 也不知道重复消息是 `sent` 还是 `unsent`
3. 就没法决定这条消息应该跳过还是补分发

### 6.3 websocket / `sent` 语义会断

如果你还保持当前分发路径：

1. 可能先 websocket 成功
2. 后 insert 还没落进去
3. `status=sent` update 先跑了但没更新到行
4. 后面也没人补

所以这个最小改法不应该做。

---

## 7. 当前版本最适合的异步化方向

如果你要的是：

**主链路真正不等 MySQL message 表写入，但又不丢可靠性。**

那最适合当前架构的方向，不是进程内内存队列，而是：

**在 Kafka 里补一层“持久化任务的可靠交接”。**

我更推荐的具体形态是：

**`chat ingress consumer -> produce persist_task topic -> commit ingress offset -> persist consumer -> MySQL batch worker`**

也就是把当前一个 consumer 的职责拆成两段：

1. 上游 ingress consumer 负责“接消息、校验、分配 `session_seq`、生成持久化任务”
2. 下游 persist consumer 负责“真实写 MySQL、duplicate decision、成功后继续分发/缓存”

这样最贴当前现有架构，原因有三个：

1. 你本来就已经用 Kafka 作为主消息总线
2. 现有 worker 可以直接下沉到 persist consumer 里复用
3. commit 前的可靠交接从“内存 channel”升级成了“Kafka durable topic”

---

## 8. 推荐做法的详细设计

下面按“当前代码最小改造方向”来展开。

## 8.1 第一步：把当前主 consumer 改成 ingress consumer

当前 `handleConsumedMessage()` 里，主链路做了太多事：

1. `session_seq`
2. `mysql_persist`
3. websocket 分发
4. Redis 消息列表维护

如果要主链路异步化，第一步应该把它收缩成：

1. 反序列化
2. 基础校验
3. `message_id` 归一化
4. `session_seq` 分配
5. 生成一条 **enriched persist task**
6. 写入 `chat_persist` topic
7. 成功后就允许 `MarkMessage / Commit`

这里的 persist task 不能只放原始前端 payload，至少要带上：

1. `message_id`
2. `session_id`
3. `send_id`
4. `receive_id`
5. `conversation_key`
6. `session_seq`
7. `created_at`
8. 规范化后的消息正文
9. 群聊需要的 `send_avatar`
10. 一个明确的 task version

原因很简单：

**下游 persist consumer 不应该重新做一次 `session_seq`。**

否则你会把顺序分配拆乱。

---

## 8.2 第二步：`chat_persist` topic 作为真正的可靠交接层

这里的关键不是“又多了一个 topic”，而是它替代了当前不可靠的内存 worker 队列，成为：

**主链路 commit 前真正可依赖的异步交接点。**

主 consumer 的成功标准变成：

1. persist task 已成功写入 Kafka
2. 才允许 commit 原始 chat topic offset

这样即使进程在 MySQL insert 前挂掉，也不会丢消息，因为 persist task 还在 Kafka。

这里要注意一件事：

当前代码没有用 Kafka transaction。

所以如果发生：

1. persist task 已 produce 成功
2. 原始 offset 还没 commit 就崩

重启后这条原始消息还会再消费一次，可能再次 produce 一个 persist task。

这不是致命问题，但会带来**重复 persist task**。

不过这件事当前是可控的，因为 message 表本来就有 `uuid` 唯一键，duplicate decision 也已经存在。

也就是说：

**当前系统更适合先走“至少一次 + 幂等落库”模型，而不是一开始就硬上 Kafka transaction。**

---

## 8.3 第三步：现有批量落库 worker 下沉到 persist consumer

这一步是“兼容现有 worker”的核心。

做法不是推翻当前 worker，而是改它的挂载位置和接口语义。

新的关系应该是：

1. persist consumer 从 `chat_persist` 拉消息
2. 按 `conversation scope` 或现有 shard 规则把任务送入 persist worker
3. worker 继续 batch insert `message`
4. duplicate 时继续沿用当前 fallback 逻辑

但需要改 3 件事：

1. worker 的输入不再是 `kafkaPersistJob{message,resultChan}`
2. worker 的输出不再回到原 ingress consumer
3. worker 的结果改为回给 **persist consumer 自己**，由 persist consumer 决定后续 commit / retry / dispatch

也就是说，这里要把当前 worker 从：

**进程内同步 RPC worker**

改成：

**异步持久化阶段的执行 worker**

这两者逻辑看起来很像，但语义已经不是一回事。

---

## 8.4 第四步：duplicate decision 继续留在 persist 阶段，不要前移

这点非常关键。

当前 duplicate 逻辑其实很有价值，异步化以后仍然应该保留在真正写库阶段：

1. insert 成功，说明是新消息
2. 命中重复键，查现有 message
3. 若现有记录 `status = sent`，说明这条消息已经完整处理过，可以跳过
4. 若现有记录还未 `sent`，说明可能只是“已经写库但没完成后续分发”，应该补后续动作

这个语义非常适合作为异步化后的崩溃恢复逻辑。

因为它天然支持下面这种场景：

1. persist consumer 已经 insert 成功
2. 但在 websocket 分发或 offset commit 前崩掉
3. 任务重放后命中 duplicate
4. 如果状态还是 `unsent`，继续补分发
5. 如果状态已经 `sent`，直接跳过

所以这里不是要废掉当前 duplicate 逻辑，而是应该把它保留下来，变成异步化后的恢复锚点。

---

## 8.5 第五步：websocket 分发不要再留在 ingress 主链路里

这是很多人最容易漏掉的一点。

如果你只把 `mysql_persist` 从 ingress 主链路摘掉，但还保留：

1. ingress consumer 里直接 websocket 分发
2. 甚至继续给 `messageBackUUID`

那链路会变得非常奇怪：

1. 用户先收到消息
2. 但 MySQL 里这条消息可能还不可见
3. `sent` 更新也可能抢跑失败

所以只要你做的是“主链路真正不等 MySQL 落库”，那 websocket 分发最稳的放置点就应该变成：

**persist 成功之后**

具体有两种可选方式：

### 方式 A：persist consumer 写库成功后直接分发

优点：

1. 最省改造量
2. 当前 `handleConsumedGroupMessage`、单聊分发逻辑更容易复用

缺点：

1. persist consumer 会同时承担“写库 + 分发”
2. 它的 backlog 里会混有 DB 压力和分发压力

### 方式 B：persist 成功后再发一个 `dispatch_task` topic

优点：

1. 责任更清晰
2. `persist` 和 `dispatch` 各自可独立扩缩
3. 监控和压测边界更干净

缺点：

1. topic 和 consumer 又多一层
2. 改造面更大

对当前版本，我更建议：

**先走方式 A 落第一版，先把 ingress 主链路从 MySQL 解开。**

等第一版跑稳了，再考虑把 `dispatch` 继续拆出去。

---

## 8.6 第六步：Redis 消息列表维护最好一起后移或直接收缩

当前 ingress consumer 里还有：

1. 单聊 `redis_read -> append -> redis_write`
2. 群聊 cache append 或 invalidate

如果主链路都已经异步化了，这部分更不应该留在 ingress 阶段。

最稳妥的做法是：

1. 第一版先把 Redis 增量维护从 ingress 主链路移除
2. 群聊先保留 invalidate
3. 单聊优先走“历史查询回源 MySQL”

原因是：

1. 主目标是先把 commit 前链路做轻
2. Redis 增量写本身不是可靠性锚点
3. 在异步 persist lag 存在时，增量 cache 更容易和 DB 可见性不一致

---

## 8.7 第七步：group async pipeline 不能和新的异步 persist 双重排队

当前群聊有一层实验性 `group async pipeline`：

1. ingress consumer 先 enqueue 群消息
2. group async worker 里再 `saveKafkaMessage`
3. 再做群分发

如果你再引入 `chat_persist` topic，就不能保持现在这种“双排队”关系不变。

否则会出现：

1. ingress 已经异步化一次
2. group async 又在进程内异步化一次
3. 排队点重复
4. 顺序边界和 backlog 判断都会变脏

所以更合理的做法是：

1. 保留 group 的 shard 规则
2. 但把 shard 语义下沉到 persist task 的 key 或 persist worker route
3. 不要再让 ingress 层保留一层 group 内存队列

一句话说：

**做了真正的异步 persist 之后，当前 group async pipeline 应该被吸收进新的 persist 阶段，而不是继续独立叠一层。**

---

## 9. 做完之后最现实的缺点

完全异步化不是白拿吞吐，它会明确带来代价。

## 9.1 用户看到消息了，但历史查询短时间可能还查不到

这是最直观的变化。

当前语义更接近：

1. 落库成功
2. 再分发
3. 所以前端一旦收到，大概率 DB 里已经有

改成异步 persist 后，如果你把分发也放在 persist 成功后，这个问题会小很多。

但只要存在异步队列 lag，就一定存在：

1. Kafka ingress 已经 commit
2. persist topic 还没完全消费完
3. 这条消息短时间内还没出现在 MySQL 历史查询里

这会把系统从“更接近同步可见”拉成“明显最终一致”。

---

## 9.2 故障排查会更难

现在一条消息基本在一个 consumer 链路里就能看完。

异步化后至少会变成两段：

1. ingress
2. persist

后面如果 dispatch 再拆，就是三段：

1. ingress
2. persist
3. dispatch

一旦用户说“我发了消息但没看到”，就要分别排查：

1. chat topic 是否收到
2. persist task 是否成功产生
3. persist topic 是否积压
4. message 表是否落了
5. dispatch 是否执行
6. `status=sent` 是否回刷

复杂度会显著上升。

---

## 9.3 至少一次语义下，重复任务会增多

如果不上 Kafka transaction，只靠：

1. produce persist task
2. 再 commit ingress offset

那么 crash 时就会出现重复 task。

这本身可以靠 `message.uuid` 唯一键和 duplicate decision 吃掉，但副作用是：

1. duplicate 查询量会上升
2. 重放路径更常见
3. 你的监控和日志里“重复消息”会比现在多

这不是 bug，而是异步化后正常要接受的代价。

---

## 9.4 `session_seq` 的空洞窗口会更明显

当前 `session_seq` 是在写 MySQL 之前先分配的。

只要在分配后到真正 insert 前的窗口里发生重试、重放、重复 task，就可能出现：

1. 某个顺序号被占用但最终没有成为可见消息
2. 或者同一 `message_id` 在重试过程中带着不同的候选 `session_seq`

现有系统本来就不是严格无洞序号。

但异步层级变多以后，这个窗口会被拉长，现象会更明显。

如果后面你对“连续无洞序号”有硬要求，那就不是只改 persist 能解决的事了，而是要重做 seq 分配模型。

---

## 9.5 需要新的 backlog 监控，否则会变黑盒

现在的慢主要能在 consumer stage 上看到。

改成异步后，如果不补这些指标，很快会变成黑盒：

1. ingress produce persist_task 成功率
2. persist topic lag
3. persist worker queue depth
4. persist flush batch size
5. duplicate replay 次数
6. persist success 到 websocket dispatch 的延迟

如果没这套指标，你只会知道“主链路看起来轻了”，但不知道压力是不是只是搬家了。

## 10. 这半份先收一个明确结论

基于当前版本代码，我的判断是：

### 10.1 当前 `mysql_persist` 不是完全异步

它只是：

**批量 worker 执行 SQL + consumer 同步等结果。**

### 10.2 现有批量落库 worker 不是完全冲突，但不能直接拿来做主链路异步确认

更准确地说：

1. 它的 batch insert / scope 路由 / duplicate fallback 都值得保留
2. 但它现在是内存队列 + 同步回包模型，不能直接承担 commit 前异步确认点

### 10.3 真正适合当前架构的做法，是先补一层 Kafka 持久化任务 topic

也就是：

1. ingress consumer 只负责校验、分配 `session_seq`、产出 persist task
2. commit ingress offset
3. persist consumer 再复用现有批量 worker 去真正写 MySQL
4. persist 成功后再做 duplicate decision、websocket 分发、缓存维护

### 10.4 最不该做的，是直接把当前 `saveKafkaMessage` 改成 fire-and-forget

因为那会同时打断：

1. 可靠性
2. duplicate decision
3. websocket `sent` 回刷语义
