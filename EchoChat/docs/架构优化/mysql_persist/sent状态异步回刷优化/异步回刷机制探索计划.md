# EchoChat 异步回刷机制探索计划

## 1. 目标

这次探索只聚焦 `status=sent` 的异步回刷链路，目标不是单纯把某个参数调大，而是回答下面 3 个问题：

1. 当前默认值为什么是现在这样。
2. 哪些参数值得调，调大或调小分别会带来什么现象。
3. 单机下应该用什么测压与观测方式，才能判断“当前值是否合适”。

本次讨论口径限定为单机场景，不讨论多机路由、跨实例回推和分布式补偿。

---

## 2. 当前实现现状

当前异步回刷链路的核心逻辑在 `internal/service/chat/status_update.go`：

- `workerCount = 4`
- 每个 worker 自己有一条 `chan statusUpdateJob`
- 每条队列长度是 `4096`
- `batchSize = 128`
- `flushInterval = 5ms`
- flush 时执行：

```sql
UPDATE message
SET status = sent
WHERE uuid IN (...)
  AND status <> sent;
```

当前机制的设计意图是：

1. websocket 写成功后，不再同步执行单条 `UPDATE`
2. 先把状态更新任务扔进后台队列
3. 由 worker 做批量合并和去重
4. 再统一回刷 MySQL

这条链路已经比“每条消息写完 websocket 就立刻打一次 DB”要好很多，但它还不是一个已经证明最优的方案。当前默认值更多是一个工程折中值，而不是经过系统化参数扫描得到的单机最优解。

---

## 3. 当前最关键的风险点

### 3.1 worker 太少的风险

如果状态更新进入速度长期高于 4 个 worker 的处理速度，会出现：

- worker 队列 backlog 持续增长
- 越来越多 flush 是“攒满 128 条才刷”
- 队列逐渐接近 `4096`
- 前台 `enqueue` 被反压
- websocket 写协程不再轻量
- 最终会把问题重新传回实时消息主链路

最直观的外部表现通常是：

- `ws_sendback_enqueue_duration` 上升
- `ws_sendback_queue_length` 上升
- 单聊或群聊的 p95 抬高

### 3.2 worker 太多的风险

worker 并不是越多越好。worker 过多时，常见问题是：

- 每个 worker 分到的任务太少
- 很多 flush 不是因为 batch 满，而是因为 `5ms` 到时了
- 平均 batch size 变小
- SQL 次数变多，批量效率下降
- MySQL 并发更新变多，连接池等待和 InnoDB 竞争加重

也就是说，worker 数拉高以后，可能只是把瓶颈从“队列积压”转移成“MySQL 并发竞争”。

### 3.3 当前现有观测还不够完整

当前 `/metrics` 里已经有：

- `echochat_ws_sendback_queue_length`
- `echochat_ws_sendback_enqueue_duration_seconds`
- `echochat_ws_write_duration_seconds`
- `echochat_ws_status_update_duration_seconds`
- `echochat_mysql_open_connections`
- `echochat_mysql_in_use_connections`
- `echochat_mysql_idle_connections`
- `echochat_mysql_wait_count_total`
- `echochat_mysql_wait_duration_seconds`

但这还不够直接回答“4 个 worker 是否合适”，因为现在缺少 3 类最关键的批处理内部指标：

1. worker 队列深度
2. flush 的真实 batch size
3. flush 是因为“batch 满”还是“timer 到”

另外，`echochat_ws_status_update_duration_seconds` 当前是把一次批量 SQL 的总耗时均摊到每条消息上的结果，它更适合看“单条平均摊销成本”，不适合直接拿来判断单次 flush 的真实 batch 效率。

## 4. 这次要重点调整的参数

本次建议优先调 5 个主参数，另外保留 2 个辅助参数用于收尾微调。

这里专门把 MySQL 连接池大小也列进主参数，不是因为它一定已经是瓶颈，而是因为你现在讨论的是单机榨干性能。在单机场景里，异步回刷 worker 数、批量大小、flush 节奏和 MySQL 并发承载能力必须一起看，不能只调应用侧，不调 DB 入口宽度。

需要注意：

1. `workerCount / batchSize / flushInterval / queueSize` 当前还是写死在 `status_update.go` 里的常量或初始化值。
2. MySQL 连接池当前已经配置化，配置项是：
   - `maxOpenConns`
   - `maxIdleConns`
   - `connMaxLifetimeMinutes`
   - `connMaxIdleTimeMinutes`
3. 现有多数单机压测配置里，连接池主配置已经在用：
   - `maxOpenConns = 500`
   - `maxIdleConns = 100`

要做系统化扫描，第一步不是直接跑压测，而是先把这 5 个主参数都做到可切换，至少可以通过配置文件或环境变量稳定调整。

### 4.1 `workerCount`

当前值：`4`

它决定：

- 状态更新链路有多少个并行消费 goroutine
- 能同时承接多少条独立 flush 流

它是这次最重要的扫描维度。

### 4.2 `batchSize`

当前值：`128`

它决定：

- 单次 SQL 最多合并多少条状态更新
- 批量写效率上限

batch 太小会导致 SQL 次数过多；batch 太大则可能导致单次等待变长，低压时更难攒满。

### 4.3 `flushInterval`

当前值：`5ms`

它决定：

- batch 在没攒满时最多等多久

它本质上是在“吞吐效率”和“状态回刷时效”之间做折中：

- 更小：更及时，但 batch 更碎
- 更大：batch 更大，但状态落库延迟更大

### 4.4 `queueSize`

当前值：`4096`

它不直接提升处理能力，但影响：

- 削峰能力
- 短时突发下是否容易倒灌前台

注意：`queueSize` 不是吞吐优化主参数，它更像稳定性缓冲参数。只有在确认 worker 处理能力足够、只是突发尖峰导致短时拥塞时，才适合后调。

### 4.5 `maxOpenConns`

当前常用值：`500`

它决定：

- 单机 MySQL 连接池允许的最大并发打开连接数
- 异步回刷 worker、消息落库、历史查询、登录鉴权等路径争用数据库时的总入口宽度

为什么这项必须放进主扫描参数里：

1. 如果 `workerCount` 提上去了，但 `maxOpenConns` 太小，worker 实际会卡在等连接，应用层看起来像“worker 不够快”，本质却是连接池卡住了。
2. 如果 `maxOpenConns` 拉得过大，单机 MySQL 可能没有真正更快，反而会出现更多并发 SQL 争抢 CPU、buffer pool、row lock，最后把延迟和抖动一起拉高。

在“榨干单机”的语境里，它的作用不是单纯把数字调大，而是让数据库入口宽度和应用侧并发度匹配。

对你当前机器更合适的理解方式是：

- `maxOpenConns` 太小：应用想并发，DB 不让进
- `maxOpenConns` 太大：应用和 DB 一起陷入过度并发
- `maxOpenConns` 合适：worker 能吃满，连接池等待不明显，DB 也没有被无意义并发打炸

### 4.6 辅助参数：`maxIdleConns` / 连接生命周期

这两类参数不是这次最核心的吞吐主旋钮，但在单机高压时值得顺带观察：

- `maxIdleConns`
- `connMaxLifetimeMinutes`
- `connMaxIdleTimeMinutes`

它们更偏“连接复用质量”和“长时间高压稳定性”：

- `maxIdleConns` 太小，连接频繁创建回收，会增加抖动
- `connMaxLifetime` 太短，连接在高压中频繁轮换，也会制造额外开销

所以这次建议把它们列成辅助调优项，而不是第一批主扫描项。

## 5. 判断“4 个 worker 是否最佳”的核心标准

不能靠拍脑袋判断，只能看下面几个现象。

### 5.1 必看现象

1. backlog 是否持续增长
2. 平均 batch size 是多少
3. 单次 flush SQL 耗时是多少
4. worker 是否经常空闲
5. DB CPU / IO / row lock wait 是否上升
6. MySQL 连接池 `in_use / open / wait_count / wait_duration` 是否抬升

### 5.2 如何解释这些现象

如果 4 个 worker 已经经常攒满 batch，而且 backlog 还在涨，说明偏少。

如果 4 个 worker 大部分 flush 都只攒到很小的 batch 就被 `flushInterval` 打断，说明不一定缺 worker，甚至可能偏多。

如果把 worker 从 `4` 提到 `8` 后 backlog 确实下降了，但 MySQL 等待、连接池等待、InnoDB 锁等待明显上升，那说明只是把瓶颈从“队列积压”挪到了“MySQL 并发竞争”。

如果你把 `maxOpenConns` 从 `500` 提高后，`wait_count` 下降了、batch 更饱满了、吞吐继续上升，而且 DB CPU / row lock wait 仍然稳定，那说明之前连接池入口确实偏窄。

反过来，如果 `maxOpenConns` 提高后，连接池等待没了，但 flush SQL 变慢、row lock wait 上升、整体 `p95` 反而恶化，那说明你只是把瓶颈往 MySQL 内部推深了。

### 5.3 一句话判断标准

- `4` 太少的信号：队列堆积、前台被反压、batch 经常打满
- `4` 太多的信号：batch 太碎、timer flush 占比高、DB 并发等待明显上升
- `4` 合适的信号：队列不持续增长，batch 比较饱满，DB 更新延迟稳定，主链路 p95 没被拖坏

同样地，连接池大小的最短判断标准也可以压缩成一句话：

- 太小：`wait_count / wait_duration` 明显上涨，`in_use` 经常贴着 `maxOpenConns`
- 太大：连接池等待下降了，但 DB CPU、锁等待、单次 flush 耗时明显恶化
- 合适：连接池等待不明显，worker 能稳定吃满，DB 内部竞争没有明显放大

## 6. 本次探索需要补的埋点

要真正把 worker 异步回刷调清楚，建议补下面这些指标。

### 6.1 backlog 指标

建议新增：

- `echochat_status_update_queue_depth{worker}`
- `echochat_status_update_queue_capacity{worker}`

作用：

- 直接看每个 worker 队列是不是长期堆积
- 判断 backlog 是短时波峰还是持续增长

### 6.2 flush 粒度指标

建议新增：

- `echochat_status_update_flush_batch_size`
- `echochat_status_update_flush_duration_seconds`
- `echochat_status_update_flush_total{reason=batch_full|timer|shutdown,result=success|failure}`

作用：

- 看真实 batch size 分布
- 看单次 flush 的真实 SQL 耗时
- 看 flush 到底主要是“攒满触发”还是“时间窗触发”

### 6.3 worker 忙闲指标

建议新增：

- `echochat_status_update_worker_idle_seconds_total{worker}`
- 或更简单的 `echochat_status_update_worker_flush_interval_seconds` / `last_flush_ts`

作用：

- 判断 worker 是长期繁忙还是大部分时间闲着

### 6.4 反压指标

建议新增：

- `echochat_status_update_enqueue_block_duration_seconds`
- `echochat_status_update_enqueue_total{result=success|blocked}`

作用：

- 直接判断后台队列是否已经把前台 websocket 写协程顶住

### 6.5 连接池利用率指标

虽然当前已经有基础连接池指标，但为了做“单机榨干”扫描，建议明确把下面这组指标纳入每轮观测清单：

- `echochat_mysql_open_connections`
- `echochat_mysql_in_use_connections`
- `echochat_mysql_idle_connections`
- `echochat_mysql_wait_count_total`
- `echochat_mysql_wait_duration_seconds`

如果后面愿意再补 1 个聚合视角，建议增加：

- `mysql_in_use_ratio = in_use / maxOpenConns`

它的作用很直接：

- 更容易一眼判断连接池是不是长期贴顶
- 更方便和 `workerCount`、`batchSize` 调整联动对比

如果暂时不想加这么多埋点，最低限度也要先补：

1. `queue_depth`
2. `flush_batch_size`
3. `flush_total{reason=...}`
4. `flush_duration_seconds`
5. MySQL 连接池等待指标

没有这组最小指标，就很难系统判断“worker 数是不是最佳”和“连接池是不是在拦你的单机上限”。

## 7. 测压方式怎么设计

本次探索不建议跳出你现有吞吐压测框架，而是建议在现有框架上增加“异步回刷专项观测”。

也就是说：

- 业务压测口径继续沿用当前单聊 / 群聊最大稳定吞吐的流程
- 但在每个 run 里，额外采集 status update worker 的内部指标和 MySQL 侧指标

### 7.1 压测目标

本次专项不是先追求绝对最大吞吐，而是先回答：

- 当前 worker 参数在哪个区间最稳
- 在给定吞吐目标下，哪个参数组合最不容易把压力反向传回主链路

### 7.2 压测场景

建议仍然按你现有单聊 / 群聊稳定吞吐流程跑，但参数扫描顺序不要乱。

推荐顺序是：

1. 先固定 `maxOpenConns = 500`，扫描 `workerCount`
2. 在较优 `workerCount` 上扫描 `batchSize`
3. 再扫描 `flushInterval`
4. 再看 `queueSize` 是否只需要做削峰缓冲
5. 最后再围绕较优应用参数组合，扫描 `maxOpenConns`

这样做的原因是：

1. 先把应用侧批处理形态定住
2. 再看数据库入口宽度是不是在限制你
3. 结论会比“5 个参数一起乱扫”干净得多

如果你就是想更激进地贴近单机上限，那 `maxOpenConns` 这轮建议至少保留 3 档：

- 保守档：低于当前值
- 基线档：`500`
- 激进档：高于当前值

但是否真的高于 `500` 更优，不能靠经验判断，只能靠下面这些联合现象：

1. 吞吐有没有继续涨
2. `p95` 有没有稳住
3. `wait_count / wait_duration` 有没有明显下降
4. DB CPU、IO、锁等待有没有被明显推高

建议固定两类场景：

1. 单聊场景
2. 群聊场景

原因：

- 单聊更容易看清“每条消息回刷一次”的基础成本
- 群聊更容易放大 fanout 后的写回密度，更接近 status update 的高压场景

### 7.3 压测强度选择

不建议一上来就扫满整条吞吐曲线，建议用 3 档固定 target 做专项比较：

1. 低压档：明显低于当前稳定上限
2. 中压档：接近当前稳定上限
3. 高压档：略高于当前稳定上限

例如：

- 单聊：`0.5x / 0.8x / 1.0x~1.1x 当前稳定吞吐`
- 群聊：`0.5x / 0.8x / 1.0x~1.1x 当前稳定吞吐`

这样做的好处是：

- 能看清参数变化对低压、中压、高压的不同影响
- 不会被整条扫描链路的其他变量淹没

### 7.4 每组参数的 run 结构

建议每组参数都按统一 run 模板执行：

1. 预热 `30s`
2. 正式压测 `180s`
3. 停止发送后额外 drain `30s`
4. 保存本轮 metrics、pprof、系统采样、MySQL 采样

原因：

- 预热可以让 worker 队列、连接池、consumer session 进入稳定态
- `180s` 足够观察 backlog 是否持续增长
- drain 可以看队列能否快速回落到低位，而不是一直拖尾

## 8. 测压脚本应该如何判断

### 8.1 复用现有脚本

压测流量继续使用现有消息压测脚本。

指标采集建议复用：

- `docs/k6_message_test/scripts/message_latency_runner.py`
- `docs/t_K6/scripts/collect_server_metrics.py`

建议方式是：

1. 现有消息压测脚本负责施压
2. `collect_server_metrics.py` 负责同步采 `/metrics`
3. 增加一个轻量 MySQL 采样脚本，按固定间隔记录：
   - `Threads_running`
   - `Threads_connected`
   - `Innodb_row_lock_time`
   - `Innodb_row_lock_waits`
   - `Innodb_buffer_pool_reads`
   - 必要时补 `SHOW ENGINE INNODB STATUS`
4. 额外采系统指标：
   - `pidstat`
   - `iostat`
   - `mpstat`

### 8.2 脚本需要产出的判断结果

每次 run 结束后，脚本至少应输出下面这些结论字段：

- `offered_throughput`
- `observed_throughput`
- `p95_ms`
- `status_update_flush_avg_batch_size`
- `status_update_flush_p95_duration_ms`
- `status_update_timer_flush_ratio`
- `status_update_queue_depth_p95`
- `status_update_queue_depth_end`
- `status_update_enqueue_block_p95_ms`
- `mysql_in_use_conn_peak`
- `mysql_wait_duration_delta`
- `innodb_row_lock_waits_delta`
- `cpu_peak`
- `iowait_peak`

### 8.3 backlog 如何判断

如果已经补了 `queue_depth` 指标，脚本直接判断：

- run 开始、中间、结束三个阶段的 queue depth 是否持续抬高
- 结束 drain 后是否仍然回不下来

推荐判断规则：

- `queue_depth_end` 明显高于 `queue_depth_start`，且 drain 后仍不回落：判定 backlog 持续增长
- `queue_depth` 在压测中有波峰，但 drain 后能回落到低位：判定只是削峰，不算失控

### 8.4 batch 是否合理如何判断

如果已补 `flush_batch_size` 与 `flush_total{reason=...}`，脚本可直接判断：

- `avg_batch_size` 长期接近上限，且 backlog 仍增长：worker 偏少
- `avg_batch_size` 很小，`timer_flush_ratio` 很高：worker 偏多或 flushInterval 偏小

### 8.5 DB 是否被打坏如何判断

脚本应重点看：

- MySQL in-use connections 是否接近上限
- `wait_count` / `wait_duration` 是否显著增加
- InnoDB row lock waits 是否显著增加
- 系统 iowait 是否明显上升

如果 worker 增多以后：

- backlog 下降了
- 但 MySQL wait 和 row lock wait 明显恶化

那就不能判定为优化成功，只能说明瓶颈转移了。

## 9. 推荐的参数扫描顺序

不要一次把所有参数全扫开，否则很难归因。建议按下面顺序逐层探索。

### 9.1 推荐扫描表

先给一版可以直接执行的推荐扫描表。

| 参数 | 当前基线 | 推荐首轮扫描档位 | 主要看什么 | 什么时候继续加 | 什么时候该停 |
| --- | --- | --- | --- | --- | --- |
| `workerCount` | `4` | `2 / 4 / 8 / 12` | backlog、`avg_batch_size`、`timer_flush_ratio`、主链路 `p95` | backlog 仍持续增长，batch 经常打满，DB 等待还稳 | queue 不再涨，但 DB wait / row lock wait 开始明显恶化 |
| `batchSize` | `128` | `64 / 128 / 256` | 单次 flush SQL 耗时、`avg_batch_size`、`timer_flush_ratio` | batch 更大后 SQL 次数下降、吞吐还涨、`p95` 不坏 | batch 继续变大但 flush 更慢、等待更长、`p95` 变差 |
| `flushInterval` | `5ms` | `1ms / 2ms / 5ms / 10ms` | timer flush 占比、状态回刷时效、主链路 `p95` | timer flush 过高，batch 太碎，希望再攒批 | flush 等太久导致状态落库拖尾，或 `p95` 开始变差 |
| `queueSize` | `4096` | `2048 / 4096 / 8192` | queue depth 波峰、drain 后回落速度、enqueue block | 只是短时削峰不够，worker 实际处理能力足够 | backlog 是持续增长，不是瞬时尖峰，这时加 queue 没意义 |
| `maxOpenConns` | `500` | `300 / 500 / 700` | `in_use/open/wait_count/wait_duration`、DB CPU、row lock wait | 连接池等待明显，worker 吃不满，提大后吞吐继续涨 | wait 降了但 DB CPU / 锁等待 / flush 耗时明显恶化 |

这张表的使用方式很简单：

1. 一次只扫一个主参数。
2. 其余 4 个主参数固定在当前最优或当前基线。
3. 每一轮先看“是不是把瓶颈打散了”，再看“是不是只是把压力推给 MySQL”。

### 9.2 第一轮：先扫 `workerCount`

固定：

- `batchSize = 128`
- `flushInterval = 5ms`
- `queueSize = 4096`
- `maxOpenConns = 500`

扫描：

- `workerCount = 2 / 4 / 8 / 12`

目标：

- 先找到 worker 数的合理区间

### 9.3 第二轮：再扫 `maxOpenConns`

固定：

- 第一轮得到的较优 `workerCount`
- `batchSize = 128`
- `flushInterval = 5ms`
- `queueSize = 4096`

扫描：

- `maxOpenConns = 300 / 500 / 700`

目标：

- 判断当前是不是 DB 入口宽度在卡你
- 找到“连接池等待不过高、MySQL 内部竞争也不过高”的单机平衡点

这里为什么把它提前到第二轮：

- 你现在目标是单机榨干性能
- 只扫应用层参数，不扫 DB 连接池宽度，结论会偏保守
- 先把 `workerCount` 和 `maxOpenConns` 对齐，后面扫 `batchSize` 和 `flushInterval` 才更像真实上限

### 9.4 第三轮：在合适 worker 区间内扫 `flushInterval`

固定：

- `batchSize = 128`
- `queueSize = 4096`
- `maxOpenConns = 第二轮较优值`

扫描：

- `flushInterval = 1ms / 2ms / 5ms / 10ms`

目标：

- 看 timer flush 比例、batch size、状态落库延迟的平衡点

### 9.5 第四轮：再扫 `batchSize`

固定前两轮得到的较优值。

扫描：

- `batchSize = 64 / 128 / 256`

目标：

- 找到 SQL 次数与单次 batch 效率的较优平衡

### 9.6 第五轮：最后才看 `queueSize`

扫描：

- `queueSize = 2048 / 4096 / 8192`

目标：

- 判断是否需要更大的削峰缓冲

注意：`queueSize` 只能缓冲，不能创造处理能力，所以它必须放在最后。

## 10. 参数优劣的最终判定规则

每组参数不只看吞吐，还要同时看主链路与回刷链路。

### 10.1 判定优先级

第一优先级：

- 不破坏你当前“稳定吞吐”的业务判定结果
- 不明显拉坏消息 p95

第二优先级：

- backlog 不持续增长
- queue depth 在 drain 后能明显回落

第三优先级：

- batch 比较饱满
- timer flush 占比不过高

第四优先级：

- MySQL wait、row lock wait、iowait 不显著恶化

### 10.2 推荐的最终选择标准

优先选择满足下面条件的参数组：

1. 单聊和群聊主链路 p95 不变差，或只小幅波动
2. status update backlog 不持续增长
3. 平均 batch size 不太碎
4. MySQL 等待没有明显恶化
5. 在上述条件下 observed throughput 更高，或者同吞吐下资源更省

如果某个参数组只是把吞吐抬高一点，但 MySQL 等待暴涨、主链路 p95 变差，那不应算成功方案。

## 11. 这份计划的落地结论

本次异步回刷探索的核心不是“把 worker 从 4 改到更大”，而是先把判断体系建立起来。

真正要回答的问题是：

- 当前 4 个 worker 是真的不足，还是已经够用
- 当前 `5ms` 是太保守，还是已经是合理折中
- 当前 `128` 的 batch 是太小，还是已经接近 MySQL 批量更新效率的甜点

因此，这次工作的优先顺序应该是：

1. 先补齐 worker 回刷链路的内部观测指标
2. 再用固定 target 的专项压测做小范围参数扫描
3. 最后再把结果带回完整吞吐压测链路验证

如果没有 backlog、batch size、flush reason 这些关键指标，只看最终吞吐和 p95，是不足以回答“4 个 worker 是否最佳”的。

## 12. 最简执行版

如果先只做一轮最小成本实验，建议直接这样执行：

1. 补 5 类关键指标：
   - `queue_depth`
   - `flush_batch_size`
   - `flush_duration_seconds`
   - `flush_total{reason=...}`
   - MySQL 连接池等待指标
2. 固定 `batchSize=128`、`flushInterval=5ms`、`queueSize=4096`、`maxOpenConns=500`
3. 先只扫 `workerCount=2/4/8/12`
4. 如果 worker 合理区间出来了，再补扫 `maxOpenConns=300/500/700`
5. 在单聊与群聊各选 3 档 target：
   - `0.5x`
   - `0.8x`
   - `1.0x~1.1x 当前稳定吞吐`
6. 每组压 `180s + 30s drain`
7. 同时采：
   - `/metrics`
   - `pidstat`
   - `iostat`
   - MySQL wait / row lock 指标
8. 按下面规则出结论：
   - backlog 涨且 batch 常满：worker 偏少
   - batch 碎且 timer flush 占比高：worker 偏多或 flushInterval 偏小
   - `in_use` 贴顶、`wait_count` 涨：连接池偏小
   - 连接池等待下降但 DB CPU / 锁等待恶化：连接池偏大或 worker 并发过强
   - backlog 下降但 MySQL 等待明显恶化：只是瓶颈转移
   - backlog 稳、batch 饱满、主链路 p95 稳：该参数组更优

如果想把这版最简执行再压缩成一句话，可以记成：

**先用 `workerCount` 找应用侧并行度，再用 `maxOpenConns` 找 DB 入口宽度，最后才微调 `flushInterval / batchSize / queueSize`。**

这才是判断异步回刷参数是否合理、并尽量逼近单机上限的最小闭环。
