## 4. 风险与建议

- 当前 Redis 去重仍是列表扫描，复杂度为 O(n)。本次按执行计划采用最小闭环，没有引入更重的索引结构。
- 旧缓存数据如果缺少 `message_id`，当前已用 `session_seq` 做兼容兜底，但它仍弱于 `message_id`。
- 建议后续在压测或重复消费演练时重点观察新增日志事件 `message.cache_duplicate_skipped`，确认 Redis 去重命中是否符合预期。

## 5. 结论

本次“前端 + Redis 去重”最小闭环已落地完成。当前状态下，同一条消息在重复分发场景里具备：

- 前端展示层幂等
- Redis 列表缓存层幂等
- 协议层可追踪的稳定 `message_id`
