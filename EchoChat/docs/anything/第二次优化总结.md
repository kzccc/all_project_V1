## 2. 第二次优化开始前，系统性能停在哪里

第二次优化开始前，系统继承的是第一次优化结束时的结果。

当时统一对外口径是：

1. 单聊真实容量：约 `350 msg/s`
2. 群聊真实容量：约 `4050 delivery/s`
3. 群聊折算消息数：约 `169 group_msg/s`

这意味着什么：

1. Kafka 基本可用了，基础可靠性和 commit 机制已经搭起来了
2. 但单聊长期停在 `350 msg/s` 左右，继续加压只会让长尾越来越重
3. 群聊虽然明显高于单聊，但也已经出现了平台和固定成本

所以第二次优化的起点不是“Kafka 不能用”，而是：

**Kafka 能用，但 consumer 后半段同步业务链路太重。**

当时最核心的问题有两个：

1. `session_seq` 每条消息都可能打一趟 MySQL
2. `mysql_persist` 每条消息同步单条落库

换句话说，第二次优化的主线从一开始就不是 producer，而是：

**消费后半段的固定成本拆解。**

## 3. 第一阶段：先做排查，不急着改代码

### 3.1 时间点

`2026-04-06 20:02` 到 `2026-04-06 22:06`

对应文档：

### 3.2 为什么先做这一阶段

当时虽然已经知道：

1. 单聊大约卡在 `350 msg/s`
2. 群聊大约卡在 `4000 delivery/s`

但还不能回答下面这些关键问题：

4. 还是连接池在等连接
5. 是 Redis 命令本身慢
6. 还是 Redis 的业务模式在放大

所以这一步的目标不是提吞吐，而是：

**先把盲区量出来。**

主要补了三类指标：

1. websocket 下游写回链路

真正被锁定出来的热点是：

1. `session_seq`

这一阶段的实际意义是：

**把第二次优化的方向从“怀疑很多东西”收敛成了“盯住 session_seq 和 mysql_persist”。**

## 4. 第二阶段：先用实验开关验证 `session_seq` 是不是真瓶颈

### 4.1 时间点

`2026-04-06 23:28`

对应文档：

1. `第二次优化_第二次排查_分析session_seq.md`

### 4.2 为什么先打这块

前一阶段已经证明：

1. websocket 不是主瓶颈
2. Redis 命令本身不是主瓶颈
3. 连接池等待也不是主瓶颈

那最值得先验证的，就是：

**每条消息都查一次 MySQL `MAX(session_seq)` 到底是不是一块大木板。**

### 4.3 这一阶段做了什么

开了实验开关：

1. `sessionSeqRedisOnlyExperimental = true`

改法是：

1. `session_seq` 只走 Redis 递增
2. 暂时不再每条消息查 MySQL floor

这一步是旁路实验，不是正式落地。

### 4.4 为什么要这么做

因为这个实验最干净。

如果旁路掉 MySQL floor 以后：

1. `session_seq` stage 明显下降
2. 吞吐明显上升
3. `p95` 明显回落

那就可以直接坐实：

**`session_seq` 是真的大瓶颈，而不是小噪音。**

### 4.5 实际效果怎么样

结果很明确。

阶段耗时：

单聊 `960`：

1. `session_seq: 3.057 ms -> 0.186 ms`
2. `total: 11.947 ms -> 5.933 ms`

群聊 `5760`：

1. `session_seq: 2.526 ms -> 0.074 ms`
2. `total: 11.728 ms -> 5.138 ms`

群聊 `11520`：

1. `session_seq: 3.811 ms -> 0.080 ms`
2. `total: 13.744 ms -> 5.100 ms`

吞吐和 `p95`：

单聊 `960`：

1. `178.607 -> 323.551 msg/s`
2. `p95: 29465.9ms -> 11276.65ms`

群聊 `5760`：

1. `1897.939 -> 3914.862 delivery/s`
2. `p95: 13125ms -> 717.1ms`

群聊 `11520`：

1. `1681.775 -> 3995.107 delivery/s`
2. `p95: 39712.1ms -> 6097ms`

但单聊高压档没有彻底被救回来：

单聊 `2880`：

1. `293.201 -> 264.857 msg/s`
2. `success: 0.918542 -> 0.840458`
3. `p95: 59489.4ms -> 58829.5ms`

### 4.6 这一阶段的结论

这一步坐实了两件事：

1. `session_seq` 确实是大瓶颈
2. 但它不是单聊高压下的最后一块木板

因为一旦把 `session_seq` 打轻以后，下一块更重的东西立刻冒出来了：

**`mysql_persist`**

## 5. 第三阶段：再用旁路实验验证 `mysql_persist` 是下一块主瓶颈

### 5.1 时间点

`2026-04-06 23:29`

对应文档：

1. `第二次优化_第二次排查_分析mysql_persist(包含打 status=sent).md`

### 5.2 为什么接着查这块

前一步已经把逻辑收得很清楚：

1. `session_seq` 去掉以后，主链路还很重
2. 此时 `mysql_persist` 已经占了 `total` 的绝大多数

所以这一轮要回答的问题就是：

**消息落库整段成本，到底是不是当前同步主链路里的第二块大木板。**

### 5.3 这一阶段做了什么

同时打开两个实验开关：

1. `sessionSeqRedisOnlyExperimental = true`
2. `mysqlPersistNoopExperimental = true`

也就是：

1. 先旁路 `session_seq` 的 MySQL floor
2. 再旁路真实消息落库

### 5.4 为什么要这么做

如果只是拿原始版本直接旁路 `mysql_persist`，结论会混入 `session_seq` 的影响。

而先把 `session_seq` 拿掉，再旁路 `mysql_persist`，就能更干净地回答：

**`mysql_persist` 本身值不值得优先优化。**

### 5.5 实际效果怎么样

结果非常激进。

阶段耗时几乎被打平：

单聊 `2880`：

1. 只旁路 `session_seq` 时：`mysql_persist = 9.223 ms`, `total = 9.996 ms`
2. 再旁路 `mysql_persist` 后：`mysql_persist = 0`, `total = 0.109 ms`

群聊 `11520`：

1. 只旁路 `session_seq` 时：`mysql_persist = 4.600 ms`, `total = 5.100 ms`
2. 再旁路 `mysql_persist` 后：`mysql_persist = 0`, `total = 0.256 ms`

吞吐和 `p95`：

单聊 `960`：

1. `323.551 -> 795.332 msg/s`
2. `p95: 11276.65ms -> 2ms`

单聊 `2880`：

1. `264.857 -> 2377.092 msg/s`
2. `success: 0.840458 -> 1.0`
3. `p95: 58829.5ms -> 2ms`

群聊 `5760`：

1. `3914.862 -> 4467.238 delivery/s`
2. `p95: 717.1ms -> 2ms`

群聊 `11520`：

1. `3995.107 -> 6605.234 delivery/s`
2. `p95: 6097ms -> 4ms`

### 5.6 这一阶段的结论

这一步也直接坐实了：

1. `mysql_persist` 是当前主瓶颈之一
2. 在 `session_seq` 打掉以后，它就是最重的同步主链路阶段

但这一步也有边界：

因为旁路真实落库的同时，也顺带旁路了后面的 `status=sent` 更新负担。

所以这一步的正确理解是：

**它证明了“消息落库相关整段成本”是核心瓶颈，但它本身不是可直接上线的最终效果。**

## 7. 第五阶段：正式落地 `session_seq` + `mysql_persist`

### 7.1 时间点

`2026-04-07 14:32` 左右形成第一版正式结果

对应文档：

1. `第二次优化_第二次排查后session_seq和mysql_persist优化测压结果.md`

### 7.2 这一步做了什么

正式改造落了两块：

第一块，`session_seq` 正式化：

1. 不再每条消息查 `message` 表 `MAX(session_seq)`
2. 改成 Redis 主分配
3. 用 Lua 保证初始化原子性
4. 新增 `conversation_sequence` 高水位表
5. 历史会话只做一次 floor 迁移

第二块，`mysql_persist` 轻量化：

1. Gorm 打开 `SkipDefaultTransaction`
2. Gorm 打开 `PrepareStmt`
3. 单条同步写改成分 worker 的批量落库
4. 按会话 scope 分流，减少多个会话挤同一出口
5. 正常路径只走 insert
6. duplicate 才走慢路径

### 7.3 为什么要这么优化

因为这一步在做的是：

1. 把每条消息打一趟 MySQL 的 `session_seq` 成本拿掉
2. 把每条消息同步单条 insert 的落库成本改成批量摊销

也就是：

**把 consumer 主链路里最重的两块固定成本同时打轻。**

### 7.4 实际效果怎么样

正式落地后的第一版容量结果是：

1. 单聊：`349 msg/s`
2. 群聊：`4746 delivery/s`
3. 群聊折算：`198 group_msg/s`

相对第二次优化起点：

1. 单聊：`350 -> 349`，基本持平
2. 群聊：`4050 -> 4746`，提升约 `17.1%`
3. 群聊折算：`169 -> 198`，提升约 `17.1%`

对应最佳稳定档的 `p95`：

1. 单聊：`348.56 msg/s @ p95 54038.2ms`
2. 群聊：`4745.605 delivery/s @ p95 5913ms`

### 7.5 这一阶段怎么理解

这一步最关键的现象是：

1. 群聊已经明显涨了
2. 单聊却几乎没动

这说明：

1. `session_seq` 和 `mysql_persist` 这条路是对的
2. 但单聊的新瓶颈已经不再只是这两块























所以第二次优化到了这里，方向又继续往后推进了：

**开始拆 `message` 表模型和 `status=sent` 路径。**

## 8. 第六阶段：收敛 `message` 表模型 + 异步化 `status=sent`

### 8.1 时间点

`2026-04-07 15:29` 左右形成第二版正式结果

对应文档：

1. `第二次优化_第二次排查后session_seq和mysql_persist优化测压结果.md`

### 8.2 这一步做了什么

继续落了两块关键优化：

第一块，收敛 `message` 表查询和索引模型：

1. 给 `message` 表增加 `conversation_key`
2. 历史查询统一走 `conversation_key + session_seq`
3. 建复合索引 `idx_message_conversation_seq`
4. 删除旧的 `send_id / receive_id / session_id / session_seq` 单列索引

第二块，`status=sent` 改成异步批量更新：

1. websocket 写协程不再同步执行 `UPDATE message SET status=sent`
2. 改成后台多 worker 批量更新
3. 写协程只负责真正发消息，然后把 uuid 投到状态更新器

### 8.3 为什么要这么优化

因为前一阶段已经说明：

1. `session_seq` 不再是第一瓶颈
2. `mysql_persist` 也已经被打轻一轮
3. 但单聊还是卡在 `350` 左右

所以这一步针对的是两类新的固定成本：

1. `message` 大表旧索引模型对写入的放大
2. websocket 成功路径上的同步 `status=sent` 更新

### 8.4 实际效果怎么样

这一步之后的新容量结果是：

1. 单聊：`387 msg/s`
2. 群聊：`5409 delivery/s`
3. 群聊折算：`225 group_msg/s`

相对第二次优化起点：

1. 单聊：`350 -> 387`，提升约 `9.3%`
2. 群聊：`4050 -> 5409`，提升约 `20.8%`
3. 群聊折算：`169 -> 225`，提升约 `20.8%`

对应常规测压最佳档的 `p95`：

1. 单聊：`386.948 msg/s @ p95 54129.65ms`
2. 群聊：`5408.771 delivery/s @ p95 2777ms`

### 8.5 这一阶段的结论

这一步是第二次优化里第一个对单聊也有明确正收益的阶段。

结论可以收成一句话：

**`conversation_key + 复合索引 + status=sent 异步批量更新`，终于把单聊从长期卡住的 `350` 平台往上推到了 `387`。**

## 10. 第八阶段：继续解释为什么单聊还是只到 `417`

### 10.1 时间点

`2026-04-07` 晚间继续拆解

对应文档：

1. `第二次优化_第二次排查后session_seq和mysql_persist优化测压结果.md`

### 10.2 这一步为什么重要

如果只看结果，会误以为：

1. 第二次优化已经把单聊从 `350` 提到了 `417`
2. 那剩下可能就是小问题了

但继续拆解后发现不是这样。

这一步真正解决的是“解释权”：

**为什么单聊不是 600，也不是 1000，而是刚好停在 410~430 这段。**

### 10.3 这一阶段发现了什么

继续拆解最佳稳定档后发现：

1. 30 对单聊会话在 3 个分区上的分布并不均匀
2. 实际落成了 `8 : 14 : 8`
3. 最热分区占比约 `46.67%`

同时：

1. `session_seq avg` 已经只有 `0.018 ~ 0.019 ms`
2. `total avg` 基本等于 `mysql_persist avg`

最关键的一组数据是：

1. `partition 1 total avg ≈ 4.913 ms`
2. 反推出全局平台约 `436 msg/s`
3. 实际测到最佳稳定档 `417.449 msg/s`

这两者非常接近。

### 10.4 这一阶段的结论

到这里，第二次优化已经把单聊剩余瓶颈解释得很清楚了：

**单聊不是又回到了 `session_seq` 问题，而是被“热分区先打满 + mysql_persist 还存在固定成本”一起卡在了 `410~430 msg/s`。**

## 11. 第九阶段：做验证实验，证明单聊确实吃分区红利

### 11.1 时间点

`2026-04-08 10:04`

对应文档：

1. `第二次优化_第二次排查后session_seq和mysql_persist优化测压结果.md`

### 11.2 这一步做了什么

为了验证“热分区先打满”这个判断，做了两个改动：

1. 单聊会话数从 `30` 对提到 `60` 对
2. Kafka 分区数从 `3` 提到 `6`

### 11.3 为什么要这么优化

逻辑很直接：

1. 会话数更多，hash 分布更容易均匀
2. 分区更多，热点更不容易提前撞满

如果单聊平台因此明显上涨，就说明前面的判断是对的。

### 11.4 实际效果怎么样

这轮结果是：

1. 单聊：`606 msg/s`
2. 群聊：`5475 delivery/s`
3. 群聊折算：`228 group_msg/s`

相对上一轮 `417 / 5462 / 228`：

1. 单聊：`417 -> 606`，提升约 `45.1%`
2. 群聊：几乎不变

对应常规测压最佳档的 `p95`：

1. 单聊：`605.653 msg/s @ p95 49805.1ms`
2. 群聊：`5474.776 delivery/s @ p95 2815ms`

### 11.5 这一阶段的结论

这一步把前面的推断正式坐实了：

1. 单聊之前卡在 `417`，确实和热分区过热有关
2. 群聊因为仍然是一个群会话，一个 key，所以基本不吃分区扩容红利

### 12.2 为什么要做这一阶段

1. 单聊很吃分区红利
2. 但你不能无限靠加分区当工程默认值

所以这一阶段要回答两个更实用的问题：

### 12.3 这一阶段做了什么

分两段跑：

第一段粗扫：

第二段围绕亚秒级 `p95` 做定点搜索：

1. `20 / 40 / 50 / 60 / 80 / 120 / 240 / 480` 分区

### 12.4 实际效果怎么样

粗扫结果说明：

单聊极限吞吐一路上升：

## 17. 当前这轮第二次优化后，还剩下什么问题

第二次优化做完以后，真正剩下的问题已经收得很明确了：

1. `session_seq` 已经不是主瓶颈
2. `mysql_persist` 仍然是最持续、最核心的固定成本之一
3. 单聊后面真正的问题是“热分区先打满 + mysql_persist 固定成本”
4. 群聊如果想继续吃分区红利，压测模型必须从“单群”改成“多群”
5. 如果继续往上提单聊，后面就该进入：
   - `mysql_persist` 固定写库成本继续优化
   - commit 前链路继续轻量化
   - websocket / Redis / 落库进一步解耦

这也是为什么第二次优化结束以后，后续主线会继续转向：

1. `mysql_persist`
2. commit 前链路轻量化
3. 更彻底的异步化与结构优化
