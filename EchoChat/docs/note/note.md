# Write() Notes

在 `Write()` 中先把 `c.SendBack` 和 `c.CriticalBack` 复制到局部变量 `normalCh`、`criticalCh`，是为了让当前写协程可以在循环内部把“已经关闭的通道”动态置为 `nil`，从而让后续 `select` 自动忽略这个分支，而不去修改 `Client` 结构体本身的共享字段；因为关闭的 channel 继续接收会立刻返回 `ok=false`，如果不摘掉，循环会反复命中已关闭分支。

## Q&A

**问题：`Conn` 的 `Read()` 和 `Write()` 是依靠什么数据类型传输的？**  
答案：底层依靠的是 `[]byte` 字节切片。`Conn.ReadMessage()` 从 WebSocket 连接中读到的是一帧原始字节数据，代码里接成 `jsonMessage []byte`，后续再通过 `json.Unmarshal(jsonMessage, &message)` 把它解析成业务结构体 `ChatMessageRequest`。反过来，服务端要写回前端时，会先把响应结构体 `json.Marshal(...)` 成 `[]byte`，再调用 `Conn.WriteMessage(...)` 把这段字节发回浏览器。所以从连接层看，WebSocket 的读和写本质上都是围绕 `[]byte` 展开的，结构体只是业务层对这些字节做了解析和封装。

**问题：`MessageBack` 结构体的作用是什么？**  
答案：`MessageBack` 是服务端写回前端时使用的包装结构，不是前端请求结构，也不是数据库模型。它里面的 `Message []byte` 保存的是已经序列化好的、准备直接写入 WebSocket 的响应内容；`Uuid string` 保存的是这条消息在数据库中的主键，用于在 `Write()` 成功执行 `Conn.WriteMessage(...)` 之后，把对应消息状态更新为已发送。如果 `Uuid` 为空，说明这只是系统提示或失败提示，不对应数据库里的正式聊天记录，此时只写给前端，不做数据库状态更新。因此，`MessageBack` 的本质作用就是把“要发给前端的字节内容”和“这条内容对应的数据库消息 ID”绑在一起，方便写协程在真正发送成功后继续做状态回写。

**问题：`normalizePath` 这个函数的设计思路是什么？**  
答案：`normalizePath` 的作用，是把前端或数据库里出现的资源地址统一规范化成后端更适合长期存储的形式，核心目标是尽量把项目自己的静态资源统一保存为站内相对路径 `/static/...`，避免把当前环境的域名、端口、协议这些易变信息直接写进数据库。因为同一个头像地址，前端有时会传完整 URL，有时又会传相对路径，如果原样入库，后续切换部署环境时会很难维护；但对于第三方外链资源，又不能强行裁成 `/static/...`，否则会把合法外链破坏掉。所以它的处理原则是：空字符串直接返回，已经是 `/static/...` 的原样保留，带 `http/https` 且内部包含 `/static/` 的裁成站内相对路径，带 `http/https` 但不包含 `/static/` 的视为第三方外链并保留原样，其他只要包含 `/static/` 的也裁成相对路径，而对于未知或非法格式则只记日志并原样返回，确保程序不会因为切片越界或误裁路径而崩溃。换句话说，这个函数遵循的核心原则就是：**能安全规范化就规范化，不能确定时宁可保留原值，也不要把路径改坏或直接崩溃。**
