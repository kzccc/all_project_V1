# 第三次优化第一次执行计划

## 1. 先说对原计划表的判断

[第三季优化_详细分析计划表_20260408.md](/workspace/czk/Personal/EchoChat/docs/kafka_plan/第三次优化/第三季优化_详细分析计划表_20260408.md) 这份表，主判断方向是对的，尤其是下面 4 点：

1. 在 `5 broker + 240 partitions` 下，Kafka 前段已经不是唯一关注点，真正要继续打的是消费者后半段。
2. 当前最值得优先打的主瓶颈，仍然是 `mysql_persist`。
3. `5 broker` 不等于 consumer group 有很多 member，member 数必须单独确认。
4. `session_seq` 现在不是第一优先级，不应该抢在 `mysql_persist` 之前继续做大手术。

但这份表如果直接照顺序落地，还有两个执行层面的风险需要先说清楚：

1. 当前仓库和最近压测流程里，chat server 还是“消费和 websocket 同进程”的模型，consumer member 扩容不只是改个参数，还会碰到部署和连接路由问题。
2. 从最近 Kafka group 日志看，压测阶段实际还是 `1 member` 在跑，所以第一次执行不能一上来同时改 `member + mysql_persist + commit`，否则很难判断收益到底来自哪里。

所以第一次执行不应该追求“一次把第三季做完”，而应该追求：

1. 先把第三季 baseline 跑实
2. 先打最粗、最确定、最低风险的收益项
3. 每轮只动一小组变量，确保结论可解释

---

## 2. 第一次执行的总目标

这次“第一次执行”只做三件事：

1. 把当前 `5 broker + 240 partitions` 下的消费者真实基线跑清楚
2. 在不改架构边界的前提下，优先验证 `mysql_persist` 并发出口是不是当前第一主瓶颈
3. 给后面的 member 扩容准备证据，而不是直接跳进多实例部署

也就是说，这次先做：

1. `S0` 基线确认
2. `S2` 的保守版前半段

这次暂时不直接做：

1. `S1` 的大规模 member 扩容
2. `S4` 的“消费完成边界轻量化”架构改造
3. `session_seq` 的新一轮复杂重构

---

## 3. 为什么第一次执行这样排

## 3.1 为什么先不直接做 S1

原计划表里把 `consumer members` 提到 `8` 作为第一结构性动作，这个方向没错，但第一次执行不建议直接上。

原因是：

1. 当前 chat server 同时承担 websocket 在线连接和 Kafka 消费。
2. 如果没有明确的多实例启动方案、连接路由方案、以及压测访问路径设计，直接扩 member 很容易把“消费收益”和“连接分布问题”混在一起。
3. 第一次执行最需要的是低风险、可证伪，不是一次性把部署复杂度拉满。

所以第一次执行里，对 `member` 的处理只做两件事：

1. 先把现状量化出来
2. 先确认后面扩 member 的运维和连接前提

## 3.2 为什么先打 S2

这是当前最值得先落地的一步，因为：

1. 当前配置里 `mysqlPersistWorkerCount = 8`，和 `240 partitions` 明显不匹配。
2. 第二次优化后，`session_seq` 已经明显变轻，`mysql_persist` 仍然长期是 stage 大头。
3. 这一步改动风险比 member 扩容小得多，收益也更直接。
4. 当前持久化是按会话 scope 进 worker，同会话顺序不会因为增加 worker 而直接被破坏。

所以第一次执行里，最核心的实质性优化动作就是：

1. 先把 `mysql_persist` 的并发出口打宽
2. 再看吞吐和 `p95/p99` 有没有同步改善

## 3.3 为什么 commit 不作为第一次主优化项

当前 `consumerCommitBatchSize = 100`、`consumerCommitIntervalMs = 250` 确实偏稳妥，但 commit 不是现在最粗的瓶颈。

第一次执行不把它放到前面，原因是：

1. commit 的收益更偏第二梯队
2. commit 变粗会带来重复消费窗口变大
3. 如果第一次就和 `mysql_persist` 一起改，结论会变脏

所以 commit 这次只做“保留观察”，不做第一轮改动。

## 4. 第一次执行范围

这次建议固定下面这些前提，不动它们：

1. Kafka 形态固定为 `5 broker`
2. chat topic 固定为 `240 partitions`
3. 先只跑 `1` 个 chat server 实例
4. 先不改 websocket/Redis sidecar 边界
5. 先不改 `consumerCommitBatchSize / consumerCommitIntervalMs`
6. 先不改 `session_seq`

这次只允许变化的变量：

1. `mysqlPersistWorkerCount`
2. `mysqlPersistBatchSize`
3. `mysqlPersistFlushIntervalMs`

## 5. 第一次执行的具体步骤

## 5.1 Step A：S0 基线确认

### 目标

先把“当前 5 broker + 240 partitions + 1 chat server 实例”的真实 baseline 跑清楚。

### 要确认的事实

1. 当前 consumer group member 数是不是仍然只有 `1`
2. 当前 `1 member` 实际 claim 了多少 partition
3. 当前高压单聊下，stage 耗时是不是仍然主要落在 `mysql_persist`
4. 当前 broker leader 分布是否基本均匀
5. 当前是否存在明显热点 partition

### 建议压测口径

先固定一套这次第三季统一 baseline 口径：

1. `BASE_CONFIG = configs/config_local_kafka5.toml`
2. `MODE = kafka`
3. `SINGLE_PAIR_COUNT = 60`
4. `GROUP_MEMBER_LIMIT = 25`
5. `SINGLE_MIN_DURATION_SEC = 8`
6. `GROUP_MIN_DURATION_SEC = 8`
7. 常规一键脚本仍然使用 `p95 < 1s` 作为过线标准

### 产出

至少要形成 3 张表：

1. consumer member 数 / partition claim 数表
2. 单聊和群聊各 stage 耗时表
3. leader 分布和热点 partition 表

### 通过标准

完成这一步后，必须能明确回答：

1. 现在是不是仍然 `1 member`
2. 现在 `mysql_persist` 是不是 stage 最大头
3. 现在热点是在 Kafka claim 侧，还是在业务后段

---

## 5.2 Step B：先做 `mysql_persist` 第一档保守实验

### 目标

在不引入部署复杂度的前提下，先验证 `mysql_persist` 出口放宽后，单聊吞吐和尾延迟会不会直接改善。

### 改动范围

只改这 3 个参数：

1. `mysqlPersistWorkerCount: 8 -> 24`
2. `mysqlPersistBatchSize: 64 -> 128`
3. `mysqlPersistFlushIntervalMs: 1 -> 2`

### 为什么第一轮就三项一起动

这三项本质上属于同一组：

1. worker 数决定出口并发宽度
2. batch size 决定单次 insert 聚合度
3. flush interval 决定等待时间和批次形成速度

如果只改 worker、不改 batch 和 flush，很可能会把批次打碎，收益不稳定。

所以第一次执行里，建议把这三项当成一个“保守版套餐”一起验证。

### 压测口径

和 Step A 完全一致，不换口径：

1. 仍然 `240 partitions`
2. 仍然 `5 broker`
3. 仍然 `1 member`
4. 仍然同一套单聊和群聊脚本参数

### 重点观察指标

1. 单聊 `observed_throughput_msg_per_sec`
2. 单聊 `p95 / p99`
3. 群聊 `observed_delivery_per_sec`
4. 群聊 `receipt p95 / p99`
5. `mysql_persist stage avg/p95`
6. lag 是否下降
7. MySQL CPU/TPS/QPS 是否明显异常

### 通过标准

至少满足下面两项中的一项：

1. 在相同口径下，单聊吞吐明显上升
2. 在相近吞吐下，单聊 `p95/p99` 明显下降

同时必须满足：

1. 群聊不能明显回退
2. MySQL 不能出现明显异常抖动

---

## 5.3 Step C：决定要不要进入第二档

这一步不是立即改，而是做判断。

如果 Step B 结果满足下面条件，就进入第二档：

1. 单聊吞吐提升明显
2. `mysql_persist` 仍然是 stage 最大头
3. MySQL 资源还明显有余量

那下一步再做第二档：

1. `mysqlPersistWorkerCount: 24 -> 32`
2. `mysqlPersistBatchSize: 128 -> 256`
3. `mysqlPersistFlushIntervalMs: 2 -> 5`

如果 Step B 已经出现下面任一情况，就先停在第一档：

1. 吞吐没有继续涨
2. `p95/p99` 明显恶化
3. MySQL CPU/TPS 抖动明显
4. 错误率上升

---

## 5.4 Step D：为下一轮 S1 做准备，但这轮不直接落地

虽然这次不直接扩 `consumer members`，但必须把后面扩 member 的前提准备清楚。

这一步要确认：

1. 当前多实例 chat server 的启动方式怎么做
2. 多实例下 websocket 连接如何分布
3. 压测访问入口是否会打到多个实例
4. consumer 和 websocket 不一致时，消息分发路径会不会出问题

这一步的产出不是代码，而是一张“下一轮 member 扩容前置条件清单”。

---

## 6. 第一次执行不做什么

这次明确不做下面 5 件事：

1. 不直接把 consumer members 一口气拉到 `8`
2. 不同时改 `mysql_persist` 和 commit 参数
3. 不在单 partition 内做消息级并行
4. 不先动 `session_seq` 大改造
5. 不先做 websocket/Redis sidecar 化

原因很简单：

1. 第一次执行的目标是“拿到干净结论”
2. 不是“把所有可能有收益的点一起改掉”

---

## 7. 第一次执行后的验收口径

第一次执行完成后，至少要给出下面 4 类结果：

### 7.1 baseline 事实表

包括：

1. 当前 member 数
2. 每个 member claim 数
3. leader 分布
4. 热点 partition 分布

### 7.2 `mysql_persist` 优化对比表

至少对比：

1. 改前配置
2. 保守版配置
3. 各自的单聊吞吐
4. 各自的单聊 `p95/p99`
5. 各自的群聊吞吐
6. 各自的 `mysql_persist stage avg/p95`

### 7.3 结论表

必须明确回答：

1. `mysql_persist` 是不是当前第一主瓶颈
2. 保守版参数有没有收益
3. 是否值得进入第二档
4. 下一轮是不是该正式进入 S1

### 7.4 下一轮动作建议

第一次执行结束后，下一轮只允许从下面两个方向二选一：

1. 如果 `mysql_persist` 仍然收益明显，就继续做第二档参数提升
2. 如果 `mysql_persist` 收益已经变小，就进入 S1 做 member 扩容试点

---

## 8. 这次第一次执行的推荐顺序

如果按真正可落地的工程顺序来排，我建议就是下面 4 步：

1. 先跑 `S0` baseline，确认现在还是不是 `1 member + 240 partitions`
2. 只改一组 `mysql_persist` 保守参数，跑同口径复测
3. 对比吞吐、`p95/p99`、stage 耗时和 DB 指标
4. 再决定下一轮是继续放大 `mysql_persist`，还是转向 `consumer members`

---

## 9. 一句话结论

这张详细分析表的大方向是对的，但第一次执行不应该直接跳进“大规模 member 扩容 + 多项并改”。第一次真正最值钱的动作，应该是：**先在 `5 broker + 240 partitions + 1 member` 的前提下把 baseline 跑实，再优先验证 `mysql_persist` 保守版参数是否能直接把单聊吞吐和尾延迟继续推上去。**
