# 第三次优化第四次执行计划

## 1. 这次要解决什么

前一轮已经确认：

1. 群聊的第一主问题不是跨实例连接分散
2. 群聊的更深层主问题是热点群消息仍然固定压在单个 Kafka partition
3. 但当前群聊链路里，远端 websocket 分发仍然是“按接收者逐个 Redis publish”

所以这次先做一个安全的结构优化：

1. 不改 Kafka key
2. 不改群内顺序语义
3. 不改 session_seq 模型
4. 只改远端 websocket 分发结构

目标是把：

1. 原来 `每个远端接收者 publish 一次`
2. 改成 `同一目标实例只 publish 一次`

---

## 2. 这次准备落地的措施

### 2.1 群聊远端分发按实例批量发送

改动点：

1. `internal/service/chat/kafka_instance_dispatch.go`

具体做法：

1. 原先 envelope 只带一个 `client_id`
2. 现在 envelope 改成可以带多个目标 `targets`
3. 发送端先按 `remote instance` 聚合目标用户
4. 每个远端实例只做一次 Redis publish

这样做的预期收益是：

1. Redis publish 次数从 `O(接收者数)` 降到 `O(目标实例数)`
2. 群聊人数越多，这个收益越明显

### 2.2 保持 sent 状态语义不变

这次不能把已有语义打坏，所以要保留：

1. 同一条消息仍然只带一个 `message UUID`
2. 仍然只触发一次 `status=sent`

也就是说：

1. 批量 envelope 里虽然会带多个目标用户
2. 但只有一个目标会携带 `UUID`

### 2.3 保持兼容

为了避免旧格式消息解码失败，订阅端需要兼容两种 payload：

1. 旧版单目标 payload
2. 新版批量 targets payload

## 4. 这次重点观察什么

除了最终吞吐，还要重点看两个中间指标：

1. `remote_publish_success` 是否明显下降
2. `websocket_dispatch` 平均耗时是否下降

如果这两个中间指标下降了，但群聊总体吞吐没有明显上涨，就说明：

1. 这次结构优化是有效的
2. 但它还不是群聊主瓶颈
3. 下一步要继续往热点 partition / 更下游阶段排查
