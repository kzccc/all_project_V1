# 第三季优化详细分析计划表

## 1. 这份计划解决什么问题

这份计划专门回答当前第三阶段 Kafka 优化里最关键的一个问题：

**在 `5 broker`、`240 partitions` 的前提下，EchoChat 的消费者链路应该如何继续优化，才能让真实单聊吞吐继续上升，同时尽量控制 `p95/p99` 尾延迟。**

这份计划不是泛泛地讨论 Kafka 调优，而是基于当前代码实现、已经完成的第二次优化结果、以及最新环境事实来制定下一步动作。

---

## 2. 当前环境和结论先对齐

### 2.1 当前 Kafka 形态

当前环境不是“5 台物理机组成的 Kafka 集群”，而是：

1. `1` 台机器
2. 运行 `5` 个 Kafka broker 进程
3. chat topic 规划为 `240` 分区

这意味着：

1. Kafka 的 leader/partition 分布已经可以比单 broker 更均匀
2. 但底层 CPU、磁盘、页缓存、网卡仍然是同一台机器
3. 因此它更像“多 broker 软件并行实验环境”，不是完整的硬件横向扩容环境

### 2.2 当前消费者实现的核心事实

当前代码里的消费模型有几个决定性特征：

1. 每个 `chat server` 进程只创建 `1` 个 `sarama.ConsumerGroup`
2. Kafka consumer group 会按分区并发调用 `ConsumeClaim`
3. 同一个 partition 内仍然按消息顺序串行处理
4. 生产端优先使用 `session_id` 作为 Kafka key
5. 因此同一会话的消息会尽量被打到同一 partition
6. 消费链路里最重的阶段是 `mysql_persist`
7. `mysql_persist` 当前并不是“按 partition 并发”，而是进入一个全局 worker 池
8. 当前配置里的 `mysqlPersistWorkerCount = 8`

### 2.3 这套事实带来的直接推论

1. `5 broker` 并不会自动提升 consumer group member 数量
2. `240 partitions` 也不会自动变成“240 路真正有效的业务处理并发”
3. 如果 chat server 实例数偏少，单实例会分到太多 partition
4. 即使 partition 很多，最终仍可能在 `mysql_persist` worker 池重新汇聚成很少的落库并发
5. 因为同会话顺序依赖 `session_id -> partition` 的稳定映射，所以不能简单在单 partition 内乱并行

---

## 3. 第三季优化总目标

第三季优化建议拆成三个层级目标：

### 3.1 P0 目标

先把消费者侧真实瓶颈定量化，不再停留在“Kafka 可能不够快”这种模糊判断。

### 3.2 P1 目标

让消费者链路先跟上 `240 partitions` 的输入能力，优先解决：

1. 单实例 claim 过多
2. `mysql_persist` 并发出口过窄
3. offset commit 过细
4. 高压下 `p95` 过快恶化

### 3.3 P2 目标

在不破坏顺序语义的前提下，把消费完成边界从“重链路全同步”改造成“持久化优先，副作用后置”，为后续更高吞吐预留架构空间。

---

## 4. 当前主要瓶颈判断

结合现有代码和最近几轮测压结果，第三季优化默认采用如下判断：

### 4.1 当前第一瓶颈：`mysql_persist`

这是当前最明确、最值得优先打的瓶颈。

原因：

1. 近几轮高压指标里，消费者 stage 时间大头集中在 `mysql_persist`
2. 当前落库 worker 数量只有 `8`
3. `240 partitions` 下，这个出口过窄
4. 分区数越高，越容易把 Kafka 前面的并行收益重新压扁成少量 DB worker 队列

### 4.2 当前第二瓶颈：consumer member 数量可能不足

需要强调，这里的“消费者数量”指的是：

**consumer group member 数量，也就是实际运行的 chat server 实例数。**

不是：

1. Kafka broker 数
2. Kafka partition 数
3. Go runtime 线程数

如果 member 数过少，就会出现：

1. 单实例分到过多 partition
2. 单实例内部 goroutine 数量过多
3. offset 管理、rebalance、内存和排队抖动一起放大

### 4.3 当前第三瓶颈：提交边界过重

当前消费流程里，消息通常要经过：

1. 反序列化
2. `session_seq`
3. `mysql_persist`
4. websocket 分发
5. Redis 读写

之后才算完成。这会让 consumer lag 和尾延迟更容易绑定在一整条业务副作用链路上。

### 4.4 当前第四瓶颈：未来的 `session_seq` 热点

当前 `session_seq` 还不是最大瓶颈，但在 `240 partitions + 更多 consumer members` 的阶段，它很可能会从“轻量热点”变成“全局热点”。

因此第三季要把它纳入中期改造名单，但不建议现在抢在 `mysql_persist` 之前动大手术。

---

## 5. 第三季优化路线图

第三季建议分为五个阶段推进。

| 阶段 | 优先级 | 主题 | 目标 |
| --- | --- | --- | --- |
| S0 | P0 | 基线确认 | 先确认 member 数、lag、claim 分布和 stage 耗时 |
| S1 | P1 | 扩 consumer 出口 | 控制每个 member 分到的 partition 数 |
| S2 | P1 | 放大 `mysql_persist` 并发和批量 | 把真正的消费主瓶颈打宽 |
| S3 | P1 | 放粗 offset commit | 减少高分区数下的提交开销 |
| S4 | P2 | 轻量化消费完成边界 | 为更高吞吐和更稳尾延迟做架构预留 |

---

## 6. 分阶段详细计划

## 6.1 S0：基线确认阶段

### 目标

先把当前环境里的几个关键数字跑清楚。

### 必须确认的数字

1. 当前 chat server 一共有多少个 consumer group member
2. `240 partitions` 下每个 member 实际分到了多少 partition
3. 每个 member 的 goroutine、CPU、内存、FD 峰值
4. `mysql_persist`、`session_seq`、`redis_*`、`websocket_dispatch` 的 stage 占比
5. 每个 broker 上的 leader 分布是否均衡
6. topic 是否存在明显热点 partition
7. 当前 consumer lag 在稳态和峰值时的变化

### 为什么先做这个

第三季不应该直接以“5 broker 已经很多”作为判断依据。

真正决定消费链路是否跟得上的，是：

1. `member` 数
2. `partition/member` 比例
3. `mysql_persist` 是否已经成为统一出口

### 执行动作

1. 固定一轮 `240 partitions` 单聊压测口径
2. 记录 consumer group 成员数量
3. 统计每个成员的 partition claim 数
4. 统计每个成员的 stage 指标
5. 汇总成一份 baseline 表

### 通过标准

至少要能回答下面三个问题：

1. 当前每个 consumer member 扛了多少 partition
2. 当前到底是 Kafka claim 太重，还是 `mysql_persist` 太窄
3. 当前 broker 增加后，热点是不是已经从 Kafka 前段转移到业务后段

---

## 6.2 S1：扩 consumer 出口阶段

### 核心判断

`5 broker` 不等于 `5 consumer members`。  
所以第三季第一个结构性动作，不是继续加 broker，而是把 consumer group member 数量提升到足以吃住 `240 partitions`。

### 建议目标区间

推荐把 `partition/member` 控制在：

1. 保守目标：`20 ~ 30`
2. 激进目标：`15 ~ 20`

换算到 `240 partitions`：

1. `8 members` 时：每个约 `30` 个 partition
2. `10 members` 时：每个约 `24` 个 partition
3. `12 members` 时：每个约 `20` 个 partition

### 建议动作

1. 如果当前 member 数少于 `8`，先扩到 `8`
2. 如果扩到 `8` 后 lag 仍明显，继续扩到 `10`
3. 如果 `10` 之后收益仍明显，再考虑 `12`

### 为什么不是无限扩

因为 member 不是越多越好。

member 过多会带来：

1. rebalance 更频繁
2. 每个实例有效工作变碎
3. WS 本地连接模型更复杂
4. 每实例批量收益下降

### 风险和注意点

如果消息消费后必须直接命中本机内存中的在线连接，那么增加纯 consumer member 会碰到“消费节点和在线连接节点不一致”的问题。

这种情况下，第三季要提前做架构判断：

1. 是维持“消费和网关同进程”
2. 还是拆成“消费 worker + websocket gateway”

### 本阶段验收指标

1. `partition/member` 明显下降
2. 单实例 lag 峰值下降
3. 单实例 goroutine/内存抖动下降
4. 端到端吞吐有提升或 `p95` 更稳

---

## 6.3 S2：放大 `mysql_persist` 并发和批量

### 核心判断

这是第三季最值得优先落地的一项。

当前 `mysql_persist` 已经是消费侧最重阶段，而配置仍然偏保守：

1. `mysqlPersistBatchSize = 64`
2. `mysqlPersistFlushIntervalMs = 1`
3. `mysqlPersistWorkerCount = 8`

### 目标

把 Kafka 前面的高分区并行能力，真正转化为更宽的持久化并发出口。

### 为什么这是第一优先级

因为当前模型里：

1. message 会按会话 scope 哈希到持久化 worker
2. 同会话仍然落在同一个 worker
3. 顺序语义不会因为增 worker 而直接被破坏

所以这是一个收益大、风险可控、改动范围相对小的优化点。

### 参数建议

#### 第一档：保守版

1. `mysqlPersistWorkerCount: 8 -> 24`
2. `mysqlPersistBatchSize: 64 -> 128`
3. `mysqlPersistFlushIntervalMs: 1 -> 2`

#### 第二档：进取版

1. `mysqlPersistWorkerCount: 24 -> 32`
2. `mysqlPersistBatchSize: 128 -> 256`
3. `mysqlPersistFlushIntervalMs: 2 -> 5`

#### 第三档：仅在 DB 明显扛得住时考虑

1. `mysqlPersistWorkerCount -> 48`

### 为什么不要一口气冲太高

因为 DB 不是无限可扩的。

worker 太多会引发：

1. 连接争用
2. SQL 批次变碎
3. MySQL CPU 上升
4. InnoDB 锁和日志压力变大

### 本阶段关注指标

1. `mysql_persist` stage avg/p95
2. DB CPU
3. DB TPS/QPS
4. insert 成功率
5. lag 是否下降
6. 端到端单聊吞吐是否上升

### 通过标准

至少满足下面两项中的一项：

1. 在相同口径下吞吐明显上升
2. 在相同吞吐附近时 `p95/p99` 明显下降

---

## 6.4 S3：放粗 offset commit 阶段

### 核心判断

在 `240 partitions` 规模下，offset commit 过细会带来额外管理开销。

当前配置：

1. `consumerCommitBatchSize = 100`
2. `consumerCommitIntervalMs = 250`

这更偏向“较稳妥的低重放窗口”，而不是“高分区吞吐优先”。

### 建议动作

#### 第一档

1. `consumerCommitBatchSize: 100 -> 300`
2. `consumerCommitIntervalMs: 250 -> 500`

#### 第二档

1. `consumerCommitBatchSize: 300 -> 500`
2. `consumerCommitIntervalMs: 500 -> 1000`

### 预期收益

1. 减少 commit 频率
2. 降低协议请求开销
3. 降低高分区数下 claim 内部定时器和 flush 频率

### 代价

1. 异常退出时重复消费窗口变大
2. 对消费幂等要求更高

### 本阶段结论原则

这项优化属于“第二梯队收益”，不应抢在 `mysql_persist` 之前做主优化项。

---

## 6.5 S4：轻量化消费完成边界

### 当前问题

当前 `MarkMessage` 的完成边界仍然偏重。  
消费者不仅要完成 durable 工作，还要同步完成 websocket 和 Redis 相关副作用。

### 第三季建议的方向

把消费完成边界逐步往“持久化成功”收敛。

### 推荐目标语义

Kafka consumer 只对以下动作负责：

1. 反序列化
2. `session_seq`
3. durable persist

然后：

1. offset 可提交
2. websocket 分发转异步 sidecar
3. Redis 缓存更新转异步 sidecar

### 预期收益

1. consumer lag 更容易压住
2. 吞吐更高
3. `p95` 不容易被 Redis/WS 波动拖崩

### 风险

1. 语义变复杂
2. “已持久化”与“已推送到在线端”不再是同一个完成点
3. 需要补偿机制、失败重试和更强监控

### 第三季定位

这一项建议作为第三季后半段的架构改造候选，不建议作为最先落地的动作。

---

## 7. 第三季不建议优先做的事情

## 7.1 不建议直接在单 partition 内做消息级并行

原因：

1. 当前会话顺序依赖 `session_id -> partition`
2. 单 partition 内并行会直接破坏顺序语义
3. offset 提交复杂度会明显上升

除非后续愿意引入：

1. partition reader 串行
2. session shard 二级分发
3. 连续位点提交机制

否则不建议在第三季优先触碰。

## 7.2 不建议把 broker 数增长当成消费者优化替代品

原因：

1. broker 数增加主要解决的是 Kafka 集群资源池和 leader 分布问题
2. 它不会自动放大应用消费并发出口
3. 当前最重阶段已经在业务后段，不在 Kafka pull 本身

## 7.3 不建议立即把 `session_seq` 改造成大规模复杂方案

原因：

1. 当前它还不是第一瓶颈
2. 先打 `mysql_persist` 回报更高
3. 第三季应先处理最粗的瓶颈，再打未来热点

## 9. 验证矩阵

第三季建议每次只改一小组参数，避免结论污染。

| 轮次 | 修改项 | 目标 |
| --- | --- | --- |
| R1 | 只增 consumer members | 判断 claim 过重是否成立 |
| R2 | 只调 `mysqlPersistWorkerCount` | 判断 DB worker 出口是否是主瓶颈 |
| R3 | 再调 batch + flush interval | 判断批量是否继续带来收益 |
| R4 | 再调 commit 参数 | 判断 offset 管理开销是否明显 |
| R5 | 对比保守版与进取版 | 选出工程默认值 |

每一轮都要固定：

1. 压测口径
2. topic 分区数
3. broker 数
4. member 数以外的其他变量

## 11. 一句话结论

在当前 `1 台机器 + 5 broker 进程 + 240 partitions` 的环境里，第三季优化的正确主线不是继续加 broker，而是：

**先增加 consumer group member 数量，再把 `mysql_persist` 的 worker 和批量能力放大，最后再放粗 commit，并逐步把 websocket/Redis 从消费完成边界里剥出去。**

如果第三季只做一件最值钱的事，那就是：

**优先扩 `mysqlPersistWorkerCount`，并把 `240 partitions` 对应的 consumer members 提到合理区间。**
