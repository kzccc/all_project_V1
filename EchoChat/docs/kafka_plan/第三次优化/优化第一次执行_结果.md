# 第三次优化第一次执行结果

## 1. 这次第一档优化，核心到底做了什么

这次第一档优化，只做了一件事：

**只调 `mysql_persist`，不动 `session_seq`、不动 commit、也不动 consumer member。**

这次改的是下面 3 个参数：

1. `mysqlPersistWorkerCount: 8 -> 24`
2. `mysqlPersistBatchSize: 64 -> 128`
3. `mysqlPersistFlushIntervalMs: 1 -> 2`

对应配置文件是：

[config_local_kafka5_mysqlpersist_tune1.toml](/workspace/czk/Personal/EchoChat/configs/config_local_kafka5_mysqlpersist_tune1.toml)

---

## 2. 这 3 个参数分别是什么意思

### 2.1 `mysqlPersistWorkerCount`

这个参数控制：

1. 有多少个持久化 worker 并发处理消息落库

从 `8` 提到 `24`，本质上就是把 MySQL 落库出口加宽。

之前的问题是：

1. Kafka 前面已经有 `240 partitions`
2. 但后面真正写 MySQL 的 worker 只有 `8`
3. 前面并行拉进来的消息，最后会在 `mysql_persist` 这里重新排队

所以这次先把它从 `8` 提到 `24`，看出口够不够宽。

### 2.2 `mysqlPersistBatchSize`

这个参数控制：

1. 每次批量写 MySQL，最多攒多少条再一起落库

从 `64` 提到 `128`，意思是：

1. 单次批量 insert 更大
2. 每条消息平均分摊下来的数据库写入成本更低
3. 更容易把 MySQL 吞吐打上去

### 2.3 `mysqlPersistFlushIntervalMs`

这个参数控制：

1. worker 最多等多久，就把当前 batch 刷出去

从 `1ms` 调到 `2ms`，意思是：

1. 稍微多给一点时间攒批
2. 避免 batch 太碎
3. 让 `128` 这个 batch 更容易真正攒起来

这一步不是单纯为了“等更久”，而是为了让批量写更成型。

---

## 3. 为什么第一档只动这 3 个参数

因为这轮目标不是一次把所有优化都做完，而是先验证一件事：

**当前单聊和群聊吞吐上不去，是不是主要卡在 `mysql_persist`。**

如果一开始同时改：

1. `mysql_persist`
2. `consumer member`
3. `offset commit`
4. `session_seq`

那最后吞吐涨了，也很难知道到底是哪一步带来的。

所以第一档故意只动 `mysql_persist` 这一组参数，这样结论最干净。

## 7. 第三档诊断说明了什么

第三档固定档诊断目录：

[diagnostic_stages_kafka_20260409_144354_thirdopt_exec1_tune3_diag](/workspace/czk/Personal/EchoChat/docs/k6_message_test/records/diagnostic_stages_kafka_20260409_144354_thirdopt_exec1_tune3_diag)

### 7.1 单聊诊断

单聊 `target=2040` 诊断结果：

1. observed `1673.835 msg/s`
2. `p95 = 109.05 ms`
3. `mysql_persist ≈ 11.896 ms`
4. `total ≈ 12.145 ms`

### 7.2 群聊诊断

群聊 `target=5040` 诊断结果：

1. observed `3579.664 delivery/s`
2. `p95 = 445 ms`
3. `mysql_persist ≈ 4.251 ms`
4. `total ≈ 4.519 ms`

### 7.3 这组诊断该怎么理解

这里最关键的一点是：

**第三档并不是 `mysql_persist` 明显变慢了，而是“参数继续加大后，整体最优工作点反而变低了”。**

也就是说：

1. 单聊诊断里的 `mysql_persist` 仍在 `11ms ~ 12ms` 这个量级
2. 群聊诊断里的 `mysql_persist` 仍在 `4ms` 左右
3. 但 throughput 脚本能通过的最佳档位却下降了

这通常说明：

1. batch 和 flush 再继续加大后，收益已经不再稳定
2. 更大的等待窗口和更大的批量，不再换来更高的稳定吞吐
3. 对群聊这种单热点分区场景，还可能放大单热点的排队波动

所以第三档的结论不是“再调一下也许会好”，而是：

**这条 `mysql_persist` 继续加大的路，已经到收益拐点了。**

所以第三档结果仍然是单实例、单 member 的结果。

### 8.2 群聊仍然是单分区热点

第三档诊断里：

2. 群聊仍然几乎全部压在分区 `188`

所以第三档群聊回退，不是 leader 分布问题，而是：

1. 单热点群会话
2. 单 member
3. `mysql_persist` 继续加大后不再适合这个场景

共同导致的。

## 9. 第三档的结论

第三档可以直接总结成一句话：

**`mysql_persist` 继续从第二档往上推，已经开始回退，第三档不是推荐配置，当前这条线应该停在第二档。**

如果拆开说，就是：

1. 做法：继续只调 `mysql_persist` 的 `worker / batch / flush`
2. 配置：`worker 32 -> 40`，`batch 256 -> 384`，`flush 5 -> 8`
3. 单聊结果：`1530.825 msg/s`，`p95 = 398.05 ms`
4. 群聊结果：`3595.012 delivery/s`，`p95 = 469 ms`
5. 群聊折算：`149.8 group_msg/s`
6. 结论：第三档已经回退，不建议继续沿这条线往上推

## 10. 当前第一次执行后的推荐停点

到现在为止，这轮“第一次执行”里，`mysql_persist` 这一条线最合适的停点就是第二档。

也就是：

1. `mysqlPersistWorkerCount = 32`
2. `mysqlPersistBatchSize = 256`
3. `mysqlPersistFlushIntervalMs = 5`

原因是：

1. 第一档吃到了最大的吞吐收益
2. 第二档基本守住了吞吐，同时明显改善了尾延迟
3. 第三档已经开始回退

所以接下来的正确方向，不应该再继续堆 `mysql_persist`，而应该转去做：

1. `consumer member` 扩容
2. 或群聊热点分摊
