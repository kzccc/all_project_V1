# 第三次优化第五次执行第一次结果

## 1. 这一步先回答什么问题

在回退第四次执行的群聊批量分发代码之后，这一步先不急着改新代码，而是先把一个关键问题定死：

1. 下一步到底应该继续压 `mysql_persist`
2. 还是应该优先做热点群分片 / 子队列 / 顺序域拆分

这一步的结论很重要，因为它决定后面优化方向。

---

## 2. 当前代码里的关键事实

先把现在群聊链路里几个最关键的事实对齐。

### 2.1 群聊消息当前按 `session_id` 选 Kafka key

当前发送端逻辑是：

1. `client.go` 里发送 Kafka 消息时调用 `ResolveMessageKey(message.SessionId)`
2. `config.go` 里 `ResolveMessageKey` 默认优先返回 `session_id`

这意味着：

1. 同一个群会话的消息，会稳定进入同一个 Kafka partition

### 2.2 Kafka partition 内部是串行消费

当前消费端是：

1. Kafka consumer group 对不同 partition 可以并发
2. 但同一个 partition 内，`ConsumeClaim` 仍然是逐条串行处理

所以：

1. 单热点群如果始终落在一个 partition
2. 它本质上就是“一条串行处理链”

### 2.3 当前消费处理不是轻链路

当前热点群消息进入消费后，要串行经过：

1. `session_seq`
2. `mysql_persist`
3. `group_member_query`
4. `websocket_dispatch`
5. 之后才会继续下一条

也就是说：

1. 单热点群现在不是被 Kafka broker 数限制
2. 而是被“单 partition 串行处理速度”限制

---

## 3. 用现有数据反推当前主上限

上一轮 `10 members` 群聊最好结果是：

1. `4099.707 delivery/s`
2. 折算约 `170.8 group_msg/s`

当时热点 owner 的 consumer 平均耗时大约是：

1. `group_member_query ≈ 0.122 ms`
2. `mysql_persist ≈ 3.864 ms`
3. `websocket_dispatch ≈ 0.682 ms`
4. `total ≈ 4.736 ms`

这组数据非常关键。

因为它说明：

1. 单条热点群消息在 consumer 里平均要花将近 `4.7 ms`
2. 单 partition 串行处理时，理论上限大约就在 `1 / 0.004736 ≈ 211 msg/s`

而现在实际打出来的是：

1. `170.8 group_msg/s`

这个量级已经非常接近“单 partition 串行处理”的上限了。

所以可以先明确一个判断：

1. 当前群聊上限首先是热点群单 partition 串行造成的
2. 不是单纯因为 worker 数还不够大

---

## 4. 为什么不能只继续调 mysql_persist

这里要分成两层看。

### 4.1 `mysql_persist` 仍然很重

这个结论没变。

当前热点群消息里：

1. `mysql_persist ≈ 3.864 ms`
2. 它占了 `total ≈ 4.736 ms` 的大头

所以它当然还值得优化。

### 4.2 但只调它，不足以突破热点群当前主上限

原因是：

1. 热点群现在先被固定到单 partition
2. 单 partition 内是串行
3. 这意味着你就算把 `mysql_persist` worker 数继续调大
4. 也不会把“单热点群只能一条一条处理”的事实改变掉

更直接地说：

1. `mysql_persist` 是当前串行链路里最重的一段
2. 但群聊真正的结构性上限来自“这整条链路是单分区串行”

所以：

1. 继续调 `mysql_persist` 有价值
2. 但它更像“继续压低单条消息成本”
3. 不是“突破热点群主容量天花板”的第一杠杆

---

## 5. 还有一个额外发现

这一步顺便发现了一个后续必须处理的点：

当前 `mysql_persist` worker 路由逻辑里：

1. `workerIndexForMessage` 是按 `conversation scope` 做 hash

而群聊 `conversation scope` 就是：

1. `group:<groupID>`

这意味着：

1. 同一个热点群消息，即使后面不再是单 Kafka partition
2. 在当前 `mysql_persist` 实现里，仍然会继续被路由到同一个 persist worker

所以这件事的意义是：

1. 它不是当前第一主瓶颈
2. 但如果后面我们做了热点群分片
3. 当前 `mysql_persist` 也要跟着改成“不要再按整个群固定单 worker”

否则会把新并发又压回去。

---

## 6. 这一步的结论

到这里，这一步可以给出一个清晰结论：

1. 下一步主线应该优先做“热点群分片 / 子队列 / 顺序域拆分”方案探索
2. `mysql_persist` 继续优化要做，但放在第二优先级

原因不是 `mysql_persist` 不重要，而是：

1. 只调 `mysql_persist`，不够突破热点群单 partition 串行上限
2. 热点群分片才是群聊容量上限的第一杠杆
3. 但在做热点群分片时，要同步改 `mysql_persist` 的 worker 路由逻辑

一句话总结就是：

1. 先解决“为什么只有一条串行处理链”
2. 再解决“这条链里最重的那一段”

---

## 7. 下一步怎么做

下一步正式进入第五次执行计划：

1. 先设计热点群的分片 / 子队列 / 顺序域方案
2. 明确顺序语义要保留到什么程度
3. 再决定配套怎么改 `mysql_persist`

当前已经先定下这轮的实现选择：

1. 先做保守版

保守版的边界是：

1. 不直接改 Kafka key
2. 先利用现有前端按 `session_seq` 排序的能力
3. 把群消息从“整条重链路串行”改成“顺序入口 + 下游并行”

也就是说：

1. 第五次执行的主线重点，不再是 fanout
2. 而是热点群单分区模型本身
