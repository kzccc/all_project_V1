# 第三次优化第二次执行第一次结果

## 1. 这次先做了什么

第二次执行没有直接开始多 member 吞吐压测，而是先处理一个前置问题：

**补齐多实例下的 websocket 跨实例分发能力。**

原因很直接：

1. 当前 Kafka 消费后的消息分发，只查本进程内存里的 `Clients`
2. 如果直接起多个 chat server 实例，消费实例和连接实例很可能不是同一个
3. 那么消息即使被 Kafka 正常消费，也不一定能真正写回正确的 websocket 连接

所以这次先做的是：

1. 补跨实例连接路由
2. 补跨实例消息转发
3. 先做双实例功能验证

这一步通过后，后面的多 member 压测结果才可信。

---

## 2. 这次实际改了什么

这次核心改动是给 Kafka 模式补了一条“实例间转发链路”。

### 2.1 在线连接路由登记

每个 Kafka chat server 实例在用户 websocket 登录时，把：

1. `用户 UUID -> 当前实例 ID`

登记到 Redis。

当前实例 ID 采用：

1. `chat-{port}`

例如：

1. `chat-18082`
2. `chat-18083`

这样消费实例就能查到“这个用户现在连在哪个实例上”。

### 2.2 Kafka 消费后的跨实例转发

Kafka 消费实例在处理完消息后，不再只查本机 `Clients`，而是：

1. 先看目标用户是不是在本机
2. 如果在本机，直接本机 `enqueueBack`
3. 如果不在本机，就查 Redis 路由
4. 如果路由显示在其他实例，就通过 Redis Pub/Sub 转发到对应实例
5. 对应实例收到后，再写回本机 websocket

### 2.3 路由清理

用户断开或登出时，会把：

1. 该用户当前实例的路由

从 Redis 里清掉，避免留下脏路由。

而且这次删除是“按值删除”，不会把已经被其他实例接管的新路由误删掉。

## 4. 这次是怎么验证的

这次没有直接跑满压，而是先做双实例功能验证。

### 4.1 验证配置

我基于第一次执行推荐停点，也就是 `mysql_persist` 第二档配置，起了两台独立 chat server：

1. 实例 1：`port=18082`
2. 实例 2：`port=18083`

两台实例共用：

1. 同一个 Kafka topic
2. 同一个实验 consumer group：`chat_cluster5_exec2_verify`

这样做的目的，是不影响当前 `8081` 那台历史进程，同时让这次验证只看“第二次执行”的多实例链路。

### 4.2 单聊验证

单聊验证方式是：

1. 发送者 websocket 连接到 `18082`
2. 接收者 websocket 连接到 `18083`
3. 发送一条单聊消息
4. 确认 `18083` 上的接收者正常收到

验证结果：

1. 单聊跨实例送达成功

### 4.3 群聊验证

群聊验证方式是：

1. 发送者连接到 `18082`
2. 多个群成员分散连接到 `18082` 和 `18083`
3. 发送一条群聊消息
4. 确认不同实例上的接收者都能收到

验证结果：

1. 群聊跨实例送达成功

### 4.4 Kafka member 验证

Kafka broker 日志里可以确认，这次实验组最终稳定为：

1. `The group has 2 members`

也就是说，这次不是“两个进程各跑各的”，而是真正形成了：

1. `2 member consumer group`

## 5. 这次验证的关键结论

这次结果最重要的结论有 3 个。

### 5.1 结论一：多实例前置正确性已经补上了

现在 Kafka 模式下，消息消费实例和 websocket 连接实例不一致时，消息仍然能送达。

这说明：

1. 第二次执行最关键的前置问题已经打通

### 5.2 结论二：现在终于可以开始做真正的多 member 压测了

在这次改造前，直接起多 member 跑压测是不可信的。  
在这次改造后，至少“消息能不能送到用户”这个正确性前提已经成立。

所以现在可以继续做：

1. `2 members` 压测
2. `4 members` 压测

并把结果和第一次执行的 `1 member` 结果做同口径对比。

### 5.3 结论三：这次还不是最终吞吐结果

这次第一次结果的重点是：

1. 先证明多实例链路能跑通

所以这份结果报告还没有给出：

1. 新的单聊最大吞吐
2. 新的群聊最大吞吐

这两个数字会放到第二次执行后续结果里。
