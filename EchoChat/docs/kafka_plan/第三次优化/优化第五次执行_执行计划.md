# 第三次优化第五次执行计划

## 1. 这次主线目标

第五次执行的主线目标很明确：

1. 不再继续做 fanout 侧的小优化
2. 直接处理群聊当前最大的结构性瓶颈

当前要解决的问题是：

1. 单热点群消息固定落在单个 Kafka partition
2. 单个 partition 内只能串行消费
3. 所以群聊真实容量很快碰到上限

---

## 2. 这次为什么先做热点群结构，而不是只调 mysql_persist

原因已经在第一次结果里确认了：

1. `mysql_persist` 仍然很重
2. 但热点群当前主上限首先来自单 partition 串行

所以第五次执行的优先级定成：

1. P0：热点群分片 / 子队列 / 顺序域拆分方案
2. P1：配套调整 `mysql_persist`

这里的意思不是不做 `mysql_persist`，而是：

1. 它作为第二步配套优化
2. 不再作为单独主线来押注

---

## 3. 第五次执行拆成三步

### 3.1 第一步：先定顺序语义边界

这一阶段先回答清楚：

1. 我们到底要求“群内绝对严格全序”
2. 还是允许“最终按 `session_seq` 展示一致，但实时到达可能短暂乱序”

这是整个方案的前置条件。

因为：

1. 如果必须严格全序
2. 那同一群消息就很难真正并行

而如果允许：

1. 服务端先分片处理
2. 前端或分发层再按 `session_seq` 重排

那才有空间做真正的热点群扩容。

### 3.2 第二步：设计热点群分片实验方案

这一步准备先出实验方案，不直接大改主线。

优先考虑两类思路：

#### 方案 A：Kafka key 分片

思路：

1. 同一群不再只用一个固定 key
2. 而是变成 `group_id + shard`
3. 把一个热点群拆到多个 partition

优点：

1. 可以直接把单热点群吃掉多个 partition 的并发

难点：

1. 群内顺序语义会被打散
2. 需要补一个新的 group 全局顺序策略

#### 方案 B：单 partition 轻消费 + 下游子队列并行

思路：

1. Kafka ingress 仍然保持单 partition
2. 但消费端先只做轻量步骤
3. 再把后续重链路拆进多个 group 子队列

优点：

1. 不直接改 Kafka key
2. 风险比方案 A 小

难点：

1. 如果最重步骤还放在 ingress 串行链路里，就吃不到收益
2. 所以必须重新定义“消费完成边界”

### 3.3 第三步：配套改 mysql_persist

不管最终选 A 还是 B，都要同步处理一件事：

1. 当前 `mysql_persist` 还是按整个 `conversation scope` 做 worker hash

所以后续要改成：

1. 不再把一个热点群固定压到单个 persist worker
2. 至少要支持“热点群子分片 -> 多 persist worker”

---

## 4. 这次我建议优先落哪条实验路线

第五次执行建议先落：

1. 方案 B 的前半段

也就是：

1. 先设计“单 partition 轻消费 + 下游 group 子队列”的实验路径

原因：

1. 比直接改 Kafka key 风险小
2. 不会立刻破坏当前生产侧 key 语义
3. 更适合先做小步验证
4. 当前前端消息列表本身已经按 `session_seq` 排序，所以“最终一致、处理中间可短暂乱序”在现有项目里有现实落地空间

只有当这条路确认收益不够时，再进入：

1. 方案 A：真正的群级 Kafka key 分片

---

## 5. 第五次执行的具体输出物

这次执行计划里，先要求产出两类东西。

### 5.1 设计输出

1. 一份热点群子队列方案说明
2. 明确哪些步骤放在 ingress 串行链路
3. 明确哪些步骤允许进入 group shard workers
4. 明确顺序语义如何保证

### 5.2 实验输出

1. 第一版实验实现
2. 一轮正式压测结果
3. 和当前基线对比：
   单聊基线：`2356.15 msg/s`
   群聊基线：`4099.707 delivery/s`
   群聊折算：`170.8 group_msg/s`

### 5.3 当前选定的实验版本

第五次执行先选：

1. 保守版

保守版具体含义是：

1. Kafka key 暂时不改
2. 群消息仍先进入单 partition 顺序入口
3. 入口只负责轻量有序步骤
4. 后续重链路进入 group shard workers 并行

激进版先只保留在备选方案里，暂不作为本轮第一实现。

---

## 6. 第五次执行当前的起步判断

先把起步判断写死，避免后面方向漂移：

1. fanout 不是主线
2. 单调 `mysql_persist` 不是主线
3. 热点群单 partition 串行，才是当前群聊上限的第一主线

所以第五次执行，从这一点开始推进。
