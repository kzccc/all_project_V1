# 第三次优化第二次执行计划

## 1. 先说为什么第二次执行不能直接上多 member 压测

第一次执行已经把 `mysql_persist` 这条线打清楚了，结论也很明确：

1. 第一档能吃到最大吞吐收益
2. 第二档更像“稳尾延迟”
3. 第三档已经开始回退

所以第二次执行的主线必须切到：

1. `consumer member` 扩容

但当前代码里有一个必须先解决的硬前提：

**Kafka 消费后的 websocket 分发，只查当前进程内存里的 `Clients`，没有跨实例转发能力。**

这意味着，如果直接把 chat server 从 `1` 个实例扩到多个实例，会出现下面这个问题：

1. 用户 A、B 的 websocket 连接在实例 `S1`
2. 他们的消息被 Kafka 分区分配到实例 `S2` 消费
3. `S2` 本地没有这两个用户的连接
4. 消息虽然被消费了，但不会真正写回到 `S1` 的 websocket

所以第二次执行不能直接跳到“多实例压测”，否则压出来的结果是不可信的。

---

## 2. 第二次执行的总目标

这次“第二次执行”分两层目标。

### 2.1 第一层目标

先把**多实例消费前的正确性前提**补齐。

也就是：

1. 让 Kafka 消费实例可以把消息转发到真正持有 websocket 连接的实例
2. 先做双实例功能验证
3. 确认“多 member 不丢消息、不串消息”

### 2.2 第二层目标

在第一层正确性通过后，再做真正的多 member 吞吐实验。

也就是：

1. 起多个 chat server 实例
2. 让多个实例加入同一个 consumer group
3. 用同口径压测对比 `1 member` 和 `N members` 的吞吐、`p95`、热点变化

---

## 3. 第二次执行这次具体要做什么

这次建议分成 4 步。

## 3.1 Step A：补跨实例 websocket 分发能力

### 目标

让消息消费实例和 websocket 连接实例不一致时，消息仍然能正确送达。

### 建议做法

先落一个最小可用方案：

1. 每个 chat server 实例在 Redis 里登记自己的在线用户路由
2. Kafka 消费实例先查目标用户当前连在哪个实例
3. 如果用户就在本机，直接走本机 `Clients`
4. 如果用户在其他实例，就通过 Redis Pub/Sub 把消息转发到对应实例
5. 对应实例收到后，再写回本机 websocket

### 为什么这一步优先级最高

因为这一步不是优化项，而是**多 member 能否成立的前提条件**。

没有这一步：

1. 多实例吞吐结果不可信
2. 单聊和群聊都会出现“消费成功但用户没收到”的假象

---

## 3.2 Step B：做双实例功能验证

### 目标

先不急着上高压，先证明多实例链路是通的。

### 要验证的场景

至少验证下面两类：

1. 单聊跨实例
   - 发送者连在实例 `S1`
   - 接收者连在实例 `S2`
   - 消息由任一实例消费，都能正确送达
2. 群聊跨实例
   - 发送者和群成员分散在多个实例
   - 群消息能正确回推到不同实例上的在线成员

### 验收标准

1. 消息能正常送达
2. 发送者回显正常
3. 群聊多接收者都能收到
4. Kafka group 日志里能看到不止 `1 member`

---

## 3.3 Step C：补多实例压测入口

### 目标

让现有压测脚本能真正打到多个 chat server 实例，而不是只打单实例。

### 要解决的问题

当前压测脚本默认只有一组：

1. `base_url`
2. `ws_base_url`

这意味着它现在天然只会连到一个实例。

所以第二次执行里，要么：

1. 给压测脚本增加多地址分配能力

要么：

1. 准备一个本地负载入口，把 websocket 连接分散到多个实例

### 验收标准

压测过程里，连接和消费都必须真正分散到多个实例，而不是只有 consumer group 多 member、连接还都集中在单实例。

---

## 3.4 Step D：跑第二次执行的第一轮多 member 压测

### 目标

在“跨实例正确性”通过后，先跑一轮保守版多 member 对比。

### 建议起步规模

先不直接上 `8 members`，从保守版开始：

1. `2 members`
2. 如果收益明确，再上 `4 members`

### 为什么先这样跑

因为当前系统还是：

1. websocket 和 consumer 同进程
2. 单机 5 broker
3. 单机多 chat server 实例

所以第二次执行第一轮更适合先看：

1. 正确性有没有问题
2. 2 member 是否已经有收益
3. 4 member 是否继续有收益

而不是一上来把部署复杂度拉满。

---

## 4. 第二次执行范围

这次第二次执行，建议固定下面这些前提：

1. Kafka 形态仍然是 `5 broker`
2. chat topic 仍然是 `240 partitions`
3. `mysql_persist` 停在第一次执行验证过的第二档
4. 先不继续改 `mysql_persist`
5. 先不动 `session_seq`
6. 先不动 commit 参数

这次只允许变化的主变量：

1. chat server 实例数
2. consumer group member 数
3. 多实例连接分布方式
4. 跨实例 websocket 分发路径

---

## 5. 第二次执行的通过标准

这次第二次执行至少要能回答下面 4 个问题：

1. 多实例下，消息能不能正确跨实例送达到 websocket 连接
2. `2 members` 相比 `1 member`，吞吐有没有提升
3. `4 members` 相比 `2 members`，吞吐有没有继续提升
4. 群聊热点在多 member 下有没有被真正改善

---

## 6. 第二次执行不做什么

这次明确不做下面 4 件事：

1. 不继续沿 `mysql_persist` 第四档、第五档往上堆
2. 不把 `consumer member` 一口气直接拉到 `8`
3. 不同时混入 commit 参数调整
4. 不直接做 `session_seq` 大改

---

## 7. 第二次执行的预期产出

这次第二次执行结束后，至少要形成两份文档：

1. `优化第二次执行_第一次结果.md`
   先记录跨实例分发改造和双实例功能验证结果
2. `优化第二次执行_第二次结果.md`
   再记录多 member 吞吐实验结果

也就是说，这次第二次执行不会追求“一次把所有结果写完”，而是：

1. 先把前置正确性打通
2. 再进入真正的吞吐对比
