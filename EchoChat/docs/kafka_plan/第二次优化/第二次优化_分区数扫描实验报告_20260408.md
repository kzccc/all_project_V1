# 第二次优化：Kafka 分区数扫描实验报告

## 1. 这次报告回答什么问题

这次不再只看“单聊极限吞吐最高是多少”，而是要回答两个更实用的问题：

1. 当前单聊在 **`p95 不要太高，最好亚秒级`** 的前提下，分区数到底该配多少
2. 为什么单聊分区数一加就涨，群聊却基本不跟着涨

本报告里的“最佳分区”分两种口径：

1. `极限口径`
   看最大吞吐，不限制 p95
2. `可用口径`
   只收 `success = 1.0` 且 `p95 < 1000ms` 的档位

你这次真正关心的，是第 2 种。

---

## 2. 实验方法

### 2.1 固定压测口径

所有实验都固定：

1. `MODE = kafka`
2. `SINGLE_PAIR_COUNT = 60`
3. `GROUP_MEMBER_LIMIT = 25`
4. `SINGLE_MIN_DURATION_SEC = 8`
5. `GROUP_MIN_DURATION_SEC = 8`

这里的单聊模型是：

1. `60` 对单聊会话并发发消息
2. Kafka 消息 key 用 `session_id`
3. 同一个单聊会话固定进同一个分区

### 2.2 为什么每个分区数都要用独立 topic

Kafka 分区数只能增不能减。

所以这次每个分区数都用了独立 `chatTopic`，避免：

1. 先跑大分区
2. 再跑小分区
3. 导致小分区实验被污染

对应配置都放在：

[configs/partition_scan](/workspace/czk/Personal/EchoChat/configs/partition_scan)

### 2.3 这次怎么“结合二分法”

这次我分两段跑：

#### 第一段：粗扫

先跑完整容量脚本，摸清趋势：

1. `3`
2. `4`
3. `6`
4. `8`
5. `10`
6. `12`
7. `20`

作用是先确认：

1. 单聊是不是会随着分区数增加持续上涨
2. 群聊是不是基本不吃这个红利

#### 第二段：围绕“亚秒级 p95”继续上探

粗扫确认趋势后，再改成定点诊断脚本，只扫单聊关键 target 段：

1. `20`
2. `40`
3. `50`
4. `60`
5. `80`
6. `120`
7. `240`
8. `480`

这一步的目的不是再找极限，而是专门找：

**`p95 < 1000ms` 时，单聊还能跑到多高。**

### 3.1 单聊极限吞吐趋势

| --- | ---: | ---: | ---: |
| 3 | `419.907 msg/s` | `56518 ms` | `41.67%` |
| 4 | `511.692 msg/s` | `56045 ms` | `30.00%` |
| 6 | `607.298 msg/s` | `49490 ms` | `25.00%` |
| 8 | `769.373 msg/s` | `45000 ms` | `16.67%` |
| 10 | `872.880 msg/s` | `21998 ms` | `13.33%` |
| 12 | `906.688 msg/s` | `55066 ms` | `14.98%` |
| 20 | `1111.473 msg/s` | `37438 ms` | `10.00%` |

从这张表已经能先收下两个结论：

1. 单聊确实会随着分区数增加继续上涨
2. 真正被打散的是“热分区先打满”这个问题

### 3.2 群聊极限吞吐趋势

| 分区数 | 群聊极限吞吐 | 群聊折算消息数 | 群聊 receipt p95 |
| --- | ---: | ---: | ---: |
| 3 | `5391.511 delivery/s` | `225 group_msg/s` | `2892 ms` |
| 4 | `5208.349 delivery/s` | `217 group_msg/s` | `3597 ms` |
| 6 | `5465.731 delivery/s` | `228 group_msg/s` | `2911 ms` |
| 8 | `5430.131 delivery/s` | `226 group_msg/s` | `2804 ms` |
| 10 | `5416.956 delivery/s` | `226 group_msg/s` | `2839 ms` |
| 12 | `5535.598 delivery/s` | `231 group_msg/s` | `2642 ms` |
| 20 | `5457.722 delivery/s` | `227 group_msg/s` | `2720 ms` |

群聊这几轮一直都在 `5200 ~ 5535 delivery/s` 之间波动，没有像单聊那样大涨。

原因很直接：

1. 单聊压测是 `60` 个不同会话
2. 每个会话都有自己的 `session_id`
3. 所以单聊可以被分散到更多分区

但群聊这轮压测不是多群，而是：

1. 一个群
2. 一个群会话
3. 一个 `session_id`

所以这个群会话本质上还是固定压在一个分区里。

也就是说：

**分区扩容对单聊有用，是因为单聊有很多不同 key。**

**分区扩容对当前这轮群聊基本没用，是因为群聊只有一个 key。**

## 4. 第二段定点实验：找亚秒级 p95 下的最佳分区

### 4.1 低 p95 口径下的最好结果

下面这张表，统一只看：

1. `success = 1.0`
2. `p95 < 1000ms`

然后从每个分区数里，取单聊吞吐最高的一档。

| 分区数 | 最佳可用吞吐 | target | offered | p95 |
| --- | ---: | ---: | ---: | ---: |
| 20 | `1018.371 msg/s` | `1320` | `1333.3` | `506 ms` |
| 40 | `1351.013 msg/s` | `1920` | `1935.5` | `681 ms` |
| 50 | `1470.823 msg/s` | `1920` | `1935.5` | `718 ms` |
| 60 | `1568.172 msg/s` | `1920` | `1935.5` | `100 ms` |
| 80 | `1962.705 msg/s` | `2560` | `2608.7` | `589 ms` |
| 120 | `2193.215 msg/s` | `3120` | `3157.9` | `349 ms` |
| 240 | `2658.204 msg/s` | `3840` | `3750.0` | `791 ms` |
| 480 | `2737.602 msg/s` | `3840` | `3750.0` | `339 ms` |

### 4.2 这张表的核心结论

先说最直接的：

1. 到目前为止，**`p95 < 1000ms` 口径下，最佳测试值是 `480` 分区**
2. 对应结果是：
   `2737.602 msg/s @ p95 339ms`

再看增长幅度：

1. `20 -> 40`：从 `1018` 提到 `1351`
2. `40 -> 60`：从 `1351` 提到 `1568`
3. `60 -> 80`：从 `1568` 提到 `1963`
4. `80 -> 120`：从 `1963` 提到 `2193`
5. `120 -> 240`：从 `2193` 提到 `2658`
6. `240 -> 480`：从 `2658` 提到 `2738`

注意最后一段：

1. `240 -> 480` 只提升了约 `+3.0%`

这说明：

**到 `240` 以后，收益已经开始明显放缓。**

## 5. 为什么 `240` 和 `480` 已经接近平台

### 5.1 `240` 分区时已经接近“两会话热点”

`240` 分区、最佳亚秒级档位 `target=3840` 时：

1. 实际用到的分区数：`51`
2. 最热分区占比：`3.33%`

这里 `3.33%` 的意思很直观：

1. `60` 对会话里
2. 最热分区上大致还有 `2` 对会话撞在一起

也就是说，热点已经被压得比较散了。

### 5.2 `480` 分区时已经接近“一会话热点 + 少量碰撞”

`480` 分区、最佳亚秒级档位 `target=3840` 时：

1. 实际用到的分区数：`56`
2. 最热分区占比：`3.33%`
3. 其余很多分区已经降到 `1.67%`

`1.67%` 其实就是：

1. 一个分区里只有 `1` 对会话

所以 `480` 分区这一步，已经非常接近“每对会话单独分散”的状态了。

这也是为什么：

1. `240 -> 480` 还能涨
2. 但涨幅已经只剩 `3%`

### 5.3 真正没变的还是单条消息固定成本

这轮里有个现象一直没变：

1. `session_seq` 已经非常轻
2. `mysql_persist` 还是主耗时

也就是说，分区继续变多，真正改变的是：

1. 哪个热点先被打满

而不是：

1. 单条消息本身的处理成本突然变低了

所以分区扫描能持续把平台往上推，但它不是凭空创造吞吐，只是在：

**把原来被热点提前卡住的能力，尽量释放出来。**

## 6. 当前怎么给出“最佳分区设置”

这次我给两个答案。

### 6.1 只看当前实测数据，最佳测试值

如果你只问：

1. 当前这套单聊模型
2. `p95 < 1000ms`
3. 吞吐尽量高

那到目前为止，最佳测试值就是：

1. 分区数：`480`
2. 单聊吞吐：`2737.602 msg/s`
3. p95：`339 ms`
4. success：`1.0`

### 6.2 如果你要一个更“工程化”的默认值

如果你不是只看 benchmark，而是还要考虑：

1. Kafka broker 元数据开销
2. topic 管理复杂度
3. rebalance 成本
4. 生产环境运维复杂度

那当前更稳一点的工程建议是：

1. 分区数：`240`

因为：

1. `240` 已经做到 `2658.204 msg/s @ p95 791ms`
2. 比 `480` 只少约 `3.0%`
3. 但分区数只有 `480` 的一半

所以现在可以这么理解：

1. `480`：当前最佳测试值
2. `240`：当前更像默认工程值

## 7. 为什么群聊和单聊差异这么大

把这件事用最直白的话说：

### 单聊

1. 现在压测是 `60` 对单聊
2. 也就是 `60` 个不同会话
3. Kafka 可以把这些会话分到更多分区
4. 所以单聊能持续吃到分区扩容红利

### 群聊

1. 现在压测是 `1` 个群
2. 也就是 `1` 个群会话
3. Kafka 还是只能把这个群会话放到 `1` 个分区
4. 所以你把 topic 从 `3` 分区加到 `480` 分区，对当前群聊模型几乎没帮助

群聊现在更像是被后面的链路卡住：

1. 消费后扇出给多个成员
2. websocket 写回
3. 状态更新
4. 缓存更新
5. 持久化

所以群聊要想吃到分区扩容红利，必须换压测模型：

1. 不是一个群一直发
2. 而是多个群同时发

只有这样，不同群会话才能真正落到多个分区。

---

## 8. 结论

这次可以把结论收成 4 句话：

1. 当前单聊在 `p95 < 1000ms` 口径下，**最佳测试值** 是 `480` 分区，约 `2737.602 msg/s @ p95 339ms`
2. 当前单聊在工程上更稳一点的 **推荐默认值** 可以先取 `240` 分区，约 `2658.204 msg/s @ p95 791ms`
3. 当前群聊基本不随分区数上涨，根本原因不是代码没变，而是 **当前群聊压测模型只有一个群会话**
4. 到这一步，单聊“热分区先打满”问题已经被拆得很开了，后面如果还想继续提，就该继续盯 `mysql_persist` 和群聊下游扇出链路

这轮不是单聊定点诊断脚本，而是回到常规一键吞吐脚本：

1. 单聊 + 群聊一起跑

对应结果目录：

1. 单聊真实可用容量：约 `2372 msg/s`
2. 群聊真实可用容量：约 `4464 delivery/s`
3. 群聊折算消息数：约 `186 group_msg/s`

这里的“可用容量”口径是：

1. success / coverage 过线
2. `p95 < 1000ms`

单聊这轮的关键档位是：

1. 最佳通过档：`2372.169 msg/s @ p95 810.05ms`
2. 对应 `target=3360`
3. 再往上一档 `target=3600` 虽然吞吐到 `2404.705 msg/s`，但 `p95 = 1272ms`，所以按新标准判失败

群聊这轮的关键档位是：

1. 最佳通过档：`4464.231 delivery/s @ p95 11ms`
2. 对应 `target=5760`
3. 再往上一档 `target=11520` 虽然吞吐到 `5481.783 delivery/s`，但 `p95 = 2928ms`，所以按新标准判失败

所以从“常规一键脚本 + 单聊群聊一起跑”的视角看，当前 `240` 分区可以先记成：

1. 单聊：约 `2372 msg/s`
2. 群聊：约 `4464 delivery/s`
3. 群聊折算：约 `186 group_msg/s`

需要注意：

1. 这个结果和上面的“单聊定点诊断脚本”不是同一口径
2. 单聊定点诊断脚本重跑时，`240` 分区在 `3840` 档还能跑到 `2567.151 msg/s @ p95 998ms`
3. 所以你后面看结果时，要先分清楚自己是在看“常规一键口径”，还是“单聊定点口径”
