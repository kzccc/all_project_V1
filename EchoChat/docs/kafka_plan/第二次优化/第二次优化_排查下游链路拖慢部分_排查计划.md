# 第二次优化：排查下游链路拖慢部分排查计划

## 1. 这次排查要解决什么问题

第一轮 Kafka 优化已经解决了几类“明显错误”问题：

1. offset commit 过频导致消费主线程阻塞
2. topic 分区和压测配置不一致
3. consumer readiness 空窗导致起步消息被跳过
4. 一部分重复 `status=sent` 更新和持锁分发问题

这些问题修完后，当前压测已经能稳定复现新的真实瓶颈：

1. 单聊真实处理平台大约已经出现在 `350 msg/s`
2. 群聊真实处理平台大约已经出现在 `4000 delivery/s`

当前最重要的问题已经不再是：

1. Kafka 有没有成功拉到消息
2. consumer 有没有 ready
3. 会不会固定漏起步消息

而是：

**Kafka 把消息拉出来之后，后面的哪一段业务链路先饱和，最终把整体吞吐卡住了。**

这份计划的目标就是把这个问题系统地查清楚。

---

## 2. 这次排查的核心思路

这次排查不是先靠猜、也不是先改一堆代码试运气。

要按下面这条路径推进：

1. 先把 Kafka 消费后的下游链路拆清楚
2. 明确现有指标已经覆盖了哪些阶段、还缺哪些关键观测点
3. 固定几档代表性压力，做同口径定位
4. 用指标 + pprof 找出最先变陡的阶段
5. 再用旁路实验验证“是不是它”
6. 最后才进入真正的吞吐优化改造

也就是说，这次第二轮优化的重点不是“马上优化”，而是先做到：

**把瓶颈归因从“怀疑”变成“证据闭环”。**

## 5. Kafka 消费后的下游链路拆解

## 5.1 单聊链路

当前单聊消费后，一条消息大致会经过下面这些阶段：

1. `deserialize`
   反序列化 Kafka payload
2. `session_seq`
   计算会话内顺序号
3. `mysql_persist`
   消息落库
4. `websocket_dispatch`
   给发送者 / 接收者找到在线连接并投递到 `SendBack`
5. `redis_read`
   读取单聊消息列表缓存
6. `redis_write`
   把 append 后的新列表整段写回 Redis
7. `MarkMessage / Commit`
   标记并提交 offset

其中：

1. `session_seq` 逻辑在 [message_sequence.go](/workspace/czk/Personal/EchoChat/internal/service/chat/message_sequence.go)
2. `mysql_persist` 逻辑在 [kafka_message_support.go](/workspace/czk/Personal/EchoChat/internal/service/chat/kafka_message_support.go)
3. 单聊 `websocket_dispatch + redis` 逻辑在 [kafka_server.go](/workspace/czk/Personal/EchoChat/internal/service/chat/kafka_server.go)

## 5.2 群聊链路

群聊在单聊基础上额外多了：

1. `group_member_query`
   查群成员列表
2. 多接收者 fanout 的 `websocket_dispatch`
3. 群聊消息列表缓存的 `redis_read / redis_write`

也就是说：

1. 单聊更可能卡在“单条消息固定成本”
2. 群聊更可能卡在“成员查询 + fanout + 更大的缓存写放大”

## 7. 当前指标还不够的地方

虽然已经有消费阶段耗时，但当前还不能直接回答“到底是谁把整体吞吐卡在这里”，因为还存在几个盲区。

## 7.1 WebSocket 真正写回耗时没有单独量出来

当前 `websocket_dispatch` 指标量的是：

1. 找在线客户端
2. 往 `SendBack` 通道里投递

但真正的网络写发生在：

[client.go](/workspace/czk/Personal/EchoChat/internal/service/chat/client.go)

具体是：

1. `Conn.WriteMessage`
2. 写回成功后的 `status=sent` DB 更新

所以现在还不能分清：

1. 是 consumer 自己慢
2. 还是 `SendBack` 通道被下游写回背压堵住

## 7.2 `status=sent` 更新没有和 websocket 写分开计时

当前 `writeMessageBack()` 里：

1. 先写 websocket
2. 再做 `status=sent` 更新

这两者的耗时现在混在一起，没有单独指标。

所以无法判断：

1. 是网络写慢
2. 还是 DB update 慢

## 7.3 MySQL 连接池等待情况没有暴露

当前虽然知道 `mysql_persist` 阶段耗时，但还不知道：

1. SQL 自身慢
2. 还是连接池等待慢

如果没有 `sql.DB.Stats()` 指标，就无法判断高压时是不是连接池被打满。

## 7.4 Redis 命令耗时还不够细

当前只有业务阶段上的 `redis_read / redis_write`，但没有拆：

1. `GET`
2. `SET`
3. `SETNX`
4. `INCR`

所以还不能确认：

1. 是 Redis 本身慢
2. 还是“整块 JSON 读改写”的业务模式在放大成本

## 7.5 当前没有直接量 `SendBack` 队列背压

也就是说，我们还不知道高压时：

1. `SendBack` 队列是不是长期接近满
2. `enqueueBack()` 是不是在阻塞等待队列腾位置

如果这点不量出来，就很容易把“下游写回慢”误判成“consumer 同步处理慢”。

## 8. 当前最值得优先怀疑的瓶颈点

在没有进一步补指标前，按代码结构和当前现象，优先怀疑顺序如下。

## 8.1 单聊优先怀疑项

### 候选 1：`session_seq`

原因：

1. 每条消息都要先查 MySQL `MAX(session_seq)`
2. 然后再走 Redis `SetNX + Incr`
3. 这是典型固定成本
4. 单聊高压时这段会被每条消息放大

### 候选 2：WebSocket 写回链路

包括：

1. `enqueueBack`
2. `SendBack` 队列等待
3. `Conn.WriteMessage`
4. `status=sent` 更新

原因：

1. 这条链跨 consumer -> channel -> websocket -> DB update
2. 当前指标还没有拆透
3. 很可能是高压时最容易形成背压的地方

### 候选 3：单聊 Redis 整块缓存覆写

原因：

1. 当前是读整个 message list JSON
2. 反序列化
3. append
4. 再整段写回

这类模式很容易在高压下形成大 value 放大。

## 8.2 群聊优先怀疑项

### 候选 1：群成员查询 + fanout

原因：

1. 每条群消息要先查成员
2. 然后 fanout 给多个接收者
3. 群人数越多，放大越明显

### 候选 2：WebSocket 真写回

原因：

1. 群聊不是单次投递，而是同一条消息向多个 receiver 写
2. 即使 `coverage` 还保持 `1.0`，也可能已经在后台排队

### 候选 3：群聊 Redis 整块缓存覆写

原因：

1. 群消息列表也在做整块 JSON 读改写
2. 群历史增长时成本会持续放大

## 9. 这次排查的总体方法

这次定位要用“三件套”：

1. 分阶段耗时指标
2. pprof
3. 旁路实验

三者缺一不可。

### 只看指标的问题

只能告诉我们“哪个阶段看起来慢”，但不一定能证明它就是主瓶颈。

### 只看 pprof 的问题

能看到 CPU / block / mutex 热点，但不一定知道这些热点是在什么业务阶段下出现的。

### 只做旁路实验的问题

如果没有阶段耗时和 pprof 支撑，实验结果容易被误读。

所以必须把三者串起来看。

## 10. 排查阶段设计

本次排查分四个阶段。

## 阶段 A：补齐缺失观测点

目标：

先把关键盲区补上，让后续定位有足够证据。

### A.1 需要补的指标

#### WebSocket 相关

1. `ws_write_duration_seconds`
   单独量 `Conn.WriteMessage`
2. `ws_status_update_duration_seconds`
   单独量 `status=sent` update

   记录 `len(SendBack)`

   量 `enqueueBack` 等待时长

2. `mysql_in_use_connections`
3. `mysql_idle_connections`
4. `mysql_wait_count_total`
5. `mysql_wait_duration_seconds`

#### Redis 相关

1. `redis_command_duration_seconds{cmd="get"}`
2. `redis_command_duration_seconds{cmd="set"}`
3. `redis_command_duration_seconds{cmd="setnx"}`
4. `redis_command_duration_seconds{cmd="incr"}`

### A.2 阶段 A 产出

产出要求：

1. 新指标可在 `/metrics` 里看到
2. 压测 stage 结束后能抓到这些指标
3. 不改业务语义，只补观测

## 阶段 B：固定代表档位做定位压测

目标：

在不改业务逻辑的前提下，看哪一段最先从“健康态”进入“饱和态”。

### B.1 单聊固定档位

建议固定三档：

1. `240`
2. `960`
3. `2880`

原因：

1. `240` 基本接近健康档
2. `960` 已经明显进入积压前沿
3. `2880` 是稳定复现极限问题的饱和档

### B.2 群聊固定档位

建议固定三档：

1. `1440`
2. `5760`
3. `11520`

原因：

1. `1440` 基本健康
2. `5760` 已接近当前平台
3. `11520` 明显进入长尾恶化区

### B.3 每档需要固定的条件

为了让结果可比，每档都要固定：

1. 同一份代码
2. 同一套 fixture
3. 同一台机器
4. 同一套 MySQL / Redis / Kafka 环境
5. 同一套压测参数

禁止同时改：

1. 并发人数
2. 数据规模
3. 机器配置
4. 配置项

否则很难比较。

### B.4 每档需要采集的东西

每一档必须采集：

1. `summary.json`
2. `metrics.prom`
3. CPU profile
4. block profile
5. mutex profile
6. goroutine profile

### B.5 阶段 B 判读方法

对每一档都看：

1. 哪个 stage 的耗时先开始明显上涨
2. 哪个 stage 的上涨和 `consumer lag` 抬头同步出现
3. 哪类 goroutine 在 block profile 里占比最高
4. ws / db / redis 资源等待是否在同一时刻抬升

阶段 B 的目标不是立刻下结论，而是缩小嫌疑范围。

## 阶段 C：旁路实验验证主瓶颈

目标：

把阶段 B 里的“高概率嫌疑点”做成可控实验，验证绕过它后吞吐会不会明显提升。

### C.1 单聊实验顺序

建议按下面顺序做：

1. 关闭单聊 Redis 缓存读写
2. 关闭 `status=sent` 更新
3. mock websocket 真写回
4. 给 `session_seq` 做实验版实现

#### 实验 1：关闭单聊 Redis 缓存读写

看什么：

1. 单聊 `observed` 是否明显高于 `350`
2. `p95` 是否明显下降
3. consumer `redis_read / redis_write` 是否从热点阶段消失

#### 实验 2：关闭 `status=sent` 更新

看什么：

1. ws 写回链路是否明显变快
2. MySQL wait / write 时间是否明显下降
3. 单聊吞吐是否明显上升

#### 实验 3：mock websocket 真写回

做法：

1. consumer 仍然做 enqueue 或统计
2. 但 `Conn.WriteMessage` 不真正发网络

看什么：

1. 如果吞吐显著跳升，说明真正的 ws 写回就是重要瓶颈

#### 实验 4：给 `session_seq` 做实验版实现

做法：

1. 暂时绕过每条消息的 MySQL `MAX(session_seq)`
2. 改为纯 Redis 递增或预热缓存方案

看什么：

1. 如果低压中压高压各档都明显抬升，则 `session_seq` 是重要固定成本瓶颈

### C.2 群聊实验顺序

建议按下面顺序做：

1. 缓存群成员列表
2. 关闭群聊 Redis 缓存读写
3. mock websocket 真写回

#### 实验 1：缓存群成员列表

看什么：

1. `group_member_query` 是否从阶段热点消失
2. 群聊平台是否从 `~4000 delivery/s` 上移

#### 实验 2：关闭群聊 Redis 缓存读写

看什么：

1. `redis_read / redis_write` 是否还是主要拖累
2. `p95` 是否明显收敛

#### 实验 3：mock websocket 真写回

看什么：

1. fanout 后真正的 socket 写是不是主瓶颈

### C.3 旁路实验的判定标准

某个阶段要被认定为“主瓶颈”或“主瓶颈之一”，至少要满足下面两条：

1. 它在阶段 B 里已经表现出明显的耗时/阻塞抬升
2. 绕过它后，整体吞吐明显上升，或 `lag/p95` 明显改善

如果只满足一条，还不能坐实。

## 11. 这次排查时要重点看的证据

## 11.1 指标证据

重点关注：

1. 哪个 stage 的 `p95/p99` 最先开始陡增
2. 哪个 stage 的总耗时占比越来越大
3. `consumer lag` 什么时候开始抬头
4. ws 队列长度是否长期高位
5. MySQL wait 是否突然上升
6. Redis 命令耗时是否明显变大

## 11.2 pprof 证据

重点关注：

1. CPU profile
   看时间花在 JSON、GORM、Redis、websocket 还是别的地方
2. block profile
   看 goroutine 主要堵在 channel、socket 还是 DB
3. mutex profile
   看有没有锁竞争热点
4. goroutine profile
   看高压时大量 goroutine 堵在哪里

## 11.3 实验证据

重点关注：

1. 绕开某段后吞吐有没有明显提升
2. `lag` 和 `p95` 是否同步改善
3. 改善是只出现在单聊，还是群聊也一起变化
