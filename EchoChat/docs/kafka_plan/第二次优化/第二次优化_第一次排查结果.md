# 第二次优化：第一次排查结果

## 1. 本轮排查目标

本轮没有直接修改业务逻辑，也没有直接开始“优化吞吐”。

本轮的目标是：

**先补齐第二次优化里最关键的一批观测点，把 Kafka 消费后的下游链路盲区缩小。**

原因是当前虽然已经知道：

1. 单聊真实处理平台大约在 `350 msg/s`
2. 群聊真实处理平台大约在 `4000 delivery/s`

但还不能精确回答：

1. 是 consumer 同步处理本身慢
2. 还是 websocket 下游写回慢，反向把 consumer 卡住
3. 是 MySQL 真正执行慢
4. 还是连接池等待把链路拖慢
5. 是 Redis 命令本身慢
6. 还是整块 JSON 读改写模式在放大成本

所以这一步先做“补观测”，这是第二次优化的第一步。

---

## 2. 本轮选择的工作步骤

本轮按排查计划，选择执行：

**阶段 A：补齐缺失观测点**

并且优先补了最影响判断的三类：

1. `ws` 写回链路
2. `MySQL` 连接池
3. `Redis` 命令耗时

之所以先做这一步，是因为当前最危险的误判是：

看起来像是 Kafka consumer 后面的同步处理慢，但实际可能是 websocket 写回或 DB 更新在后面堵住，进而把 `SendBack` 通道打满，最后反向拖慢 consumer。

如果不先补这批指标，后面就算继续压测，也只能得到“整体慢了”，还是很难坐实到底是谁在拖慢。

## 3. 本轮实际做了什么

## 3.1 补了 websocket 下游写回链路指标

修改文件：

1. [metrics.go](/workspace/czk/Personal/EchoChat/internal/observability/metrics.go)
2. [client.go](/workspace/czk/Personal/EchoChat/internal/service/chat/client.go)

新增指标：

1. `echochat_ws_sendback_queue_length`
2. `echochat_ws_sendback_enqueue_duration_seconds`
3. `echochat_ws_write_duration_seconds`
4. `echochat_ws_status_update_duration_seconds`

### 这批指标分别在回答什么问题

#### 1. `echochat_ws_sendback_queue_length`

记录 enqueue 前 `SendBack` 队列长度采样。

它主要回答：

1. 高压时 `SendBack` 队列是不是长期高位
2. websocket 下游是不是已经在形成明显背压

#### 2. `echochat_ws_sendback_enqueue_duration_seconds`

记录往 `SendBack` 通道塞消息花了多久。

它主要回答：

1. `enqueueBack()` 是不是本身就在等
2. consumer 里的 `websocket_dispatch` 慢，到底是不是因为往通道里塞不进去

#### 3. `echochat_ws_write_duration_seconds`

单独记录 `Conn.WriteMessage` 的耗时。

它主要回答：

1. 真正慢的是不是网络写
2. ws 写回是不是单聊/群聊平台的关键拖慢点

#### 4. `echochat_ws_status_update_duration_seconds`

单独记录 websocket 写回成功后，`status=sent` 的 DB 更新耗时。

它主要回答：

1. 是 socket 写慢
2. 还是“消息已送达后的状态更新”慢

这一步的意义很大，因为之前 `writeMessageBack()` 里这两段是混着的，看不出来谁更慢。

如果高压时 `wait_count` 和 `wait_duration` 明显上涨，那就说明“拖慢”不一定在 SQL 语句本身，也可能在连接池资源竞争。

## 13. 第一轮定位里最关键的指标结论

这一轮真正有价值的，不是 summary 本身，而是新补的指标和 stage duration 联合起来后的结论。

## 13.1 websocket 下游目前不像主瓶颈

这轮最重要的排除项是：

**当前主瓶颈看起来不是 websocket 下游写回。**

依据如下。

### 1. `SendBack` 队列几乎没有积压

从 `echochat_ws_sendback_queue_length` 看：

1. 单聊 `240 / 960 / 2880`
2. 群聊 `1440 / 5760 / 11520`

这几档的平均队列长度基本都接近 `0`

这意味着：

1. consumer 往 `SendBack` 塞消息时，没有出现明显“通道打满”
2. 当前 `enqueueBack()` 不是主要拖慢点

### 2. `enqueueBack` 本身非常快

从 `echochat_ws_sendback_enqueue_duration_seconds` 看：

1. 单聊平均大约 `0.002 ms`
2. 群聊平均大约 `0.001 ms`

这说明：

1. consumer 往下游通道投递本身几乎不耗时
2. `websocket_dispatch` 虽然在链路里，但不是当前最重阶段

### 3. 真正的 websocket 写非常快

从 `echochat_ws_write_duration_seconds` 看：

1. 单聊三档平均都大约 `0.015 ms`
2. 群聊三档平均大约 `0.006 ~ 0.008 ms`

这说明：

1. 真正的 `Conn.WriteMessage` 不是当前主瓶颈
2. 至少在这轮定位里，没有看到“socket 写慢导致系统卡死”的证据

### 4. `status=sent` 更新明显比 socket 写贵，但它不在 consumer 同步主链路内

从 `echochat_ws_status_update_duration_seconds` 看：

#### 单聊

1. `240` 档约 `6.689 ms`
2. `960` 档约 `8.469 ms`
3. `2880` 档约 `5.440 ms`

#### 群聊

1. `1440` 档约 `3.628 ms`
2. `5760` 档约 `7.894 ms`
3. `11520` 档约 `9.045 ms`

这说明：

1. 写回成功后的 DB 状态更新并不便宜
2. 但它属于下游异步链路
3. 当前它更像是一个“额外数据库负载源”，而不是 consumer 同步主线程里的第一主瓶颈

换句话说：

1. 它值得后续继续优化
2. 但目前还没有证据说明它是把单聊卡在 `~300 msg/s`、把群聊卡在当前平台的第一原因

## 14. 分区分布给出的额外结论

这轮还有一个很关键的新证据：

### 单聊

单聊 producer 分区分布依然是三分区工作：

1. `partition 0`
2. `partition 1`
3. `partition 2`

例如 `2880` 档：

1. `6400 / 11200 / 6400`

说明：

1. 单聊不是被“所有消息都打到一个分区”卡住
2. 单聊当前平台更像是“三个消费分区分别都被同步业务成本卡住”

### 群聊

群聊这轮所有消息都落在同一个分区：

1. `target_1440 -> partition 2`
2. `target_5760 -> partition 2`
3. `target_11520 -> partition 2`

说明：

1. 当前群聊定位压测只有一个群
2. 同一个群的消息 key 稳定落在同一分区
3. 群聊当前平台，本质上是**单分区下一个消费链路实例的处理上限**

这点非常重要，因为它让群聊结果更容易解释：

1. `5760` 档观测值 `1897.939 delivery/s`
2. 换算成群消息数约 `1897.939 / 24 ≈ 79.1 group_msg/s`
3. `11520` 档观测值 `1681.775 delivery/s`
4. 换算成群消息数约 `1681.775 / 24 ≈ 70.1 group_msg/s`

而这两个值，和 `total ≈ 11.728 ms / 13.744 ms` 对应的“单条群消息同步处理能力”非常接近。

所以群聊当前结果基本可以理解成：

**单群消息 key 固定落单分区后，真正把群聊平台卡住的也是这个分区上单条消息的同步业务处理成本。**

---

## 15. pprof 这轮能提供什么、不能提供什么

这轮 `pprof` 有产出，但也有局限。

### 15.1 本轮 pprof 的有效结论

`goroutine` profile 里能看到：

1. 消费 goroutine 正常存在
2. MySQL watcher goroutine 数量不低
3. 没有看到大量 goroutine 堵在 websocket 写通道上

这和指标结论是一致的：

1. websocket 下游当前不像第一主瓶颈

### 15.2 本轮 pprof 的局限

这轮：

1. `cpu profile` 只有少数 stage 采到了有效样本
2. `block` profile 基本没有有效数据
3. `mutex` profile 也没有明显热点

这说明：

1. 当前默认的 block/mutex 采样设置还不够支持深挖锁竞争或阻塞
2. 下一轮如果要继续用 pprof 深拆，最好显式打开：
   - `runtime.SetBlockProfileRate`
   - `runtime.SetMutexProfileFraction`

所以这轮 pprof 更多起到的是：

1. 辅助排除 websocket / 锁热点
2. 还不足以单独做最终归因

## 16. 本轮结论更新

结合这轮定位压测，当前已经可以把问题进一步收敛成下面这版结论。

### 结论 1：当前第一嫌疑点已经从“下游 ws / Redis / 连接池等待”收敛到“consumer 同步 DB 固定成本”

更准确地说：

1. `session_seq`
2. `mysql_persist`

是当前最像第一主瓶颈的两段。

### 结论 2：单聊当前更像是“三分区并行下，每分区都被同步业务成本限制”

也就是：

1. 单聊不是分区没打散
2. 而是每个分区上的单条消息同步处理成本仍然偏高

### 结论 3：群聊当前更像是“单群 key 固定落单分区后，被单分区同步处理能力限制”

也就是：

1. 群聊不是全链路 fanout 写回先炸
2. 更像是同一个群在单分区上的消费业务处理先到平台

### 结论 4：`status=sent` 更新值得后续继续打，但它现在更像次级成本，而不是这轮的第一根木板

原因是：

1. 它确实不便宜
2. 但它不在 consumer 同步主路径上
3. 当前还没有证据表明它先于 `session_seq + mysql_persist` 把平台卡住

---

## 17. 下一步建议

基于这轮结果，下一步不应该先去打 websocket。

更合理的顺序是：

### 优先级 1：继续拆 `session_seq`

因为当前 `session_seq` 里还包含：

1. 每条消息一次 MySQL `MAX(session_seq)`
2. 然后 Redis `SetNX + Incr`

建议下一步直接做实验：

1. 临时绕过每条消息的 `MAX(session_seq)` 查询
2. 只保留 Redis 递增实验版
3. 再跑同样定位档位

如果吞吐明显提升，就能基本坐实：

`session_seq` 是当前主瓶颈之一。

### 优先级 2：继续拆 `mysql_persist`

建议继续做实验：

1. 旁路掉真正落库
2. 或做轻量 mock persist
3. 再跑单聊 `960 / 2880` 和群聊 `5760 / 11520`

如果吞吐和 `p95` 明显改善，就能坐实：

`mysql_persist` 是当前主瓶颈之一。

### 优先级 3：后续再回头打 `status=sent`

它仍然值得优化，但建议放在：

1. `session_seq`
2. `mysql_persist`

之后。

## 18. 一句话总结

这轮第一次定位压测已经把问题显著收敛了：当前单聊和群聊的平台现象，第一嫌疑点已经不再是 websocket 写回、Redis 命令或 MySQL 连接池等待，而是 Kafka consumer 同步主链路里的数据库固定成本，尤其是 `session_seq` 和 `mysql_persist`；同时群聊还额外受“单群 key 固定落单分区”这一事实约束。
