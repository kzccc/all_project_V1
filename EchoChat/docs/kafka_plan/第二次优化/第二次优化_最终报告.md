## 4. 第二次优化全过程

### 4.1 第一阶段：先把真正瓶颈找出来

这一步先做的是“排查”，不是盲改。

主要结论是：

1. Producer 不是主瓶颈
2. `session_seq` 在每条消息上有额外查询成本
3. `mysql_persist` 是 Consumer 主链路里最重的一段固定成本
4. 单聊长期卡住，不只是 Kafka 本身慢，而是 Kafka 后面的业务链路在拖

这一阶段的意义是把优化方向收敛了：

1. 不再继续纠结 Producer
2. 重点转向 `session_seq`
3. 重点转向 `mysql_persist`
4. 后面所有改动都围绕“减少每条消息固定成本”展开

### 4.2 第二阶段：`session_seq` 正式化

这一步改了：

1. 不再每条消息都去 `message` 表查 `MAX(session_seq)`
2. 改成 Redis 主分配 `session_seq`
3. 用 Lua 保证 Redis 初始化原子性
4. 增加 `conversation_sequence` 高水位表
5. 旧会话第一次命中时，只做一次 floor 迁移

这一步解决的问题是：

1. `session_seq` 不再频繁打 MySQL
2. 消息主链路里序号生成明显变轻
3. MySQL 不再承担“每条消息一次 floor 查询”的角色

这一步之后的结果还没有立刻把单聊平台大幅抬高，但它把后面真正能提吞吐的前置条件补齐了。

### 4.3 第三阶段：`mysql_persist` 批量化和多 worker 化

这一步改了：

1. Gorm 打开 `SkipDefaultTransaction`
2. Gorm 打开 `PrepareStmt`
3. `mysql_persist` 从单条同步写，改成批量写
4. 按会话 scope 分流到不同 worker
5. 正常路径只走 insert
6. duplicate 才回退到慢路径

这一步之后，系统第一次出现了明确收益：

1. 单聊：`350 -> 349 msg/s`
2. 群聊：`4050 -> 4746 delivery/s`
3. 群聊折算：`169 -> 198 group_msg/s`

按第一次优化基线计算：

1. 单聊变化：约 `-0.3%`
2. 群聊提升：约 `+17.2%`
3. 群聊折算提升：约 `+17.2%`

这里最关键的事实是：

1. 单聊几乎没涨
2. 群聊已经明显上涨

这说明：

1. `mysql_persist` 这条路是对的
2. 但单聊的新主瓶颈还没有完全拆到

### 4.4 第四阶段：收敛 `message` 表索引和查询模型

继续往下拆后，又做了两块关键收敛：

1. 给 `message` 表新增 `conversation_key`
2. 历史查询统一走 `conversation_key + session_seq`
3. 新建复合索引 `idx_message_conversation_seq`
4. 删除旧的 `send_id / receive_id / session_id / session_seq` 单列索引

这样做的原因是：

1. 原来的索引模型同时放大读和写成本
2. 历史查询和写入模型没有真正统一
3. 大表越大，旧索引越会拖慢 insert

### 4.5 第五阶段：`status=sent` 改成异步批量更新

这一步改了：

1. websocket 写协程不再同步执行 `UPDATE message SET status=sent`
2. 改成后台多 worker 批量更新
3. 主写回路径只负责把消息真正发给前端

这一步的收益是单聊和群聊都继续上涨：

1. 单聊：`350 -> 387 msg/s`
2. 群聊：`4050 -> 5409 delivery/s`
3. 群聊折算：`169 -> 225 group_msg/s`

按第一次优化基线计算：

1. 单聊提升：约 `+10.6%`
2. 群聊提升：约 `+33.6%`
3. 群聊折算提升：约 `+33.1%`

这是第二次优化里第一个对单聊也有明确正收益的阶段。

### 4.6 第六阶段：继续调 `mysql_persist` 调度参数

这一步继续做了小优化：

1. `mysqlPersistFlushIntervalMs` 从 `5ms` 降到 `1ms`
2. `mysqlPersistWorkerCount` 调到 `8`

这样做的目的很明确：

1. 降低消息在批量落库器里的等待时间
2. 提高多分区、多会话场景下的并发落库能力

这一步之后，常规测压结果进一步抬到了：

1. 单聊：`417 msg/s`
2. 群聊：`5462 delivery/s`
3. 群聊折算：`228 group_msg/s`

按第一次优化基线计算：

1. 单聊提升：约 `+19.1%`
2. 群聊提升：约 `+34.9%`
3. 群聊折算提升：约 `+34.9%`

这一阶段可以视为“纯代码链路优化”的最终完成版。

### 4.7 第七阶段：继续定位为什么单聊还只在 `417` 左右

继续拆后发现，单聊新的限制项变成了：

1. `session_seq` 已经很轻，不再是主瓶颈
2. `mysql_persist` 仍然有固定 insert 成本
3. 更重要的是，单聊会话在少量分区上分布不均
4. 热分区先被打满，导致整体平台提前到顶

这一阶段最重要的结论不是“又优化出一波数据”，而是把单聊剩余瓶颈解释清楚了：

**单聊不是还卡在 `session_seq`，而是卡在“热分区过热 + `mysql_persist` 固定成本”。**

### 4.8 第八阶段：验证“热分区先打满”假设

为验证上面的判断，又做了一轮实验：

1. 单聊会话数从 `30` 对提到 `60` 对
2. Kafka 分区数从 `3` 提到 `6`

这轮实验结果：

1. 单聊：`606 msg/s`
2. 群聊：`5475 delivery/s`
3. 群聊折算：`228 group_msg/s`

和上一轮 `417 / 5462 / 228` 对比：

1. 单聊提升：约 `+45.3%`
2. 群聊变化：约 `+0.2%`

这一步证明了两件事：

1. 单聊确实吃分区扩容和会话规模扩大的红利
2. 群聊当前压测模型只有一个群会话，所以基本不吃这个红利

### 4.9 第九阶段：做分区扫描，找第二次优化的工程默认值

这一步不再只看“极限吞吐”，而是开始引入更实用的口径：

1. success 要过线
2. `p95 < 1000ms`

这一步得到两个关键结论：

1. `480` 分区是单聊当前最佳测试值
2. `240` 分区是更像工程默认值的平衡点

分区扫描报告里的关键结果是：

1. `240` 分区：`2658.204 msg/s @ p95 791ms`
2. `480` 分区：`2737.602 msg/s @ p95 339ms`

但因为：

1. `480` 只比 `240` 高约 `3.0%`
2. 分区数却翻倍

所以这一步最终给出的工程建议是：

1. 最佳测试值：`480`
2. 推荐工程默认值：`240`

### 4.10 第十阶段：把常规一键脚本改成真正的“可用容量”口径

最后，我把常规一键脚本的过线规则改掉了：

1. 以前主要看 success / coverage
2. 现在增加 `p95 > 1000ms` 直接判失败

这样做以后，常规一键脚本不再只是找“能跑过去的最高成功率档”，而是找：

**高吞吐同时，延迟也还能过线的可用容量。**

然后按这个新规则，在 `240` 分区上重新跑了一轮常规一键脚本。

最终记录结果是：

1. 单聊真实可用容量：约 `2372 msg/s`
2. 群聊真实可用容量：约 `4464 delivery/s`
3. 群聊折算消息数：约 `186 group_msg/s`

这里必须说明：

1. 这组是“常规一键脚本 + 单聊群聊一起跑 + `p95 < 1s`”口径
2. 它和上面的“单聊定点诊断脚本”不是一个口径

所以第二次优化最后，实际上沉淀出了两套都很重要的结果：

1. 纯代码链路优化最终把基线从 `350 / 4050 / 169` 提到了 `417 / 5462 / 228`
2. 最终工程推荐口径下，`240` 分区、`p95 < 1s` 的可用容量是 `2372 / 4464 / 186`

## 7. 第二次优化最后收下来的结论

第二次优化最后，真正可以稳定收下来的结论有 6 条：

1. `session_seq` 已经不再是当前主瓶颈。
2. `mysql_persist` 是第二次优化里最核心、最持续的主瓶颈之一，优化它以后群聊和单聊都能拿到收益。
3. `conversation_key + 复合索引` 这一步很关键，它把 `message` 大表的读写模型真正收敛了。
4. 同步 `status=sent` 更新会明显拖慢 websocket 主路径，改成异步批量更新后，单聊和群聊都继续上移。
5. 单聊后面真正剩下的大问题，不是 Kafka 本身，而是“热分区先打满 + `mysql_persist` 固定成本”。
6. 第二次优化最后，工程上最值得采用的默认值已经不是 `3` 分区，而是 `240` 分区这一档。

---

## 8. 第二次优化的一句话总结

第二次优化做的不是单点提速，而是把 Kafka 消费后半段整条链路从头到尾重新梳了一遍：先把 `session_seq` 从 MySQL 上拿下来，再把 `mysql_persist` 批量化、多 worker 化，再收敛 `message` 表索引和查询模型，最后把同步 `status=sent` 改成异步批量更新，并用分区扫描把单聊热分区问题拆开。最终，纯代码链路优化把容量从第一次优化的 `350 / 4050 / 169` 提到了 `417 / 5462 / 228`；如果按第二次优化最后推荐的工程口径 `240` 分区并要求 `p95 < 1s`，当前可以记录成 `2372 / 4464 / 186`。

---

## 9. 后续优化方向

第二次优化做完后，后面最值得继续做的事还有 4 条：

1. 继续拆 `mysql_persist` 固定写库成本，看是否要进一步异步化或旁路化。
2. 继续细分 Consumer 主链路指标，把“已拉取 / 已处理 / 已提交 / 已发给前端 / 已落库”拆得更细。
3. 群聊如果要继续吃分区红利，压测模型要改成“多群同时发”，而不是一直一个群。
4. 在当前 `240` 分区推荐值基础上，再补一轮和 5 节点 Kafka 集群完全同口径的正式压测，把部署口径和容量口径最终统一。
