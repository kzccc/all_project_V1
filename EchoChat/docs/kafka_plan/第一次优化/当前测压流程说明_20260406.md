# EchoChat 当前测压流程说明（精简版，2026-04-06 口径）

这份说明只保留当前最值得记住的内容，方便后面快速判断一份吞吐报告到底在测什么。

---

## 1. 这套测压在测什么

这套脚本不是打一档固定压力，而是在固定口径下自动升压，找出：

1. 单聊最大稳定吞吐
2. 群聊最大稳定吞吐

这里的“稳定”，在这份 `2026-04-06` 说明对应的旧口径下，主要是：

### 单聊

1. `delivery_success_rate >= 0.995`
2. `errors.json` 中错误数为 `0`

### 群聊

1. `delivery_coverage_rate >= 0.995`
2. `full_coverage_message_rate >= 0.99`
3. `errors.json` 中错误数为 `0`

注意：

1. **这份旧说明对应的 pass 条件当时还没有把延迟 SLA 纳入门槛**
2. 所以它找出来的是“成功送达口径下的最大稳定吞吐”
3. 不一定等于“业务可接受延迟口径下的最大吞吐”

## 3. 这套测压怎么加压

最容易混的地方只有一句话：

**它不是单纯靠“多开人”加压，而是在固定并发规模下，通过不断缩短发送间隔来加压。**

一次完整 run 里，下面这些是固定的：

### 单聊固定项

1. `SINGLE_PAIR_COUNT`
2. `SINGLE_MIN_DURATION_SEC`
3. `SINGLE_MAX_MESSAGES`

### 群聊固定项

1. `GROUP_MEMBER_LIMIT`
2. `GROUP_MIN_DURATION_SEC`
3. `GROUP_MAX_MESSAGES`

真正不断往上调的是：

1. `single target_rate`
2. `group target_rate`

随着 `target_rate` 提高，会出现两个连锁变化：

1. `send_interval_ms` 变小
2. `messages_per_sender` 变大

翻成更直白的话就是：

1. 每个发送者发消息更快了
2. 每个发送者这一档里发的总消息数也更多了

---

## 4. 单聊是怎么升压的

单聊每一档的核心关系是：

```text
send_interval_ms = round(SINGLE_PAIR_COUNT * 1000 / target_rate)
actual_offered_rate = SINGLE_PAIR_COUNT * 1000 / send_interval_ms
messages_per_sender = ceil(SINGLE_MIN_DURATION_SEC * 1000 / send_interval_ms)
```

含义很简单：

1. 单聊 pair 数固定
2. `target_rate` 越大，发送间隔越短
3. 间隔越短，在固定最小时长内，每个 sender 发的消息就越多

所以单聊不是“多开连接”来加压，而是：

1. 固定多少对人
2. 让每对人发得越来越快

---

## 5. 群聊是怎么升压的

群聊的核心思路和单聊一样，只是它控制的是 `delivery` 速率。

因为：

1. 一条群消息会被多个接收者收到
2. 所以群聊的 `target_rate` 本质上是按 `delivery/s` 控制，而不是按“群消息条数/s”控制

这意味着看群聊结果时一定要先分清：

1. `observed_delivery_per_sec`
2. 折算后的 `group_msg/s`

否则很容易把群聊能力高估很多倍。

## 6. 最常用的三套口径

### 6.1 基线口径

适合：

1. 日常回归
2. 版本间最常规对比

典型参数：

1. `SINGLE_PAIR_COUNT = 30`
2. `GROUP_MEMBER_LIMIT = 25`
3. `SINGLE_MIN_DURATION_SEC = 8`
4. `GROUP_MIN_DURATION_SEC = 8`

### 6.2 高压口径

适合：

1. 看更高并发规模下的上限
2. 看优化在更高连接数下是否仍然有效

典型参数：

1. `SINGLE_PAIR_COUNT = 60`
2. `GROUP_MEMBER_LIMIT = 40`
3. 初始 `target` 更高

### 6.3 严格长时口径

适合：

1. 查尾部消息问题
2. 查稳定性问题
3. 查“只差最后一点成功率”的问题

典型特征：

1. 测试时长更长
2. 成功率阈值更严格

## 7. 看报告时只要抓住这几个字段

每次看报告时，先看这几个字段：

1. `target`
2. `offered`
3. `observed`
4. `success / coverage`
5. `p95`
6. `pass`

可以这样理解：

1. `target`：想打到的目标速率
2. `offered`：实际灌进系统的速率
3. `observed`：系统真正处理出来的速率
4. `pass`：这一档是否满足当前口径定义的“稳定”

所以“最大稳定吞吐”本质上就是：

1. 在所有 `pass = true` 的档位里
2. 找 `observed` 最高的那一档
