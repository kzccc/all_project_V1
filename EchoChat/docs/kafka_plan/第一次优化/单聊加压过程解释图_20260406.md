# 单聊加压过程解释图（为什么 p95 会涨到 51 秒）

这份图解专门解释当前单聊压测里最容易看懵的那一档：

`readyfix_smoke -> single -> step_007_refine_target_2880`

对应结果来自：

`/workspace/czk/Personal/EchoChat/docs/k6_message_test/records/throughput_capacity_kafka_20260406_001343_readyfix_smoke/single/step_007_refine_target_2880/summary.json`

---

## 1. 先说结论

这一档不是 Kafka broker 发不出去。

这一档的真实含义是：

1. 压测端在很短时间内，持续往系统里塞了非常多单聊消息
2. 发送速度已经远远超过当前单聊消费链路的真实处理能力
3. 消息没有丢，但大量排队
4. 一排队，`p95` 就会从百毫秒级迅速膨胀到几十秒

所以当前问题可以直接理解成：

**单聊链路在高压下的“消费+业务处理+回写”整体处理速度，还不够快。**

---

## 2. `2880` 这一档到底是什么意思

这里的 `2880` 是本 stage 设定的目标发送速率：

`target_rate = 2880 msg/s`

但因为当前脚本单聊是 `30` 对用户并发发送，最终真正打出去的速率要按发送间隔折算。

当前这档的关键参数是：

1. `single_pair_count = 30`
2. `target_rate = 2880 msg/s`
3. runner 算出 `send_interval_ms = 10`
4. 所以实际发送速率约等于：

`30 * 1000 / 10 = 3000 msg/s`

也就是报告里看到的：

`offered = 3000.0 msg/s`

而这次每个 sender 会连续发：

`messages_per_sender = 800`

总消息数就是：

`30 * 800 = 24000`

所以这一档完整翻译成人话就是：

**30 对单聊连接，同时每 10ms 发一条，连续发 800 轮，总共往系统里压 24000 条消息。**

---

## 3. 当前这档测出来的真实结果

这一档最终 summary 是：

1. `expected_messages = 24000`
2. `received_messages = 24000`
3. `delivery_success_rate = 1.0`
4. `duration_sec = 68.356`
5. `observed_throughput = 351.105 msg/s`
6. `latency.p50 = 20660.5 ms`
7. `latency.p95 = 51782.2 ms`
8. `latency.p99 = 56509.06 ms`

这几个数要这样理解：

1. `24000 / 24000 = 1.0`
   说明消息最后都送达了
2. `observed = 351.105 msg/s`
   说明系统整条单聊链路，最终每秒真正处理完的大约只有 `351` 条
3. `duration = 68.356s`
   说明这轮测试大部分时间都不是“正在发”，而是“前面已经堆积的消息还在慢慢消化”
4. `p95 = 51782.2ms`
   说明 95% 的消息端到端延迟都已经到了 `51.8s` 以内，这已经是明显积压态

---

## 4. 一张最直观的过程图

```text
发送端持续灌入
3000 msg/s
    |
    v
+------------------+
| Kafka Topic      |
| + 消费等待队列   |
+------------------+
    |
    | 当前单聊链路只能稳定处理
    | 约 351 msg/s
    v
+------------------+
| Consumer         |
| 业务处理/落库/   |
| 回写/投递        |
+------------------+
    |
    v
接收端最终收到
351 msg/s 左右
```

如果把这轮压测换成“水管”来理解：

```text
进水速度：3000 msg/s
出水速度： 351 msg/s

每秒净积压：
3000 - 351 = 2649 msg
```

也就是说，只要发送还在继续，队列就会越堆越高。

---

## 5. 为什么 p95 会突然暴涨

这里最关键的一点是：

**p95 变大，不代表消息坏了，而代表大部分消息都在排队。**

这轮发送大约持续 `8s`，但系统处理完全部 `24000` 条消息一共花了 `68.356s`。

所以过程大致是这样的：

```text
0s ---------------- 8s ------------------------------ 68s
|      持续高速发送      |        发送结束，系统继续清积压        |
```

在前 `8s` 里：

1. 压测端持续按 `3000 msg/s` 往里打
2. 系统只能按 `351 msg/s` 左右往外处理
3. 大量消息开始在 Kafka 消费后链路里排队

发送停掉以后：

1. 新消息不再进入
2. 但前面积压的几万条消息还没处理完
3. 系统继续花几十秒把队列清掉

于是消息延迟会分成两类：

1. 前面少量消息，几乎不排队，延迟较低
2. 后面大部分消息，要先等前面积压被处理，延迟就被拉到几十秒

这就是为什么：

1. `success = 1.0`
2. 但 `p50 = 20.66s`
3. `p95 = 51.78s`

也就是说，虽然最后都送达了，但已经不是业务可接受的状态。

---

## 6. 为什么之前是 167ms，现在变成 51 秒

这两个数不是同一压力下测出来的。

`167.55 ms` 来自更早的 `batch_commit_baseline` 第一档，那个阶段压力很低，单聊观测吞吐只有：

`13.343 msg/s`

当时系统远没被打满，所以：

1. 消息基本不用排队
2. commit 优化一做完，延迟就直接掉到百毫秒级

而现在这档是：

1. `offered = 3000 msg/s`
2. 实际只能处理 `351 msg/s`

这不是“优化失效了”，而是：

**我们把系统继续往上压，终于把真正的极限压出来了。**

所以正确理解应该是：

1. `33.6s -> 167ms`
   说明我们先解决了之前那个明显错误的瓶颈，尤其是 commit 过频阻塞
2. `167ms -> 51.8s`
   说明在更高压力下，新的真实瓶颈暴露了，系统进入了严重排队态

## 7. 从各档位看，单聊是怎么一步步进入积压的

来自 `readyfix_smoke/report.md` 的单聊各档结果：

| 档位 | offered | observed | success | p95 |
| --- | ---: | ---: | ---: | ---: |
| 120 | 120.0 | 100.041 | 1.0 | 108.15 ms |
| 240 | 240.0 | 199.672 | 1.0 | 96.0 ms |
| 480 | 483.9 | 322.373 | 1.0 | 2299.05 ms |
| 960 | 967.7 | 350.301 | 1.0 | 11086.65 ms |
| 1920 | 1875.0 | 346.164 | 1.0 | 30355.25 ms |
| 2880 | 3000.0 | 351.105 | 1.0 | 51782.2 ms |
| 3360 | 3333.3 | 351.482 | 0.989914 | 58486.0 ms |

这个表最值得注意的不是成功率，而是这两列：

1. `offered`
2. `observed`

能看到一个非常明显的现象：

1. offered 一直在涨
2. observed 到了 `~350 msg/s` 左右后几乎不再上涨

这说明单聊链路当前的真实处理上限，大约就卡在这里。

超过这个上限以后，新增压力不会换来更高吞吐，只会换来：

1. 更长的队列
2. 更差的 p95 / p99
3. 最终成功率开始掉

---

## 8. 这张图最终想说明什么

如果只看当前压测结果，单聊现在已经不是“会固定丢起步消息”的问题了。

那个问题已经被 readiness 修掉了。

现在真正的问题是：

**单聊消费侧整条业务链路，在高压下只能稳定跑到约 `350 msg/s`，再往上就会明显积压，进而把 p95 拉到几十秒。**

所以后面优化方向不是继续盯着“消息有没有开始消费”，而是要继续拆：

1. consumer 处理并发度是否够
2. 单聊热路径里有没有串行点
3. DB / Redis / WebSocket 回写有没有新的饱和点
4. 哪一段把总处理能力卡在了 `~350 msg/s`

---

## 9. 一句最适合直接对外解释的话

当前这轮单聊压测测出来的，不是 Kafka 发不动，而是系统端到端单聊处理链路在高压下的真实容量大约只有 `350 msg/s`。继续加压时，消息虽然最终还能送达，但因为处理速度跟不上发送速度，队列持续积压，所以 `p95` 会从百毫秒迅速涨到几十秒。
