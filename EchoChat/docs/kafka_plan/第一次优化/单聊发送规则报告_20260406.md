# 单聊发送规则报告（截至 2026-04-06）

这份文档只解释一件事：

**当前单聊吞吐压测里，一档压力到底是怎么定义、怎么计算、怎么落到最终结果上的。**

对应脚本：

- [run_max_throughput.sh](/workspace/czk/Personal/EchoChat/docs/k6_message_test/scripts/run_max_throughput.sh)
- [throughput_capacity_runner.py](/workspace/czk/Personal/EchoChat/docs/k6_message_test/scripts/throughput_capacity_runner.py)
- [message_latency_runner.py](/workspace/czk/Personal/EchoChat/docs/k6_message_test/scripts/message_latency_runner.py)

---

## 1. 单聊压测到底在控制什么

当前单聊压测不是直接写死：

1. 每个人发多少条
2. 每秒固定发多少条

而是先给一个目标压力档位：

`target`

然后脚本再根据当前这轮固定参数，自动算出：

1. 发送间隔 `send_interval_ms`
2. 每个 sender 要发多少条 `messages_per_sender`
3. 这一档总共要发多少条 `expected_messages`
4. 实际打出去的发送速率 `offered`

所以单聊的真实规则是：

**先固定发送人数，再通过缩短发送间隔来逐档升压。**

---

## 2. 单聊一轮压测里，哪些参数是固定的

以当前常用基线口径为例：

```bash
LABEL=baseline \
MODE=kafka \
SINGLE_PAIR_COUNT=30 \
SINGLE_MIN_DURATION_SEC=8 \
bash /workspace/czk/Personal/EchoChat/docs/k6_message_test/scripts/run_max_throughput.sh
```

这意味着这一轮单聊压测固定的是：

1. `SINGLE_PAIR_COUNT = 30`
2. `SINGLE_MIN_DURATION_SEC = 8`

它们的意思是：

1. 一共有 `30` 对单聊用户同时在发
2. 每一档至少持续 `8` 秒

这里的 `30` 对，等价于：

1. `30` 个 sender
2. `30` 个 receiver

每一对 sender/receiver 互相独立。

## 3. 单聊每一档真正变化的是什么

每一档真正变化的是：

`target_rate`

也就是报告里的：

`target`

脚本不会直接按 `target` 发消息，而是先把它换算成发送间隔。

对应代码在：

[throughput_capacity_runner.py#L304](/workspace/czk/Personal/EchoChat/docs/k6_message_test/scripts/throughput_capacity_runner.py#L304)

公式是：

`send_interval_ms = round(SINGLE_PAIR_COUNT * 1000 / target_rate)`

然后再算：

`actual_offered_rate = SINGLE_PAIR_COUNT * 1000 / send_interval_ms`

`messages_per_sender = ceil(SINGLE_MIN_DURATION_SEC * 1000 / send_interval_ms)`

---

## 4. 这些公式分别是什么意思

### 4.1 发送间隔

`send_interval_ms`

意思是：

**每个 sender 两条消息之间间隔多少毫秒。**

`target` 越大，`send_interval_ms` 就越小。

也就是：

1. 压力越高
2. 每个 sender 发得越快

### 4.2 实际发送速率

`actual_offered_rate`

意思是：

**按照整数毫秒间隔折算后，这一档真正打出去的消息速率。**

因为脚本最终只能按整数毫秒发送，所以：

1. `target` 是目标值
2. `offered` 是折算后的真实值

这也是为什么报告里经常会出现：

1. `target = 2880`
2. `offered = 3000`

### 4.3 每个 sender 发多少条

`messages_per_sender`

意思是：

**为了保证这一档至少持续 `SINGLE_MIN_DURATION_SEC` 秒，每个 sender 需要发多少条消息。**

所以：

1. 间隔越短
2. 在固定 `8s` 内
3. 每个 sender 就必须发更多消息

### 4.4 总消息数

`expected_messages = SINGLE_PAIR_COUNT * messages_per_sender`

意思是：

**这一档总共向系统施加多少条单聊消息。**

## 5. 用 `target=2880` 这一档做完整例子

这一档来自：

[step_007_refine_target_2880/summary.json](/workspace/czk/Personal/EchoChat/docs/k6_message_test/records/throughput_capacity_kafka_20260406_001343_readyfix_smoke/single/step_007_refine_target_2880/summary.json)

当前固定参数是：

1. `SINGLE_PAIR_COUNT = 30`
2. `SINGLE_MIN_DURATION_SEC = 8`

### 第一步：算发送间隔

`send_interval_ms = round(30 * 1000 / 2880)`

`= round(30000 / 2880)`

`= round(10.416...)`

`= 10 ms`

这一步的意思是：

**30 个 sender，每个人每隔 `10ms` 发 1 条。**

### 第二步：算真实发送速率

`offered = 30 * 1000 / 10`

`= 3000 msg/s`

所以报告里这档显示的是：

1. `target = 2880`
2. `offered = 3000`

### 第三步：算每个 sender 要发多少条

`messages_per_sender = ceil(8 * 1000 / 10)`

`= ceil(800)`

`= 800`

也就是：

**每个 sender 连续发 `800` 条。**

### 第四步：算总消息数

`expected_messages = 30 * 800`

`= 24000`

所以这档的完整含义就是：

**30 对单聊连接，每个 sender 每隔 `10ms` 发 1 条，连续发 `800` 条，总共往系统里压 `24000` 条消息。**

---

## 6. 这档最终结果又是怎么来的

这一档 summary 的关键结果是：

1. `expected_messages = 24000`
2. `received_messages = 24000`
3. `delivery_success_rate = 1.0`
4. `duration_sec = 68.356`
5. `observed_throughput_msg_per_sec = 351.105`
6. `latency.p95_ms = 51782.2`

这些字段的计算方式是：

### 6.1 成功率

`delivery_success_rate = received_messages / expected_messages`

也就是：

`24000 / 24000 = 1.0`

### 6.2 实际处理吞吐

`observed_throughput_msg_per_sec = received_messages / duration_sec`

也就是：

`24000 / 68.356 = 351.105`

这个数才是系统真实处理能力的体现。

### 6.3 延迟

`p95`

表示：

**95% 的消息端到端延迟不超过多少毫秒。**

当前这档是：

`51782.2 ms`

也就是大约 `51.8s`

---

## 7. 为什么 `target=2880`，最后却只有 `351 msg/s`

因为：

1. `target` 是你施加给系统的目标压力
2. `offered` 是压测端真实打出去的速率
3. `observed` 才是系统真正消化完的速率

所以这档真正发生的是：

1. 压测端按 `3000 msg/s` 往系统里灌
2. 系统最终只能按 `351 msg/s` 左右处理完
3. 多出来的消息全部在队列里排队
4. 所以虽然最后都送达了，但 `p95` 被拉到 `51.8s`

一句话说就是：

**`2880` 不是系统容量，`351` 才是这轮压测下测出来的单聊真实处理能力。**

## 8. 单聊所有档位都按同样规则计算

以当前基线参数 `30 对 / 8 秒` 为例，各档大致如下：

| target | send_interval_ms | offered | messages_per_sender | expected_messages |
| ---: | ---: | ---: | ---: | ---: |
| 120 | 250 | 120.0 | 32 | 960 |
| 240 | 125 | 240.0 | 64 | 1920 |
| 480 | 62 | 483.9 | 130 | 3900 |
| 960 | 31 | 967.7 | 259 | 7770 |
| 1920 | 16 | 1875.0 | 500 | 15000 |
| 2880 | 10 | 3000.0 | 800 | 24000 |
| 3360 | 9 | 3333.3 | 889 | 26670 |

从这张表能直接看到：

1. `target` 越高
2. `send_interval_ms` 越小
3. `offered` 越高
4. `messages_per_sender` 越多
5. `expected_messages` 也越多

所以单聊的加压，不是“多开人”，而是：

**在固定 `30` 对连接下，让每个 sender 发得越来越快，而且每档总消息数也越来越大。**

## 9. `2880` 这一档为什么会被选出来

当前自动升压逻辑是：

1. 从初始档开始
2. 通过就翻倍
3. 失败后在“最后一个通过档”和“第一个失败档”之间做二分 refine

对应代码在：

[throughput_capacity_runner.py#L444](/workspace/czk/Personal/EchoChat/docs/k6_message_test/scripts/throughput_capacity_runner.py#L444)

这次单聊路径大致是：

1. `120` 通过
2. `240` 通过
3. `480` 通过
4. `960` 通过
5. `1920` 通过
6. `3840` 失败
7. refine 取中间值 `2880`
8. `2880` 通过
9. 再 refine 到 `3360`
10. `3360` 失败

所以 `2880` 的意义是：

**它是当前成功和失败边界之间，一个被脚本自动试出来的中间档位。**

---

## 10. 最终怎么读一档单聊结果

以后你看到任意一档单聊结果，都可以按这个顺序理解：

1. `target`
   想把压力抬到哪个目标档位
2. `send_interval_ms`
   每个 sender 实际每隔多少毫秒发 1 条
3. `offered`
   压测端真实打出去的速率
4. `messages_per_sender`
   每个 sender 发多少条
5. `expected_messages`
   这一档总共施加多少条消息
6. `success`
   最后有多少真的送达
7. `observed`
   系统最终每秒真正处理完多少条
8. `p95`
   在这档压力下，尾延迟已经坏到什么程度

---

## 11. 一句总结

当前单聊压测的发送规则，本质上是：

**固定 sender 数量和最短压测时长，按 `target` 计算发送间隔，再据此算出每个 sender 要发多少条消息；随着 `target` 升高，发送间隔越来越短、每人发的消息越来越多、总施压消息量也越来越大。**
