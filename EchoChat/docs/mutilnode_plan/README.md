# EchoChat 多节点改造与微服务演进计划

## 1. 文档目标

本文档结合当前 `EchoChat` 的实际代码结构，给出一版可执行的演进计划，目标分为两步：

1. 先把当前项目从“单进程可运行”提升到“多节点可部署、可扩容、可观测”的分布式单体。
2. 在第一阶段稳定之后，再逐步演进为微服务架构，而不是一开始就硬拆服务。

这份计划的核心原则不是“拆得快”，而是“先把链路跑稳，再拆边界”。

## 2. 当前项目现状

从当前代码结构看，项目已经具备部分分布式组件，但核心业务链路仍然偏单体实现：

1. 服务启动入口仍是一个统一进程，由 `cmd/echo_chat_server/main.go` 负责启动 HTTP、WebSocket、消息服务。
2. 消息链路支持两种模式：
3. `channel` 模式：消息通过本地内存 channel 转发。
4. `kafka` 模式：消息先写 Kafka，再由服务内部消费。
5. WebSocket 在线连接保存在进程内存 `Clients map[string]*Client` 中，`channel` 和 `kafka` 两种模式下都是本地内存管理。
6. MySQL 承担用户、联系人、会话、消息等主要业务数据存储。
7. Redis 主要承担 token、消息顺序号、部分消息列表和会话列表缓存。
8. 文件与头像当前通过本地静态目录提供访问，路径由 Gin 的 `Static` 路由直接暴露。

### 2.1 当前具备的可复用能力

这部分能力在后续多节点改造中可以继续保留，不需要推倒重来：

1. JWT + Redis 的会话与 refresh token 机制已经具备无状态服务基础。
2. Kafka 已经进入消息主链路，说明异步解耦的基础设施已经存在。
3. 消息表已经有 `session_seq` 字段，项目也已经在尝试保证会话内顺序。
4. HTTP、WS、Kafka、MySQL、Redis 的基础接入已经打通。
5. 项目已经有一定的压测与观测基础，`docs/t_K6` 和 `internal/observability` 可继续扩展。

### 2.2 当前真正限制多节点的地方

当前系统还不能直接认为是“真正分布式”，主要限制有以下几类：

1. 在线连接状态仅保存在本地进程内存。
2. Kafka 只负责消息入队，但消息最终仍要靠当前节点本地 `Clients` 找连接并推送。
3. 文件与头像是本地磁盘存储，多节点部署后天然会出现文件不一致。
4. 一部分 Redis 用法依赖模糊查询和 `KEYS`，节点规模和缓存规模增大后容易成为风险点。
5. 消息顺序逻辑虽然已存在，但 Kafka 分区键与真正的“聊天流”边界还没有完全统一。

## 3. 为什么当前阶段不建议先拆微服务

对于即时通讯项目，最复杂的部分通常不是用户、群组、联系人这些 CRUD 接口，而是下面几件事：

1. WebSocket 长连接管理。
2. 在线状态同步。
3. 消息顺序保证。
4. 多节点投递路由。
5. 离线消息与历史消息查询。
6. 文件上传、下载与消息引用的一致性。

如果这些基础问题还没有在多节点场景下跑稳，就直接拆成多个微服务，会出现几个非常现实的问题：

1. 服务数量增加了，但核心瓶颈没有解决。
2. 原本进程内调用变成跨服务调用，链路更长，问题更难排查。
3. 数据边界还没稳定时硬拆服务，后续会频繁回滚或二次合并。
4. 对当前项目体量来说，运维成本和开发成本会上升得很快。

因此，这个项目更适合采用下面的演进策略：

1. 先做“分布式单体”。
2. 再做“连接层与消息处理层分离”。
3. 最后再根据稳定的业务边界演进到微服务。

## 4. 第一阶段总体目标：先做分布式单体

这一阶段不拆服务，不改成多个独立业务系统，而是在保留现有代码组织方式的基础上，把当前服务改造成可以多实例部署的形态。

### 4.1 第一阶段目标

1. 支持同一套服务多节点部署。
2. 任意节点都能接入 HTTP 和 WebSocket 流量。
3. 消息能跨节点正确投递，不依赖单节点本地内存。
4. 文件和头像不再依赖单机磁盘。
5. 具备基本的节点观测能力、故障排查能力和压测基线。

### 4.2 第一阶段完成后的系统定位

完成后，项目仍然是一个统一仓库、统一部署单元、统一业务服务，但具备如下能力：

1. 可横向扩容。
2. 节点可替换。
3. 节点宕机后可恢复。
4. 核心消息链路不再被单机状态绑定。

## 5. 第一阶段核心改造思路

## 5.1 先统一架构目标，而不是先改代码细节

第一阶段建议把系统抽象成四个层次，但它们仍然运行在同一个项目里：

1. 接入层：HTTP + WebSocket。
2. 状态层：Redis 中的在线状态、路由状态、缓存状态。
3. 消息层：Kafka 负责消息入队和异步消费。
4. 存储层：MySQL + 对象存储。

这里的关键不是代码文件拆不拆，而是把“状态应该放哪儿”先理顺。

## 5.2 优先把本地状态外置

当前项目最强的单点特征是 `Clients map[string]*Client`。这部分如果不外置，节点再多也只是“多个彼此隔离的单机实时服务”。

因此第一优先级应该是：

1. 把“用户在线在哪个节点”从本地内存视角提升为全局状态。
2. 把“消息应该由哪个节点投递”变成显式路由。
3. 把“文件在哪儿”从本地路径改成共享存储。

## 6. 第一阶段详细规划

## 6.1 阶段 A：部署形态调整

### 目标

先让当前项目具备基础的多节点部署条件。

### 工作项

1. 在部署层引入统一入口，例如 Nginx 或云负载均衡，前面承接 HTTP 和 WS 流量。
2. `echo_chat_server` 支持多副本部署，实例通过配置区分 `node_id`。
3. 每个节点启动时上报自身标识、启动时间、健康状态。
4. 统一配置加载方式，区分本地开发配置和多节点部署配置。

### 本阶段结果

1. 同一个版本的服务可以启动多个实例。
2. 基础接口可以通过负载均衡访问。
3. WebSocket 可以分散建立在不同节点上。

## 6.2 阶段 B：在线状态与节点路由

### 目标

解决“连接在 A 节点，消息被 B 节点消费后不知道怎么投递”的问题。

### 工作项

1. 为每个服务节点增加全局唯一 `node_id`。
2. 用户建立 WS 连接后，在 Redis 记录：
3. `online_user:{user_id} -> node_id`
4. `node_conn:{node_id}:{user_id} -> connection meta`
5. 连接保持心跳续约，断开时主动删除，异常下线依赖 TTL 清理。
6. 群发时不再简单依赖本地 `Clients` 遍历，而是先获取在线成员对应节点，再决定投递目标。
7. 明确一个用户是否允许多端在线。

### 建议策略

当前项目如果要降低复杂度，第一阶段建议先采用“单用户单连接”策略：

1. 一个用户同一时间只保留一个主连接。
2. 新连接建立时覆盖旧连接。
3. 这样可以明显降低在线路由、消息回显和状态同步难度。

后续如果再做多端在线，可以再扩展为：

1. `user_id -> connection list`
2. `connection_id -> node_id`

### 本阶段结果

1. 系统知道每个在线用户当前在哪个节点。
2. 在线状态不再依赖某个单独进程内存。
3. 节点宕机后，在线状态可以靠 TTL 和重连逐步恢复。

## 6.3 阶段 C：消息投递链路改造

### 目标

把“消息消费”和“消息投递”彻底从单节点假设中解耦出来。

### 当前问题

虽然当前项目的 `kafka` 模式已经能把消息写入 Kafka，但消费完成后仍然会直接访问本节点内存 `Clients` 进行推送。这说明 Kafka 目前只做到了入队，没有完成真正的跨节点投递闭环。

### 工作项

1. 保留 Kafka 作为消息入口。
2. 消费消息后，先确定消息接收方对应的目标节点。
3. 如果目标用户在线在当前节点，则直接本地投递。
4. 如果目标用户在线在其他节点，则通过“节点间投递事件”发送给目标节点。
5. 节点间投递事件可以采用两种方式：
6. 方案一：Redis Pub/Sub
7. 方案二：Kafka 新增 delivery topic

### 建议

第一阶段建议优先选择 Redis Pub/Sub 或 Redis Stream 作为“节点间投递事件”方案，原因如下：

1. 当前项目已经接入 Redis。
2. 节点间投递消息量通常比业务消息总量小，先用 Redis 实现成本更低。
3. 后续如果投递事件复杂度变高，再迁移到 Kafka 专用 topic 更平滑。

### 本阶段结果

1. 任意节点消费到消息后，都能把消息发给真正持有连接的节点。
2. 多节点场景下，消息投递逻辑形成闭环。

## 6.4 阶段 D：统一消息顺序语义

### 目标

确保消息顺序的定义与 Kafka 分区策略、数据库查询策略一致。

### 当前现状

项目当前已经有基于聊天流的 `session_seq` 递增逻辑：

1. 单聊使用排序后的双方用户 ID 作为顺序范围。
2. 群聊使用群 ID 作为顺序范围。

这套思路是对的，应该继续保留并正式化。

### 建议改造

引入明确的 `conversation_id` 概念：

1. 单聊：`conversation_id = min(userA, userB) + ":" + max(userA, userB)`
2. 群聊：`conversation_id = group_id`

后续统一用于：

1. Kafka 分区 key
2. 消息顺序号生成 scope
3. 消息历史查询条件
4. 缓存 key
5. 监控指标维度

### 为什么这一点重要

如果 Kafka key、数据库排序字段、缓存作用域、消息顺序号不是同一套语义，后期节点越多，顺序问题越难定位。

### 本阶段结果

1. 会话内消息顺序定义统一。
2. Kafka 分流与数据库查询方向一致。
3. 后续分页、补拉、回放更容易做。

## 6.5 阶段 E：文件与头像改造

### 目标

解决多节点部署后本地磁盘资源不一致的问题。

### 当前问题

头像和文件当前依赖本地静态目录，多实例下会出现：

1. A 节点上传文件，B 节点访问不到。
2. 扩容和滚动发布时文件一致性无法保证。
3. 备份和迁移成本高。

### 工作项

1. 引入对象存储，建议使用 MinIO 或 S3 兼容服务。
2. 上传接口改为：
3. 文件上传到对象存储。
4. 数据库或消息中保存对象地址或对象 key。
5. 静态资源访问不再依赖 Gin 本地目录，而是走对象存储访问地址或专门下载代理。

### 本阶段结果

1. 文件与头像成为共享资源。
2. 节点无状态程度提高。
3. 服务部署与资源存储分离。

## 6.6 阶段 F：缓存治理与 Redis key 规范化

### 目标

避免多节点下 Redis 成为隐性瓶颈。

### 当前问题

当前项目有一部分 key 查询和删除依赖 `KEYS` + 模糊匹配。这种方案在数据量小的时候简单，但在多节点和高并发场景下风险较高。

### 工作项

1. 重新梳理 Redis key 设计，区分：
2. 在线状态 key
3. 会话缓存 key
4. 消息列表缓存 key
5. token key
6. 顺序号 key
7. 删除和查询策略尽量改成确定性 key。
8. 对必须遍历的场景改成 `SCAN` 或集合索引方案。
9. 给高价值缓存加统一 TTL 和失效策略。

### 本阶段结果

1. Redis 访问模型更稳定。
2. 缓存设计不再依赖模糊查询。
3. 后续扩容时更可控。

## 6.7 阶段 G：观测、压测与验收

### 目标

给多节点改造建立明确的验收标准，而不是“能跑就算完成”。

### 工作项

1. 扩展现有 metrics，增加：
2. 节点在线连接数
3. 节点间投递次数
4. 本地投递命中率
5. 跨节点投递耗时
6. Kafka 消费延迟
7. WS 重连次数
8. 建立多节点压测场景：
9. 单聊高频发送
10. 群聊广播
11. 节点宕机恢复
12. 文件上传与下载一致性

### 第一阶段验收标准

1. 多个节点同时运行时，用户可随机连入任意节点。
2. 发送方和接收方处于不同节点时，消息仍能实时到达。
3. 节点宕机后，用户可重连到其他节点继续使用。
4. 文件与头像不依赖本地节点。
5. 压测结果与单节点基线相比，整体吞吐和稳定性可接受。

## 7. 第一阶段建议里程碑

为了避免改造范围过大，建议拆成四个里程碑推进。

### 里程碑 M1：多节点可启动

1. 多实例部署完成。
2. `node_id` 接入完成。
3. 基础健康检查和配置管理完成。

### 里程碑 M2：在线状态全局化

1. Redis 在线状态模型落地。
2. WS 建连、断连、心跳与 TTL 完成。
3. 单用户单连接策略跑通。

### 里程碑 M3：跨节点消息投递打通

1. Kafka 消费后可跨节点投递。
2. 单聊消息链路在多节点下跑通。
3. 群聊在线成员投递逻辑打通。

### 里程碑 M4：资源与缓存治理完成

1. 对象存储接入完成。
2. Redis key 规范完成。
3. 多节点压测与验收完成。

## 8. 第一阶段暂时不做的事情

为了让项目稳定推进，下面这些事情建议先明确不做或延后做：

1. 不先拆成多个独立仓库。
2. 不先拆用户服务、群组服务、联系人服务。
3. 不先做复杂服务网格。
4. 不先做分库分表。
5. 不先支持非常复杂的多端在线同步策略。

这些能力未来都可以做，但不应该抢在“多节点链路跑稳”之前。

## 9. 第二阶段：从分布式单体演进到微服务

当第一阶段稳定后，系统的状态边界、消息边界、连接边界会更清晰，这时候再拆服务才有依据。

### 9.1 微服务演进的前提条件

至少满足下面几个条件，再进入微服务阶段更合适：

1. 多节点线上链路已经稳定。
2. 在线状态与节点路由设计已经成熟。
3. `conversation_id`、消息顺序、缓存策略已经稳定。
4. 对象存储已经替代本地文件。
5. 监控和压测基线已经形成。

### 9.2 微服务阶段的拆分方向

建议优先按“技术边界 + 运行特征”拆，而不是按 controller 名字硬拆。

#### 方向一：接入层独立

先拆 `ws-gateway`：

1. 专门负责 WebSocket 接入。
2. 维护连接、认证、在线状态、节点路由。
3. 不直接承担复杂业务落库。

这样做的原因是接入层的运行特征和普通业务接口差异最大，最值得先拆。

#### 方向二：消息处理层独立

再拆 `message-service`：

1. 负责消费消息。
2. 负责消息落库。
3. 负责顺序号生成。
4. 负责离线消息和历史消息查询。

这一步拆完后，IM 核心链路会更清楚：

1. `ws-gateway` 负责接入与投递。
2. `message-service` 负责持久化与消息处理。

#### 方向三：用户与关系域拆分

后续可逐步拆出：

1. `auth-service`
2. `user-service`
3. `contact-service`
4. `group-service`
5. `session-service`
6. `media-service`
7. `admin-service`

这里不要求一次拆完，可以按业务增长情况逐步推进。

### 9.3 推荐的微服务拆分顺序

建议顺序如下：

1. `ws-gateway`
2. `message-service`
3. `media-service`
4. `auth-service`
5. `user/contact/group/session` 领域服务
6. `admin-service`

原因很简单：

1. 前三者运行特征最独立，收益最大。
2. `auth` 独立后更利于统一鉴权。
3. 用户、联系人、群组、会话这些业务服务拆分收益没有实时链路那么高，可以后移。

## 10. 微服务阶段的目标架构方向

未来可以逐步收敛为下面这种结构：

1. `api-gateway` 或 `bff`
2. `ws-gateway`
3. `message-service`
4. `auth-service`
5. `user-service`
6. `contact-service`
7. `group-service`
8. `session-service`
9. `media-service`
10. `admin-service`
11. `mysql`
12. `redis`
13. `kafka`
14. `object storage`

其中：

1. `ws-gateway` 负责长连接与在线投递。
2. `message-service` 负责消息处理、落库、顺序和补拉。
3. 其他服务以业务查询和管理为主。

## 11. 风险与注意事项

### 11.1 第一阶段的主要风险

1. 在线状态与节点路由设计不清，会导致跨节点消息不稳定。
2. 群聊在线成员投递在成员多时压力较大，需要评估 Redis 查询与投递风暴。
3. 文件迁移到对象存储后，前端资源访问路径需要统一梳理。
4. 多节点后日志量和指标量会上升，观测能力要同步补齐。

### 11.2 微服务阶段的主要风险

1. 服务拆太细，导致链路复杂度超过项目当前收益。
2. 数据边界未稳定就拆服务，后期会频繁返工。
3. 团队如果没有现成的运维和治理能力，微服务会显著提高维护成本。

## 12. 推荐落地策略总结

如果以当前仓库为基础，推荐的落地顺序可以浓缩成一句话：

先把当前 `EchoChat` 做成“多节点可运行、跨节点可投递、文件共享、状态外置”的分布式单体，再基于已经稳定的连接层、消息层和业务域边界，逐步演进为微服务。

对应到实际推进上，建议严格按下面顺序执行：

1. 多节点部署能力。
2. 在线状态与节点路由。
3. 跨节点消息投递。
4. `conversation_id` 与顺序语义统一。
5. 对象存储替换本地文件。
6. Redis key 治理与缓存规范。
7. 观测、压测、验收。
8. `ws-gateway` 与 `message-service` 优先拆分。
9. 再考虑其他业务微服务。

## 13. 文档结论

对 `EchoChat` 来说，最合适的优化路径不是“直接改成微服务”，而是：

1. 先解决多节点实时链路问题。
2. 先把本地状态迁出去。
3. 先形成稳定的消息边界和顺序边界。
4. 再把已经成熟的部分拆成服务。

这样做的好处是：

1. 改造路径更稳。
2. 风险更可控。
3. 每个阶段都有明确收益。
4. 后续做简历、面试、项目答辩时，架构演进逻辑也更完整。
