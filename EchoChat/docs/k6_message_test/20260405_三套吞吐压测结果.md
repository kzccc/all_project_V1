## 汇总结论

当前版本三套口径都没有通过第一档，`summary.json` 中 `single` 和 `group` 都是 `null`，说明脚本没有找到“满足稳定性阈值”的档位。

核心现象不是 Kafka 生产端扛不住，而是消费侧仍然存在尾部消息没有完成消费或确认的问题：

1. 单聊三套都出现 `producer_success > consumer_pulled`
2. 群聊三套都出现 `producer_success > consumer_pulled`
3. `consumer_lag = 0`，说明不是 Kafka backlog 堆住，更像是测试窗口结束时仍有一小段消息没有进入最终可见结果
4. 消费处理阶段本身仍然很快，瓶颈主要不在 `deserialize/mysql/redis/ws` 业务处理
5. offset commit 虽然已经批量化，但单聊平均 commit 仍然偏慢

## 对比前一版已知较优结果

参考：

`/workspace/czk/Personal/EchoChat/docs/k6_message_test/records/throughput_capacity_kafka_20260405_155921_batch_commit_baseline`

这版已知较优结果里：

1. 单聊成功率是 `0.96875`
2. 单聊 p95 是 `167.55 ms`
3. 群聊覆盖率是 `0.95082`
4. 群聊 p95 是 `141 ms`

和这次三套结果相比，当前 workspace 下的单聊稳定性明显波动更大，尤其：

1. `baseline_current` 单聊成功率掉到 `0.9375`
2. `baseline_current` 单聊 p95 拉高到 `9478.15 ms`
3. `highload_current` 单聊成功率接近阈值，但 p95 仍然高到 `30207 ms`

这说明当前工作区代码相对 `batch_commit_baseline` 那次运行，单聊路径上又出现了新的波动或回退。

## 建议你后续固定的三套命令

后面每次做完一版 Kafka 优化，直接复跑下面三套：

1. 基线对比：看最常用口径是否提升
2. 高压对比：看更大连接规模下吞吐有没有提升
3. 严格长时对比：看尾部消息丢失和稳定性有没有改善

如果只想先做日常回归，优先跑前两套；如果某次优化涉及 consumer commit、停机收尾、消息确认链路，就把第三套也一起跑。

## 当前最值得继续查的点

1. `ConsumeClaim` 结束前是否还有已处理但未最终记账的消息
2. benchmark 在发完消息后是否过早结束，导致尾部消费来不及进入结果统计
3. 单聊路径里 websocket 回执或消息确认链路是否存在尾部 race
4. 群聊仍然存在固定比例的少量尾差，说明还缺一次最终 drain/flush
