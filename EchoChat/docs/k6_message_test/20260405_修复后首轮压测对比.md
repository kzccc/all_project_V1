## 这轮修复实际带来的变化

这轮修复的收益非常明确，但收益主要体现在“延迟”和“Kafka 分区利用率”上，还没有解决“尾部固定缺几条消息”的问题。

### 已经明显改善的部分

1. 单聊 p95 延迟大幅下降
2. 群聊 p95 延迟大幅下降
3. offset commit 次数显著减少，批量更大
4. 单聊已经不再是单分区，确认三分区都在工作

### 仍未解决的部分

1. 三套口径仍然都没有通过第一档
2. 单聊成功率没有根本提升
3. 群聊覆盖率几乎没有变化
4. 尾部消息缺失仍然存在，而且缺失模式很稳定

## 最关键的判断

### 1. commit 阻塞问题已经被明显压下去了

从结果上看，这一步是有效的：

1. 单聊 p95 从秒级降到了百毫秒级
2. 群聊 p95 也被压到了 `6 ms`
3. commit 次数从几十次降到个位数或十几次
4. commit batch 从约 `2~30` 提高到 `58~92`

这说明前面判断是对的：原先的主要延迟放大器，确实是消费主线程里的 commit 频繁阻塞。

### 2. Kafka 已经吃到了多分区并行

修复前单聊几乎全在 `partition=0`。

修复后单聊 producer 分布已经变成：

1. baseline：`256 / 448 / 256`
2. highload：`513 / 675 / 432`
3. strict_long：`384 / 672 / 384`

这说明 topic 分区数和测试配置这块已经生效了，当前不是“单分区假压测”。

### 3. 剩下的问题已经收缩成“尾部收尾问题”

虽然延迟降下来了，但成功率和覆盖率没有随之提升，说明剩余问题不是处理慢，而是固定数量的尾部消息没有在统计窗口内完成。

具体表现非常稳定：

#### 单聊

1. baseline：`960 -> 900`，少 `60`
2. highload：`1620 -> 1560`，少 `60`
3. strict_long：`1440 -> 1380`，少 `60`

#### 群聊

1. baseline：`61` 条群消息只完成了 `58` 条，差 `3` 条整消息
2. highload：`82` 条群消息只完成了 `78` 条，差 `4` 条整消息
3. strict_long：`91` 条群消息只完成了 `88` 条，差 `3` 条整消息

这不是随机抖动，更像是测试结束前总会有固定几条尾消息没被完整 drain 完。

## 当前阶段的结论

到这一步，我们已经把问题拆成两层：

### 已解决

1. commit 过频导致消费主线程阻塞
2. topic / test config 落回单分区的问题
3. websocket 写回重复更新消息状态带来的部分额外开销

### 未解决

1. stage 结束前，consumer 仍有固定尾部消息没被完整消费/统计
2. 压测脚本会在 timeout 后直接结束并停服务，这很可能在放大尾差
3. `lag=0` 不是可靠的最终 backlog 指标，不能证明消息真的全部 drain 完

## 下一步最值得做的事

下一步不该再盲目调 commit 了，应该直接围绕“尾部 drain”来修：

1. 给 benchmark 增加发送完成后的 drain 等待期，不要一到 timeout 就立刻停服务
2. 给 Kafka consumer 增加更准确的“已拉取 / 已处理 / 已标记 / 已提交”分段指标
3. 检查 stage 结束时是否还有未消费完的分区消息
4. 检查 websocket 收尾时，最后几条消息是否还在 `SendBack` 队列里
