# 调整分区专用测压脚本

这套脚本的目的不是单纯找“更高吞吐”，而是专门回答下面这个问题：

**分区继续增加之后，收益还值不值，副作用是不是还可控。**

入口脚本：

- `/workspace/czk/Personal/EchoChat/docs/k6_message_test/scripts/partition_tuning_runner.py`

输出目录：

- `/workspace/czk/Personal/EchoChat/docs/k6_message_test/partition_tuning_records/`

## 1. 为什么要单独做这套脚本

只看 `throughput` 和 `p95`，很容易漏掉几类问题：

1. 吞吐还在涨，但热点分区其实还没散开
2. 吞吐只涨了一点，但 commit / consumer 管理成本明显上去了
3. Kafka 侧没坏，但 MySQL 连接池、WS 回推队列、FD、RSS 已经开始变重
4. 某个更大分区只是“测试值更高”，但从工程默认值角度并不划算

所以这套脚本把“分区副作用”一起量化。

## 2. 这套评估体系看什么

### 2.1 业务结果

先看主指标：

1. `throughput`
2. `p95_ms`
3. `p99_ms`
4. `success / coverage`

这是第一层门槛。

### 2.2 分区是否真的被用起来

这部分回答“加的分区有没有换来更多有效车道”：

1. `producer_used_partitions`
2. `consumer_used_partitions`
3. `keys_per_used_partition`
4. `partitions_per_active_key`
5. `producer_skew_factor`
6. `consumer_skew_factor`

你可以这么理解：

1. `used_partitions` 越多，说明有效并行车道越多
2. `keys_per_used_partition` 越低越好，表示每条活跃车道上挤的会话更少
3. `skew_factor` 越接近 `1` 越均匀，越大说明热点越明显

### 2.3 Kafka consumer 侧副作用

这部分回答“分区增加后，consumer 自己的管理成本有没有被放大”：

1. `consumer_lag_p95`
2. `consumer_lag_max`
3. `consumer_total_p95_ms`
4. `heaviest_stage`
5. `heaviest_stage_share_pct`
6. `offset_commit_p95_ms`
7. `offset_commit_batch_avg`

如果分区继续加大后：

1. `lag` 开始明显抬头
2. `commit p95` 变高
3. `total stage p95` 变高

那就说明分区副作用已经开始吃掉收益。

### 2.4 MySQL 出口副作用

这部分回答“Kafka 前面并行度更高以后，落库出口有没有被带坏”：

1. `mysql_open_conns`
2. `mysql_in_use_conns`
3. `mysql_in_use_ratio`
4. `mysql_wait_count_total`
5. `mysql_wait_duration_ms_total`
6. `mysql_wait_ms_per_1k_msgs`

这里最重要的是：

1. `mysql_in_use_ratio`
2. `mysql_wait_ms_per_1k_msgs`

因为这两个指标最容易告诉你：

1. 连接池是不是开始顶满
2. 分区继续加大后，是不是把压力往 MySQL 里硬推了

### 2.5 WS 实时分发副作用

这部分回答“消息到了在线推送环节，分区增大有没有把回推链路拖坏”：

1. `ws_sendback_queue_p95`
2. `ws_enqueue_p95_ms`
3. `ws_write_p95_ms`
4. `ws_status_update_p95_ms`

如果这些值开始恶化，说明问题已经不只在 Kafka 内部。

### 2.6 单机资源副作用

这部分回答“机器成本是不是还值得”：

1. `rss_peak_mb_total`
2. `threads_peak_total`
3. `fd_peak_total`
4. `server_process_peak`
5. `throughput_gain_vs_prev_pct`
6. `rss_growth_vs_prev_pct`
7. `fd_growth_vs_prev_pct`

这里最关键的是看“边际收益”：

1. 吞吐比上一档涨了多少
2. 资源比上一档涨了多少

如果吞吐只涨一点，但 RSS / FD 继续明显上涨，就不值得继续加分区。

## 3. 脚本怎么给建议

它不是只给你原始表，而是会自动产出三类建议：

1. `当前分区副作用可控`
2. `可以继续试更大分区，但要优先盯热点和 lag`
3. `不建议继续加分区`

当前默认规则重点看：

1. 吞吐相对上一档的增益是不是已经低于 `5%`
2. `consumer skew / lag` 有没有明显恶化
3. `mysql_in_use_ratio / mysql_wait_ms_per_1k` 有没有开始变坏
4. WS queue / enqueue 有没有抬头
5. RSS / FD 增长和吞吐增长相比是否已经不划算

## 4. 推荐使用方式

### 4.1 直接跑一轮新的分区扫描

示例：

```bash
python3 /workspace/czk/Personal/EchoChat/docs/k6_message_test/scripts/partition_tuning_runner.py \
  --scenario single \
  --label part_scan_single \
  --instance-ports 18082,18083,18084,18085,18086,18087,18088,18089,18090,18091 \
  --client-instance-ports 18082,18083,18084,18085,18086,18087,18088,18089,18090,18091 \
  --partition-config 10=/workspace/czk/Personal/EchoChat/configs/config_local_singlebroker_part10_mysqlpersist_tune2.toml \
  --partition-config 60=/workspace/czk/Personal/EchoChat/configs/partition_scan/config_local_part60.toml \
  --partition-config 240=/workspace/czk/Personal/EchoChat/configs/config_local_singlebroker_part240_mysqlpersist_tune2.toml
```

### 4.2 只分析已经跑完的旧结果

示例：

```bash
python3 /workspace/czk/Personal/EchoChat/docs/k6_message_test/scripts/partition_tuning_runner.py \
  --scenario single \
  --label analyze_existing \
  --instance-ports 18082,18083,18084,18085,18086,18087,18088,18089,18090,18091 \
  --client-instance-ports 18082,18083,18084,18085,18086,18087,18088,18089,18090,18091 \
  --partition-config 10=/workspace/czk/Personal/EchoChat/configs/config_local_singlebroker_part10_mysqlpersist_tune2.toml \
  --partition-config 240=/workspace/czk/Personal/EchoChat/configs/config_local_singlebroker_part240_mysqlpersist_tune2.toml \
  --partition-run 10=/workspace/czk/Personal/EchoChat/docs/k6_message_test/records/throughput_capacity_kafka_20260412_221954_singlebroker_part10_tune2_members10 \
  --partition-run 240=/workspace/czk/Personal/EchoChat/docs/k6_message_test/records/throughput_capacity_kafka_20260412_214740_singlebroker_part240_tune2_members10
```

脚本会自动兼容两种情况：

1. 顶层 `summary.json` 完整存在
2. run 中断，但各个 `step_*` 还在，脚本会自动回收最佳通过档

## 5. 建议你实际怎么用

单机调最佳分区时，建议按这个顺序看：

1. 先看 `throughput / p95`
2. 再看 `producer_used_partitions / consumer_used_partitions / skew`
3. 再看 `lag / commit_p95`
4. 再看 `mysql_in_use_ratio / mysql_wait_ms_per_1k`
5. 最后看 `RSS / FD` 的边际成本

判断停点时，不要只问“还能不能再涨”，而要问：

**再加一档分区，吞吐多出来的这一点，值不值得我为它付出额外的 Kafka / consumer / MySQL / WS / 资源成本。**
