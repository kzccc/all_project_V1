## 结论

### 1. 之前的“固定尾差”主因已经确认

ready fix 生效后，第一档结果直接变成：

#### 单聊 step 1

1. `960 / 960 = 1.0`
2. `p95 = 108.15 ms`
3. `duration_sec = 9.596`

#### 群聊 step 1

1. `1464 / 1464 = 1.0`
2. `full_coverage = 1.0`
3. `p95 = 5 ms`

这说明前面固定少掉的那批消息，并不是“消费到尾部还没 drain 完”，而是：

服务 HTTP 已经起来了，但 Kafka consumer group 还没完成 setup，压测就开始发消息了。

由于配置是 `OffsetNewest`，所以 consumer ready 之前写入 topic 的那一小批消息，会被直接跳过。

## 为什么这个结论成立

### 1. 分区差值模式高度吻合“启动空窗”

此前 smoke run 中：

1. 单聊 producer `960`，consumer `900`
2. 三个分区分别差 `16 / 28 / 16`
3. 群聊 producer `61`，consumer `58`

这个模式更像“consumer 接管前错过一小段最早写入的数据”，而不像“尾部排空失败”。

### 2. 加 drain 没有补回消息

上一轮 drain 验证里：

1. `drain_recovered_messages = 0`
2. `drain_recovered_receipts = 0`

说明尾差不是测试结束太早。

### 3. 加 ready 后，第一档立即 100% 成功

这一步是关键证据：

1. 没有继续固定少 60 条
2. 没有继续固定少 3~4 条群消息
3. drain 也没有参与恢复

所以根因已经基本锁定。

## 新发现

虽然“固定尾差”问题解决了，但现在又暴露出一个新的真实瓶颈：

### 单聊吞吐上限大约在 `350 msg/s`

当前 smoke run 里：

1. 单聊在 `2880 target / 3000 offered` 时，成功率还能到 `1.0`
2. 但 `p95` 已经来到 `51782 ms`
3. 再往上到 `3360` 和 `3840`，成功率开始掉

说明单聊现在不是“会丢起步消息”了，而是进入了明显的积压态。

### 群聊吞吐明显更高

当前 smoke run 里：

1. 群聊一直扩到 `23040 target / 24000 offered`
2. 仍然保持 `coverage = 1.0`
3. 但 `p95` 已经涨到 `14.4 s`

这说明群聊吞吐上限比单聊高，但也已经在高档位出现明显排队和延迟膨胀。

## 当前阶段的判断

到这一步，问题已经从：

1. 测试启动过早
2. commit 阻塞
3. 单分区误压测

进一步收缩成：

1. 单聊高压下的真实积压与长尾延迟
2. 群聊高压下虽然全送达，但延迟已经明显不可接受

## 下一步应该做什么

下一步不该继续围绕“尾差”打转了，因为这个问题已经基本解决。

应该转到新的真实瓶颈：

1. 单聊为什么在约 `350 msg/s` 左右开始明显积压
2. 高压下延迟为什么膨胀到秒级甚至几十秒
3. 是否需要把“稳定通过”标准从只看成功率，升级成“成功率 + 延迟阈值”
