# 测试设计

## 目标

验证主链路中顺序号分发 `session_seq` 的真实耗时，并分别统计：

1. 冷启动 `cold_start`
2. 热启动 `hot_path`
3. Redis 顺序号 key 丢失后返回 `1` 触发 floor 恢复 `redis_floor_recovery`

## 设计原则

1. 不做脱离业务链路的微基准。
2. 直接复用当前单聊压测 runner，保持 Kafka -> consumer -> `session_seq` -> `mysql_persist` -> websocket 整条链路一致。
3. 只通过 bench admin 控制顺序号状态，不改消息链路、不旁路发送。

## 场景设计

1. `cold_start`
   重置 trace、重置会话顺序号初始化状态，直接压入一批大消息量，统计 trace 中 `seq_mode=cold_start` 的 `session_seq` 耗时。
2. `hot_path`
   先做全会话预热，让顺序号 key 和高水位都稳定，再开始主压，统计 `seq_mode=hot_path` 的 `session_seq` 耗时。
3. `redis_floor_recovery`
   在预热完成后，强制 flush 高水位，删除目标会话的 Redis 顺序号 key，但不重置初始化状态，让后续一批消息触发 `value == 1` 分支，统计 `seq_mode=redis_floor_recovery` 的耗时。

## 核心指标

1. `session_seq_ms.avg/p50/p95/p99/max`
2. 三种 `seq_mode` 的平均耗时与样本数
3. 恢复场景的异常样本比例
4. 是否存在不合理结果：
   - `hot_path` 明显慢于 `cold_start`
   - `redis_floor_recovery` 明显快于 `hot_path`
   - 某一类样本数明显不足

## 数据异常处理

1. 先用原始 trace 检查 `seq_mode` 分类是否正确。
2. 再检查极端样本是否集中在单一会话、单一时间点或单一恢复批次。
3. 若异常是框架导致，则修正脚本并自动重跑。
